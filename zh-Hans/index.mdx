    <AccordionGroup>
      <Accordion title="什么是浏览器扩展？">
        浏览器扩展是为 Chrome、Firefox、Edge 或其他浏览器添加功能的软件。扩展可以注入 UI、响应浏览器事件，并调用页面本身无法使用的特权 API。完整入门请参见 [什么是浏览器扩展？](/docs/concepts/what-is-a-browser-extension)。
      </Accordion>
      <Accordion title="Extension.js 是 Chrome 扩展框架吗？">
        是。Extension.js 既是 Chrome 扩展框架，也可作为 Firefox 和 Edge 扩展框架，且来自同一个项目。通过 `--browser=chrome`、`--browser=firefox` 或 `--browser=edge` 选择目标。参见 [可用浏览器](/docs/browsers/browsers-available)。
      </Accordion>
      <Accordion title="Extension.js 在开发中如何处理模块热替换？">
        运行 `npx extension dev`，CLI 会监听项目变更。对 popup、内容脚本、service worker 与选项页的编辑会触发最快且安全的更新路径。在支持时使用模块热替换（HMR），否则进行定向重载。CLI 会自动解决常见痛点，例如陈旧的 service worker 与漏掉的内容脚本更新。
      </Accordion>
      <Accordion title="我如何为 Chrome、Edge 和 Firefox 构建同一个扩展？">
        Extension.js 将单一代码库编译为面向特定浏览器的输出。一组命令即可为 Chrome、Edge、Firefox 以及其他基于 Chromium 的浏览器产出正确打包的构建。每个构建都得到正确的 manifest 格式、路径与平台适配。你的持续集成（CI）流水线只需在同一个构建步骤上使用不同的浏览器参数。这能减少各应用商店提交之间的差异。
      </Accordion>
      <Accordion title="我可以使用 React、Preact、Vue、Svelte 或 TypeScript 吗？">
        可以。Extension.js 提供面向 React、Preact、Vue、Svelte、TypeScript 和原生 JavaScript 的生产级模板。你可以用与其他前端项目一样的方式编写组件。框架选择不会影响核心工作流。
      </Accordion>
      <Accordion title="Extension.js 支持 Manifest V3 吗？">
        支持。工具链围绕 Manifest V3 以及浏览器应用商店当下强制执行的约束设计：service worker、声明式 API、更新后的权限以及更严格的打包。Manifest V3（MV3）是默认选项，而非事后补丁。常见陷阱可参见 [Manifest V3 排查指南](/docs/concepts/manifest-v3)。
      </Accordion>
      <Accordion title="如何排查 Manifest V3 的 service worker 与权限问题？">
        大多数 Manifest V3 排查问题分为几类：service worker 注册与 `type: "module"`、新的 `web_accessible_resources` 形态、`permissions` 与 `host_permissions` 的拆分，以及 Firefox 中 `declarative_net_request` 的差异。修复方法参见 [Manifest V3 排查指南](/docs/concepts/manifest-v3)。
      </Accordion>
      <Accordion title="如何从自定义 webpack 或 Vite 配置迁移？">
        Extension.js 取代了手写打包器配置中与扩展相关的"胶水代码"。它负责 manifest 版本管理、多浏览器输出、内容脚本注入与重载编排。把源文件和 manifest 移入 Extension.js 项目，CLI 会处理剩下的工作。你的 React/Vue/Svelte/TS 代码无需重写。
      </Accordion>
      <Accordion title="Extension.js 是否支持 monorepo 和环境变量？">
        Extension.js 支持 pnpm、npm 和 Yarn 的 workspaces。路径解析与共享依赖跨工作区边界正常工作。Extension.js 按浏览器与构建模式注入环境变量，在 JS、JSON 和 HTML 中完成替换。开发环境的密钥不会泄漏到生产产物中。
      </Accordion>
      <Accordion title="Extension.js 是免费的吗？">
        是。Extension.js 基于
        [MIT 许可证](https://github.com/extension-js/extension.js/blob/main/LICENSE) 开源。
        没有付费档位、用量上限或必须开启遥测的模式。框架由赞助者和贡献者资助。具体数据形态请查看
        [遥测契约](/docs/features/telemetry-and-privacy)。
      </Accordion>
      <Accordion title="Extension.js 支持 Safari 吗？">
        目前还不支持。`--browser=safari` 目前不是 CLI 目标。你仍然可以用共享 Web 代码编写 Safari 扩展，并在此工作流之外加入特定于 Safari 的构建步骤。当前范围请参见 [可用浏览器](/docs/browsers/browsers-available#safari-and-other-webkit-targets)。
      </Accordion>
      <Accordion title="Extension.js 会给我的包增加多少内容？">
        生产构建（`extension build`）只输出扩展引用到的代码，外加一个小型的重载运行时（在生产中会被移除）。开发构建会包含额外的 HMR 与重载胶水代码。输出位于 `dist/<browser>` 下，你可以方便地比较各浏览器之间的体积。内容脚本包的体积接近于"你的源码 + 所选 UI 框架"自身产出的体积。
      </Accordion>
      <Accordion title="如何从 CRXJS、WXT 或 Plasmo 迁移？">

        [带浏览器前缀的键](/docs/features/browser-specific-fields)），以及更新
        `package.json` 中的脚本。详细步骤请参见 [从 CRXJS 迁移](/docs/migrate/from-crxjs)。
