      <Accordion title="什么是浏览器扩展？">
        浏览器扩展是为 Chrome、Firefox、Edge 或其他浏览器添加功能的软件。
        扩展可以注入 UI、响应浏览器事件，并调用页面本身无法调用的特权 API。
        参阅 [什么是浏览器扩展？](/docs/concepts/what-is-a-browser-extension) 获取完整入门指南。
      </Accordion>
      <Accordion title="Extension.js 是 Chrome 扩展框架吗？">
        是的。Extension.js 可以作为 Chrome 扩展框架使用，同时也可作为 Firefox 和
        Edge 扩展框架，在同一个项目中工作。用 `--browser=chrome`、`--browser=firefox`
        或 `--browser=edge` 选择目标。参阅 [可用浏览器](/docs/browsers/browsers-available)。
      </Accordion>
      <Accordion title="Extension.js 在开发阶段如何处理热模块替换？">
        运行 `npx extension dev`，CLI 就会监听你的项目变化。
        对 popup、内容脚本、service worker 和 options 页面的编辑都会触发
        最安全且最快的更新路径。在支持时 CLI 使用热模块替换（HMR），否则使用
        定向 reload。它会自动解决诸如过期的 service worker 与未应用的内容脚本
        更新等常见痛点。
      </Accordion>
      <Accordion title="如何用同一份代码构建 Chrome、Edge 和 Firefox 扩展？">
        Extension.js 将单一代码库编译为浏览器特定的产物。
        同一套命令为 Chrome、Edge、Firefox 以及其他基于 Chromium 的浏览器
        生成正确打包的构建。每个构建都会获得正确的 manifest 格式、路径和平台适配。
        你的持续集成（CI）流水线使用同一个构建步骤、配合不同的浏览器标志即可，
        从而减少在各应用商店提交之间的差异。
      </Accordion>
      <Accordion title="我可以使用 React、Preact、Vue、Svelte 或 TypeScript 吗？">
        可以。Extension.js 为 React、Preact、Vue、Svelte、TypeScript 与纯 JavaScript
        提供了生产级模板。你编写组件的方式和在任何前端项目里完全一致。
        框架选择不会影响核心工作流。
      </Accordion>
      <Accordion title="Extension.js 支持 Manifest V3 吗？">
        支持。整个工具链是围绕 Manifest V3 以及当今浏览器商店所执行的
        约束构建的：service worker、声明式 API、更新后的权限以及更严格的打包。
        Manifest V3（MV3）是默认配置，而非事后补丁。常见陷阱请参阅
        [Manifest V3 故障排查](/docs/concepts/manifest-v3)。
      </Accordion>
      <Accordion title="如何排查 Manifest V3 的 service worker 和权限问题？">
        大多数 Manifest V3 的故障排查可归为几类：service worker 注册与
        `type: "module"`、新的 `web_accessible_resources` 结构、`permissions`
        与 `host_permissions` 的拆分，以及 Firefox 中 `declarative_net_request`
        的差异。修复方法请参阅 [Manifest V3 故障排查](/docs/concepts/manifest-v3)。
      </Accordion>
      <Accordion title="如何从自定义的 webpack 或 Vite 配置迁移？">
        Extension.js 替换了手写打包器配置中与扩展相关的胶水代码。
        它负责 manifest 版本、跨浏览器产物输出、内容脚本注入以及 reload 编排。
        把你的源码与 manifest 移入一个 Extension.js 项目，剩下的就交给 CLI。
        你的 React/Vue/Svelte/TS 代码无需重写即可工作。
      </Accordion>
      <Accordion title="Extension.js 支持 monorepo 和环境变量吗？">
        Extension.js 支持 pnpm、npm 和 Yarn 的 workspace。路径解析与
        共享依赖在 workspace 边界之间都能正常工作。Extension.js 会按浏览器
        和构建模式注入环境变量，并在 JS、JSON 和 HTML 中替换这些值。
        开发环境下的 secret 永远不会泄漏到生产包中。
      </Accordion>
      <Accordion title="Extension.js 是免费使用的吗？">
        是的。Extension.js 是基于
        [MIT 许可证](https://github.com/extension-js/extension.js/blob/main/LICENSE) 的开源项目。
        没有付费层级、使用上限，也没有强制要求遥测的模式。
        该框架由赞助商和贡献者资助。具体数据形状请审阅
        [遥测契约](/docs/features/telemetry-and-privacy)。
      </Accordion>
      <Accordion title="Extension.js 支持 Safari 吗？">
        暂不支持。`--browser=safari` 目前不是 CLI 的目标之一。你仍然可以
        使用共享的 Web 代码编写 Safari 扩展，并在此工作流之外添加 Safari
        特定的构建步骤。当前范围请参阅 [可用浏览器](/docs/browsers/browsers-available#safari-and-other-webkit-targets)。
      </Accordion>
      <Accordion title="Extension.js 会给我的产物增加多少体积？">
        生产构建（`extension build`）只会产出你扩展实际引用的代码，
        外加一个在生产环境会被移除的小型 reload 运行时。开发构建会包含
        额外的 HMR 与 reload 胶水。产物位于 `dist/<browser>` 下，便于你
        在不同浏览器之间比较体积。内容脚本的产物大小，与你源码加上所选
        UI 框架本身的产物大小相近。
      </Accordion>
      <Accordion title="如何从 CRXJS、WXT 或 Plasmo 迁移？">
        大多数 React、Vue 和 Svelte 源码文件可以无需重写直接复制过来。
        要做的工作集中在三处：替换打包器配置；把生成的 manifest 改回
        手写的 `manifest.json` 并使用[带浏览器前缀的键](/docs/features/browser-specific-fields)；
        以及更新 `package.json` 中的脚本。详细步骤请参阅
        [从 CRXJS 迁移](/docs/migrate/from-crxjs)。
