      <Accordion title="什么是浏览器扩展？">
        浏览器扩展是为 Chrome、Firefox、Edge 或其他浏览器添加功能的软件。扩展可以注入 UI、响应浏览器事件，并调用页面本身无法使用的特权 API。详细介绍见[什么是浏览器扩展？](/zh-Hans/docs/concepts/what-is-a-browser-extension)。
      </Accordion>
      <Accordion title="Extension.js 算 Chrome 扩展框架吗？">
        是的。Extension.js 既是 Chrome 扩展框架，也是 Firefox 和 Edge 的扩展框架，全部基于同一份项目。用 `--browser=chrome`、`--browser=firefox` 或 `--browser=edge` 指定目标。参见[支持的浏览器](/zh-Hans/docs/browsers/browsers-available)。
      </Accordion>
      <Accordion title="Extension.js 在开发时如何处理热模块替换？">
        运行 `npx extension dev`，CLI 会监听项目变更。对 popup、content script、service worker 和 options 页面的修改都会走"在安全前提下尽可能快的更新路径"。CLI 在支持的情况下使用热模块替换（HMR），否则使用定向的重载。它会自动解决 service worker 残留、content script 漏更新等常见痛点。
      </Accordion>
      <Accordion title="如何用一份代码同时构建 Chrome、Edge 和 Firefox 扩展？">
        Extension.js 把同一份代码编译成各浏览器对应的产物。一组命令就能为 Chrome、Edge、Firefox 以及其他基于 Chromium 的浏览器生成正确打包的构建。每份构建都会拿到对应的 manifest 格式、路径和平台调整。你的持续集成（CI）流水线用同一个构建步骤、传不同的浏览器参数即可。这能减少各商店提交版本之间的漂移。
      </Accordion>
      <Accordion title="可以用 React、Preact、Vue、Svelte 或 TypeScript 吗？">
        可以。Extension.js 提供可直接发布的 React、Preact、Vue、Svelte、TypeScript 以及原生 JavaScript 模板。你编写组件的方式和在任何前端项目里一样，框架选择不会影响核心工作流。
      </Accordion>
      <Accordion title="Extension.js 支持 Manifest V3 吗？">
        支持。整条工具链都是围绕 Manifest V3，以及当今各应用商店强制的限制构建的：service worker、声明式 API、新版权限模型，以及更严格的打包。Manifest V3（MV3）是默认设定，不是事后补丁。常见问题见 [Manifest V3 排错](/zh-Hans/docs/concepts/manifest-v3)。
      </Accordion>
      <Accordion title="如何排查 Manifest V3 的 service worker 和权限问题？">
        Manifest V3 的大部分问题集中在几类：service worker 注册与 `type: "module"`、新版 `web_accessible_resources` 结构、`permissions` 和 `host_permissions` 的拆分，以及 Firefox 里的 `declarative_net_request` 差异。修复办法见 [Manifest V3 排错](/zh-Hans/docs/concepts/manifest-v3)。
      </Accordion>
      <Accordion title="如何从手写的 webpack 或 Vite 配置迁移过来？">
        Extension.js 替你处理那些扩展专有的胶水代码：manifest 版本、跨浏览器产物、content script 注入和重载协调。把源码和 manifest 放进 Extension.js 项目里，剩下的交给 CLI。你的 React/Vue/Svelte/TS 代码无需重写。
      </Accordion>
      <Accordion title="Extension.js 支持 monorepo 和环境变量吗？">
        Extension.js 支持 pnpm、npm 和 Yarn 工作区。路径解析与共享依赖可以跨工作区工作。Extension.js 会按浏览器和构建模式注入环境变量，并在 JS、JSON、HTML 中替换值。开发期的敏感信息绝不会泄漏进生产包。
      </Accordion>
      <Accordion title="Extension.js 是免费的吗？">
        是的。Extension.js 是基于 [MIT 协议](https://github.com/extension-js/extension.js/blob/main/LICENSE)的开源项目。没有付费档位、用量上限，也没有"必须启用遥测"的模式。框架由赞助者与贡献者资助。具体的数据契约请参阅[遥测与隐私](/zh-Hans/docs/features/telemetry-and-privacy)。
      </Accordion>
      <Accordion title="Extension.js 支持 Safari 吗？">
        目前还不支持。`--browser=safari` 目前不是 CLI 的目标之一。你仍然可以基于共享的 Web 代码编写 Safari 扩展，并在本工作流之外加入 Safari 专有的构建步骤。当前覆盖范围见[支持的浏览器](/zh-Hans/docs/browsers/browsers-available#safari-and-other-webkit-targets)。
      </Accordion>
      <Accordion title="Extension.js 会给我的包加什么？">
        生产构建（`extension build`）只会输出扩展实际引用的代码，加上一段在生产中会被移除的小型重载运行时。开发构建额外包含 HMR 和重载胶水。产物位于 `dist/<browser>` 下，方便你在浏览器之间比对体积。content script 的体积接近于"你的源码 + 所选 UI 框架"的预期产物。
      </Accordion>
      <Accordion title="如何从 CRXJS、WXT 或 Plasmo 迁移？">
        大多数 React、Vue、Svelte 源文件可以直接复制过来，无需重写。工作主要集中在三处：替换打包器配置、把自动生成的 manifest 改回手写的 `manifest.json`（配合[带浏览器前缀的键](/zh-Hans/docs/features/browser-specific-fields)），以及更新 `package.json` 的脚本。逐步指南见[从 CRXJS 迁移](/zh-Hans/docs/migrate/from-crxjs)。
      </Accordion>
