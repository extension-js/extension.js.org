Manifest V3(MV3)用 service worker 替换了后台页面,收紧了内容安全策略,并重塑了扩展如何声明网络和主机访问。大多数日常痛点都有一个共同根源:MV3 假定后台代码是短暂的、事件驱动的。Chrome 和 Firefox 在几个键的解释上也有所不同。

本页汇总了构建 MV3 扩展时最常见的问题,给出确切的修复方法,以及 Extension.js 为你处理了哪些。

## Manifest V3 后台 `service_worker` 与 `type: "module"`

Chromium 使用 service worker 作为 MV3 后台。要从中导入 ES 模块,清单需要 `type: "module"`:

```json
{
  "manifest_version": 3,
  "background": {
    "service_worker": "service_worker.js",
    "type": "module"
  }
}
```

如果没有 `"type": "module"`,`import` 语句会在注册时失败,扩展控制台中也没有清晰的错误。有了它,你就可以在 worker 文件中编写现代 ES 模块语法。

当你的后台入口使用 ES 模块语法时,Extension.js 会自动设置 `type: "module"`。它还会把 `.ts`/`.tsx` worker 编译为单个打包输出,使模块解析在生产中也能工作。

## 为什么 `background.js` 在 Manifest V3 中行为不同

在 Manifest V2 中,`background.js` 运行在带有 DOM 的常驻后台页面里。在 Chromium 上的 Manifest V3 中,后台作为 service worker 运行:

- 没有 DOM。`window`、`document`、`XMLHttpRequest` 和 `localStorage` 都消失了。请使用 `fetch` 和 `chrome.storage`。
- worker 在空闲时 **随时可能被终止**,并在下一个事件触发时被唤醒。不要把状态保存在模块作用域变量中;请持久化到 `chrome.storage`。
- 允许顶层 `await`,但长时间的初始化本身不会让 worker 保持存活。
- 在文件顶部 **同步** 注册事件监听器,而不要放在异步回调中。在异步回调中注册的监听器不会在加载该 worker 的唤醒事件上触发。

Firefox 保留了非持久化事件页面,而不是 service worker。Extension.js 从同一源把 Chromium 路由到 `service_worker`,把 Firefox 路由到 `scripts`,因此同一份后台入口可以按目标正确编译。

## Manifest V3 中的 `web_accessible_resources`

Manifest V3 将 `web_accessible_resources` 从扁平的文件数组改为 `{ resources, matches }` 块列表:

```json
{
  "web_accessible_resources": [
    {
      "resources": ["images/logo.png", "pages/injected.html"],
      "matches": ["https://example.com/*"]
    }
  ]
}
```

常见错误:

- 列出了不在构建输出中的路径。Extension.js 只会发出从 `manifest.json`、入口 HTML 或导入代码引用的文件。把该文件添加为资源,使它落到 `dist/<browser>`。
- 忘记 `matches`。没有它,该资源不会对任何来源暴露。
- 使用 V2 风格的扁平字符串。浏览器在 MV3 中会默默忽略它们。

参见 [`web_accessible_resources` 实现](/docs/implementation-guide/web-accessible-resources) 了解完整模式,包括把扩展 URL 注入到内容脚本。

## `host_permissions` 与 `permissions`

Manifest V3 把主机访问从 `permissions` 数组中拆分出来:

| 键                                                     | 控制的内容                                                                                  |
| ------------------------------------------------------ | ------------------------------------------------------------------------------------------- |
| `permissions`                                          | 命名 API 接口(`storage`、`tabs`、`cookies`、`scripting`、`alarms` 等)。                  |
| `host_permissions`                                     | 扩展可以读取或修改的 URL 匹配模式(`https://*/*`、`*://api.example.com/*`)。               |
| `optional_permissions` 和 `optional_host_permissions`  | 通过 `chrome.permissions.request` 在运行时请求的权限。                                      |

混淆时的症状:

- `chrome.cookies.get` 对你已有访问权的站点返回空:该 URL 需要 `host_permissions`,而不仅仅是 `cookies`。
- `chrome.scripting.executeScript` 失败并提示 "Cannot access contents of url":将该 URL 加入 `host_permissions`。
- 当你只需要一个来源时,网店却警告用户存在"全站点"访问:请收窄主机模式。

参见 [权限与主机权限](/docs/implementation-guide/permissions-and-host-permissions) 了解按 API 的参考。

## Firefox 中的 `declarative_net_request`

`declarative_net_request`(DNR)是 MV3 中替代阻塞型 `webRequest` 的方案。Firefox 支持它,但有一些约束:

- 静态规则资源文件(`rule_resources`)必须是有效的 JSON 数组。Firefox 对空或格式错误的规则文件比 Chrome 更严格。
- Chrome 支持的某些规则操作和条件在 Firefox 中仍是部分实现。请查阅 [MDN 的 `declarativeNetRequest` 参考](https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/API/declarativeNetRequest) 了解当前矩阵。
- DNR 规则要在跨域请求上生效,需要 `host_permissions`(或 `<all_urls>`)。

Extension.js 在构建时验证 DNR 资源,并按浏览器发出构建产物,以避免 Firefox 专属规则文件落到 Chrome 构建中。

## 内容脚本、worker 和扩展 URL

MV3 中容易出错的三处:

- **内容脚本无法访问页面的 JavaScript 作用域。** 它们共享 DOM,而不共享 window。要与页面代码通信,请使用 `window.postMessage`(或一个指向 `web_accessible_resources` 条目的 `<script>` 标签)。
- **Service worker 在安装期间无法可靠地访问 `chrome.runtime.getURL`。** 请在需要它的事件处理程序内部惰性解析 URL。
- **扩展 URL 是按来源作用域的。** 你的扩展中的两个页面可以互相通信,但除非它们被列入 `web_accessible_resources`,否则你无法从页面 fetch 它们。

## Extension.js 如何帮助你应对 Manifest V3

- 自动将后台入口路由到 `service_worker`(Chromium)或 `scripts`(Firefox)。
- 当后台使用 ES 模块语法时设置 `type: "module"`。
- 在构建时根据当前清单版本验证 `web_accessible_resources`、`permissions` 和 `host_permissions`。
- 按浏览器发出 `dist/<browser>` 构建产物,以隔离 Chrome 和 Firefox 之间的 MV3 差异。
- 在 `extension dev` 期间,保存即重载内容脚本、service worker 和 HTML 页面。
