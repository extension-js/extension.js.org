浏览器扩展框架负责处理浏览器自身不做的工作。它为每个扩展面编译 TypeScript 和现代 JavaScript,将清单与真实输出文件连接起来,在开发期间重载 service worker 和内容脚本,并按浏览器发出构建产物。本页比较当前活跃的选择,帮助你在掌握完整信息后做出决定。

## 这些框架

| 维度             | Extension.js                          | WXT                          | CRXJS                          | Plasmo                        |
| ---------------- | ------------------------------------- | ---------------------------- | ------------------------------ | ----------------------------- |
| 模型             | `manifest.json` 作为唯一来源          | 文件约定 + 配置              | 读取清单的 Vite 插件           | 文件约定                      |
| 打包器           | Rspack(内部,零配置)                | Vite                         | Vite(插件层)                 | Parcel                        |
| 跨浏览器         | 从一套源码输出 Chrome、Edge、Firefox  | Chrome、Firefox,支持 MV2    | 专注 Chromium                  | 目标 Chrome、Firefox          |
| 配置             | 无需配置                              | `wxt.config.ts`              | `vite.config.ts` + 插件        | `package.json` 中的清单字段   |
| 打包器耦合度     | 内部细节                              | 跟随 Vite 大版本             | 打包器变更时会损坏             | 与 Parcel 绑定                |

有两项结构性差异比任何功能行都更重要:

- **清单存放在哪里。** Extension.js 读取你的 `manifest.json`,并编译它所声明的一切。WXT 和 Plasmo 根据文件约定和配置生成清单。CRXJS 也读取清单,但通过 Vite 的插件 API 进行解析,因此打包器升级可能会破坏它(参见 [Vite 8 fileName 错误](/docs/migrate/crxjs-content-script-filename-undefined))。
- **谁拥有打包器。** 包裹通用打包器的框架会继承该打包器的破坏性变更。Extension.js 把打包器当作内部细节:你永远不需要配置它,升级是框架的问题,而不是你的。

## 无需承诺即可尝试

只需一条命令,运行任意扩展模板,或 GitHub 上的任意扩展仓库:

```bash
npx extension@latest dev
```

你的组件、`chrome.*` 调用和样式可以从上述任意框架延续过来;[迁移指南](/docs/migrate/from-crxjs) 涵盖了需要改动的连接部分。
