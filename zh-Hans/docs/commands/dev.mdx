`dev` 是日常浏览器扩展开发的命令，提供监听模式、浏览器启动以及上下文感知的更新行为。

`dev` 会运行开发流水线并监听你的项目文件。它会根据改动内容选择更新策略：热模块替换 (HMR)、硬重载，或者必须时执行完整重启。

## 何时使用 `dev`

- 边开发边实时验证修改。
- 在一个或多个浏览器目标中调试扩展行为。
- 使用 `--source` 运行源码检查工作流。

构建生产产物使用 `build`，构建后立即启动使用 `start`，仅运行已有构建产物使用 `preview`。

如果你的扩展位于 monorepo / submodule 中，请回顾 `extension.config.*` 加载 env 文件的方式（包括 workspace 根级回退）：[环境变量](/zh-Hans/docs/features/environment-variables#how-it-works)。

## Dev 命令能力

| 能力                 | 你能得到什么                                          |
| -------------------- | ----------------------------------------------------- |
| 监听模式迭代         | 编写代码时形成紧凑的「编辑 → 重新构建 → 校验」循环    |
| 浏览器目标控制       | 每条命令都可显式跨浏览器校验                          |
| 感知 profile 的运行  | 根据工作流需要可靠地选择全新或持久 profile            |
| 源码检查模式         | 通过 source flag 提供结构化输出与诊断信息             |

<CodeGroup>

如果你省略路径，Extension.js 会使用当前工作目录。你也可以传入 **GitHub tree URL**（例如 `https://github.com/user/repo/tree/main/path`）。Extension.js 会下载该仓库，并在本地副本上运行开发模式。

## 自动化元数据（推荐给脚本 / agent）

`dev` 运行时，Extension.js 会在以下位置写出机器可读的元数据：

- `dist/extension-js/<browser>/ready.json`
- `dist/extension-js/<browser>/events.ndjson`（按行分隔的 JSON）

在自动化场景（Playwright、持续集成 (CI)、AI agent）中，请优先使用这些文件，而不是去解析终端日志。

把 `ready.json` 作为就绪契约来对待：

- 启动中：`status: "starting"`
- 运行时就绪：`status: "ready"`
- 启动 / 编译失败：`status: "error"`
- `runId` 唯一标识一次运行时会话
- `startedAt` 标记运行时会话的开始时间戳

### `--no-browser` 与就绪同步

`--no-browser` 只会禁用浏览器启动，并不会在编译完成之前阻塞外部运行器。

对于 Playwright / CI / AI 工作流：

1. 把 `extension dev --no-browser` 作为长期运行的进程启动。
2. 把 `extension dev --wait --browser=<browser>` 作为就绪门控运行。
3. 仅在 `status: "ready"` 之后再启动外部浏览器自动化。

`--wait` 面向第二个进程（或 CI 步骤），遇到 `error` / 超时会以非零状态码退出。
当 `--wait` 检测到 `ready.json` 是一个来自已死进程的旧记录（`pid` 已不再存活）时，它会继续等待活跃的生产者。
如果你在同一次命令调用中同时传入 `--wait` 与 `--no-browser`，`--wait` 优先。命令会以仅等待模式运行。

### 通过 `--no-reload` 得到干净的开发包

`--no-reload` 会跳过 content script 的重新注入包装器以及重新构建后的重载派发。开发产物 `dist` 更接近生产构建，文件变化时也不会打扰已经打开的标签页。需要你手动重载扩展或页面来获取改动。

`--no-reload` 仅支持 `extension dev`。把它传给 `start`、`preview` 或 `build` 会以错误退出。在内部它会设置 `EXTENSION_NO_RELOAD=true`，让 develop 进程在 CLI argv 之外也能读到它。

## 源码检查工作流

source 选项只能与 `dev` 搭配使用。源码检查会在浏览器中打开目标 URL，并在你的 content script 执行**之后**打印实时的完整 HTML。用它来确认扩展实际改变了什么。

- `--source [url]` 用于检查一个页面（默认为 `--starting-url` 或 `https://example.com`）
- `--watch-source` / `--no-watch-source` 每次重新构建时重新打印 HTML

`start` 与 `preview` 不支持源码检查相关 flag。

### Source flag

| Flag                                            | 作用                                       | 默认行为                          |
| ----------------------------------------------- | ------------------------------------------ | --------------------------------- |
| `--source [url]`                                | 启用源码检查流程。                         | 禁用                              |
| `--watch-source [boolean]`                      | 监听到更新时重新打印 source 输出。         | `--source` 启用时为 `true`        |
| `--source-format <pretty\|json\|ndjson>`        | source 检查的输出格式。                    | 回退到日志格式或 `json`            |
| `--source-summary [boolean]`                    | 输出紧凑摘要。                             | 禁用                              |
| `--source-meta [boolean]`                       | 包含页面元数据。                           | source 启用时启用                  |
| `--source-probe <selectors>`                    | 逗号分隔的 CSS 选择器探针。                | unset                             |
| `--source-tree <off\|root-only>`                | 包含紧凑的扩展树详情。                     | unset                             |
| `--source-console [boolean]`                    | 包含 console 摘要。                        | 禁用                              |
| `--source-dom [boolean]`                        | 包含 DOM 快照 / 差异。                     | `watch-source` 启用时启用          |
| `--source-max-bytes <bytes>`                    | 限制输出载荷大小。                         | unset                             |
| `--source-redact <off\|safe\|strict>`           | source 输出的脱敏策略。                    | 取决于格式                        |
| `--source-include-shadow <off\|open-only\|all>` | 控制 Shadow DOM 的包含范围。               | source 启用时为 `open-only`       |
| `--source-diff [boolean]`                       | 监听更新时包含 diff 元数据。               | `watch-source` 启用时启用          |

## Monorepo 与 workspace 根

你可以让 `dev`（以及 `build`）指向 **monorepo 的根目录**，而不是扩展包本身。Extension.js 会检测 workspace 根并自动解析其中的扩展包：

```bash
extension dev .            # 在 monorepo 根目录运行
```

当恰好找到一个扩展包时，Extension.js 会解析它并打印：

```text
Workspace root detected — resolved extension package: packages/my-extension
```

当存在多个候选时，它会列出来，让你显式指向你想要的那个：

```bash
extension dev packages/my-extension
```

## 最佳实践

- **浏览器兼容性：** 在不同浏览器中测试你的扩展，确保它在每个目标上都能工作。
- **Polyfill：** 如果同时面向 Firefox 或 Gecko 系浏览器，使用 `--polyfill`。它会在 Chromium 系浏览器中启用 `browser.*` API 兼容。
- **源码检查工作流：** 当你需要源码检查选项时使用 `dev`（而不是 `start` / `preview`）。
- **自动化的可靠性：** 把 `dev` 视为监听模式伴侣（`--no-browser` + `dev --wait`）；把 `start` 视为生产模式伴侣（`--no-browser` + `start --wait`）。脚本与 CI 自动化使用 `--wait-format=json`。
