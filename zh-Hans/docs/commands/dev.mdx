`dev` 用于日常的浏览器扩展开发,提供监听模式、浏览器启动以及上下文感知的更新行为。

`dev` 会运行开发管线并监听你的项目文件。它会根据变更内容应用相应的更新策略:热模块替换 (HMR)、硬重载,或在必要时进行完整重启。

## 何时使用 `dev`

- 实时构建功能并验证变更。
- 在一个或多个浏览器目标中调试扩展行为。
- 使用 `--source` 运行源码检查工作流。

使用 `build` 生成生产产物,使用 `start` 进行生产构建并启动,使用 `preview` 仅运行已有的构建输出。

如果你的扩展位于 monorepo/子模块中,请查看 `extension.config.*` 如何加载环境变量文件(包括工作区根目录回退):[环境变量](/docs/features/environment-variables#how-it-works)。

## Dev 命令能力

| 能力               | 它能提供什么                                          |
| ------------------ | ----------------------------------------------------- |
| 监听模式迭代       | 编码时获得紧凑的编辑 → 重建 → 验证循环                |
| 浏览器目标控制     | 每条命令都能显式进行跨浏览器验证                      |
| 配置感知的运行     | 根据工作流需要使用可靠的全新或持久化配置文件          |
| 源码检查模式       | 通过源码相关选项获取结构化输出和诊断                  |

## 自动化元数据(推荐用于脚本/代理)

`dev` 运行时,Extension.js 会在以下位置输出机器可读的元数据:

- `dist/extension-js/<browser>/ready.json`
- `dist/extension-js/<browser>/events.ndjson`(以换行分隔的 JSON)

对于自动化场景(Playwright、持续集成 (CI)、AI 代理),建议优先使用这些文件,而不是解析终端日志。

把 `ready.json` 当作就绪契约:

- 启动过程中为 `status: "starting"`
- 运行时就绪后为 `status: "ready"`
- 启动/编译失败时为 `status: "error"`
- `runId` 唯一标识一次运行时会话
- `startedAt` 记录运行时会话的开始时间戳

### `--no-browser` 与就绪同步

`--no-browser` 仅禁用浏览器启动,不会在编译完成前阻塞外部执行器。

对于 Playwright/CI/AI 工作流:

1. 将 `extension dev --no-browser` 作为长期运行的进程。
2. 将 `extension dev --wait --browser=<browser>` 作为就绪门控。
3. 仅在出现 `status: "ready"` 之后再启动外部浏览器自动化。

`--wait` 面向第二个进程(或 CI 步骤),在 `error`/超时情况下以非零状态退出。
当 `--wait` 检测到来自已退出进程的陈旧 `ready.json`(`pid` 已不存在)时,它会继续等待存活的生产者。
如果在同一条命令中同时传入 `--wait` 和 `--no-browser`,`--wait` 优先生效,命令运行在只等待模式下。

### 使用 `--no-reload` 获得干净的开发打包

`--no-reload` 会跳过内容脚本的重新注入包装以及重建后的重载分发。开发 `dist` 将更接近生产打包,文件变更也不会打扰已经打开的标签页。请你手动重载扩展或页面以应用变化。

`--no-reload` 仅在 `extension dev` 中支持。将其传给 `start`、`preview` 或 `build` 都会以错误退出。在内部它会设置 `EXTENSION_NO_RELOAD=true`,以便开发过程可以从 CLI argv 之外读取它。

## 源码检查工作流

源码相关选项只可与 `dev` 一起使用。源码检查会在浏览器中打开一个目标 URL,并在你的内容脚本执行 **之后** 打印完整、实时的 HTML。用它来验证你的扩展实际改动了什么。

- `--source [url]` 用于检查页面(默认值为 `--starting-url` 或 `https://example.com`)
- `--watch-source` / `--no-watch-source` 用于在每次重建后重新打印 HTML

`start` 和 `preview` 不支持源码检查相关选项。

### 源码选项

| 选项                                            | 作用                                       | 默认行为                              |
| ----------------------------------------------- | ------------------------------------------ | ------------------------------------- |
| `--source [url]`                                | 启用源码检查流程。                         | 禁用                                  |
| `--watch-source [boolean]`                      | 在被监听的更新发生时重新打印源码输出。     | 启用 `--source` 时为 `true`           |
| `--source-format <pretty\|json\|ndjson>`        | 源码检查的输出格式。                       | 回退到日志格式或 `json`               |
| `--source-summary [boolean]`                    | 输出紧凑的摘要。                           | 禁用                                  |
| `--source-meta [boolean]`                       | 包含页面元数据。                           | 启用源码时启用                        |
| `--source-probe <selectors>`                    | 用逗号分隔的 CSS 选择器进行探测。          | 未设置                                |
| `--source-tree <off\|root-only>`                | 包含紧凑的扩展树详情。                     | 未设置                                |
| `--source-console [boolean]`                    | 包含控制台摘要。                           | 禁用                                  |
| `--source-dom [boolean]`                        | 包含 DOM 快照/差异。                       | 启用 watch-source 时启用              |
| `--source-max-bytes <bytes>`                    | 限制输出负载大小。                         | 未设置                                |
| `--source-redact <off\|safe\|strict>`           | 源码输出的脱敏策略。                       | 取决于格式                            |
| `--source-include-shadow <off\|open-only\|all>` | 控制是否包含 Shadow DOM。                  | 启用源码时为 `open-only`              |
| `--source-diff [boolean]`                       | 在监听更新时包含差异元数据。               | 启用 watch-source 时启用              |

## Monorepo 与工作区根

你可以让 `dev`(以及 `build`)指向 **monorepo 的根目录**,而不是扩展包本身。Extension.js 会检测工作区根目录并自动解析其中的扩展包:

```bash
extension dev .            # run from the monorepo root
```

当恰好找到一个扩展包时,Extension.js 会解析它并打印:

```text
Workspace root detected — resolved extension package: packages/my-extension
```

当有多个候选时,会列出来让你指明要使用哪一个:

```bash
extension dev packages/my-extension
```

## 最佳实践

- **浏览器兼容性:** 在不同的浏览器中测试扩展,确保它在每个目标上都能工作。
- **Polyfill:** 如果目标中包含 Firefox 或基于 Gecko 的浏览器,请使用 `--polyfill`。这个选项会在基于 Chromium 的浏览器中启用 `browser.*` API 兼容。
- **源码检查工作流:** 需要源码检查相关选项时,请使用 `dev`(而不是 `start`/`preview`)。
- **自动化可靠性:** 将 `dev` 视为监听模式伙伴(`--no-browser` + `dev --wait`),将 `start` 视为生产伙伴(`--no-browser` + `start --wait`)。在脚本和 CI 自动化中使用 `--wait-format=json`。
