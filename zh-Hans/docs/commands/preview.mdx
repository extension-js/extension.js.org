启动一个已经构建好的扩展产物，进行接近生产环境的手动测试。

`preview` 不会编译你的项目。它会加载一个已经存在的未打包扩展根目录，并运行浏览器启动流程。

## 何时使用 `preview`

- 在不重新构建的情况下运行现有的构建产物。
- 快速对比同一份打包产物在不同浏览器目标下的行为。
- 调试与生产产物（而非 dev/watch 模式）相关的运行时问题。

## preview 命令的能力

| 能力             | 你能得到什么                                          |
| ---------------- | ----------------------------------------------------- |
| 构建产物验证     | 在不重新构建的前提下测试真实的生产产物                |
| 浏览器目标检查   | 在所选浏览器目标上运行编译产物                        |
| 启动器控制       | 根据工作流需要启动或跳过浏览器启动器                  |
| 快速手动 QA      | 在发布前快速验证打包就绪的行为                        |

> `preview` 仅用于运行。当 `dist/<browser>` 输出存在时优先使用。你也可以让它指向另一个已经包含 `manifest.json` 的未打包扩展文件夹。

## 用法

<CodeGroup>

```bash npm
extension preview [project-path] [options]
```

```bash pnpm
extension preview [project-path] [options]
```

```bash yarn

```

```bash bun
extension preview [project-path] [options]
```

```bash bun
extension preview [project-path] [options]
```

</CodeGroup>

如果省略路径，Extension.js 会使用当前工作目录。

## `preview` 如何选择要运行的内容

`preview` 按如下顺序检查这些位置：

1. 所选浏览器目标对应的 `dist/<browser>`。
2. 提供的项目路径或当前工作目录。

该文件夹需要包含一个带有 `manifest.json` 的未打包扩展。是否在同一条命令里运行过 build 并不重要。

## 参数与标志

| 标志                        | 别名               | 作用                                                                          | 默认值                 |
| --------------------------- | ------------------ | ----------------------------------------------------------------------------- | ---------------------- |
| `[path]`                    | -                  | 从指定项目路径预览已构建的扩展。                                              | `process.cwd()`        |
| `--browser <browser>`       | -                  | 浏览器/引擎目标（`chromium`、`chrome`、`edge`、`firefox` 等）。               | `chromium`             |
| `--profile <path\|boolean>` | -                  | 浏览器 profile 路径，或布尔型 profile 模式。                                  | 全新 profile           |
| `--chromium-binary <path>`  | -                  | [自定义 Chromium 系列二进制路径](/zh-Hans/docs/browsers/running-other-browsers)。 | 系统默认               |
| `--gecko-binary <path>`     | `--firefox-binary` | [自定义 Gecko 系列二进制路径](/zh-Hans/docs/browsers/running-other-browsers)。    | 系统默认               |
| `--starting-url <url>`      | -                  | 启动浏览器时打开的初始 URL。                                                  | 未设置                 |
| `--no-browser`              | -                  | 跳过浏览器启动。                                                              | 启用浏览器启动         |
| `--port <port>`             | -                  | 启用 runner 时的 runner/devtools 端口。使用 `0` 代表由操作系统分配端口。      | `8080`                 |
| `--host <host>`             | -                  | 用于绑定 dev server 的 host。在 Docker/开发容器中使用 `0.0.0.0`。              | `127.0.0.1`            |
| `--extensions <list>`       | -                  | 逗号分隔的伴随扩展或商店 URL 列表。                                           | 未设置                 |
| `--author`                  | `--author-mode`    | 启用维护者诊断信息。                                                          | 禁用                   |

## 自动化元数据

`preview` 会将就绪状态元数据写入：

- `dist/extension-js/<browser>/ready.json`

对于 `--no-browser` 流程，这提供了确定性的命令状态：

- `starting`：命令正在初始化
- `ready`：仅运行模式的校验已完成
- `error`：所需的输出缺失，或启动失败
- `runId` 与 `startedAt`：用于在脚本/agent 中做会话关联

`preview` 不提供 `--wait` 阻塞标志。在 `preview` 自动化场景下，请直接消费 `ready.json`。

## 行为说明

- `preview` 仅用于运行，从不编译项目。
- `preview` 优先使用已存在的构建产物（`dist/<browser>`），也可以回退到另一个未打包的扩展根目录。
- `preview` 不会运行 watch 模式或热模块替换（HMR）。
- `preview` 不支持源码探查标志；这类工作流请使用 `dev`。
- 对于脚本/agent，请依赖 `ready.json`，避免解析终端输出。

## 最佳实践

- 在测试全新的生产产物前，先运行 `build` 再运行 `preview`。
- 当你的未打包扩展不在默认的项目输出位置时，传入项目路径参数。
- 在打包之前，使用 `--browser` 在多个目标上验证行为。
