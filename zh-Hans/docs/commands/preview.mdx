启动一个已构建的扩展输出,用于类生产的手动测试。

`preview` 不会编译你的项目。它加载一个已有的未打包扩展根目录并运行浏览器启动流程。

## 何时使用 `preview`

- 在不重新构建的情况下运行已有的构建输出。
- 快速比较打包后的行为在不同浏览器目标之间的差异。
- 调试与生产产物(而非开发/监听模式)相关的运行时问题。

## Preview 命令能力

| 能力                 | 它能提供什么                                            |
| -------------------- | ------------------------------------------------------- |
| 构建输出验证         | 测试真实的生产产物,无需重新构建                        |
| 浏览器目标检查       | 在所选浏览器目标上运行编译后的输出                      |
| 运行器控制           | 根据工作流需要启动或跳过浏览器运行器                    |
| 快速人工 QA          | 在发布前快速验证打包就绪的行为                          |

> `preview` 只负责运行。当 `dist/<browser>` 输出存在时,它会优先使用该目录。你也可以让它指向另一个已经包含 `manifest.json` 的未打包扩展文件夹。


如果省略路径,Extension.js 会使用当前工作目录。

## `preview` 如何选择要运行的内容

`preview` 会按以下顺序检查这些位置:

1. 所选浏览器目标对应的 `dist/<browser>`。
2. 提供的项目路径或当前工作目录。

目标文件夹需要包含一个带 `manifest.json` 的未打包扩展。同一条命令中是否运行过构建并不重要。

## 参数与选项

| 选项                        | 别名               | 作用                                                                       | 默认值                 |
| --------------------------- | ------------------ | -------------------------------------------------------------------------- | ---------------------- |
| `[path]`                    | -                  | 从项目路径预览已构建的扩展。                                               | `process.cwd()`        |
| `--browser <browser>`       | -                  | 浏览器/引擎目标(`chromium`、`chrome`、`edge`、`firefox` 等)。            | `chromium`             |
| `--profile <path\|boolean>` | -                  | 浏览器配置文件路径或布尔型配置模式。                                       | 全新配置               |
| `--chromium-binary <path>`  | -                  | [自定义 Chromium 家族二进制路径](/docs/browsers/running-other-browsers)。  | 系统默认               |
| `--gecko-binary <path>`     | `--firefox-binary` | [自定义 Gecko 家族二进制路径](/docs/browsers/running-other-browsers)。     | 系统默认               |
| `--starting-url <url>`      | -                  | 在启动的浏览器中打开的初始 URL。                                           | 未设置                 |
| `--no-browser`              | -                  | 跳过浏览器启动。                                                           | 默认启用浏览器启动     |
| `--port <port>`             | -                  | 启用运行器时的运行器/DevTools 端口。使用 `0` 表示由操作系统分配。          | `8080`                 |
| `--host <host>`             | -                  | 开发服务器绑定的主机。在 Docker/开发容器中使用 `0.0.0.0`。                 | `127.0.0.1`            |
| `--extensions <list>`       | -                  | 用逗号分隔的伴随扩展列表或商店 URL。                                       | 未设置                 |
| `--author`                  | `--author-mode`    | 启用维护者诊断模式。                                                       | 禁用                   |

## 自动化元数据

`preview` 会将就绪元数据写入:

- `dist/extension-js/<browser>/ready.json`

对于 `--no-browser` 流程,它提供确定性的命令状态:

- 命令初始化期间为 `starting`
- 仅运行的验证完成后为 `ready`
- 缺少所需输出或启动失败时为 `error`
- `runId` 和 `startedAt` 用于在脚本/代理中关联会话

`preview` 不提供 `--wait` 门控选项。对于 `preview` 自动化,请直接消费 `ready.json`。

## 行为说明

- `preview` 只负责运行,绝不会编译项目。
- `preview` 优先使用已有的构建输出(`dist/<browser>`),但可以回退到另一个未打包扩展根目录。
- `preview` 不运行监听模式,也不支持热模块替换 (HMR)。
- `preview` 不支持源码检查相关选项;请改用 `dev`。
- 对于脚本/代理,请依赖 `ready.json`,不要解析终端输出。

## 最佳实践

- 在测试一个全新的生产产物时,先运行 `build` 再运行 `preview`。
- 当未打包扩展位于默认项目输出之外时,请传入项目路径参数。
- 在打包前使用 `--browser` 在多个目标上验证行为。
