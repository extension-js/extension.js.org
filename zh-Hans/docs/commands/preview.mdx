为类生产的手动测试启动一个已构建的扩展输出。

`preview` 不编译你的项目。它会加载一个现有的未打包扩展根目录并运行浏览器启动流程。

## 何时使用 `preview`

- 在不重新构建的情况下运行已有的构建输出。
- 在不同浏览器目标之间快速比较已打包的行为。
- 调试与生产产物(而不是 dev/监听模式)绑定的运行时问题。

## Preview 命令能力

| 能力                    | 它能带给你什么                                         |
| ----------------------- | ------------------------------------------------------ |
| 构建输出验证            | 在不重新构建的情况下测试真实的生产产物                 |
| 浏览器目标检查          | 在选定的浏览器目标上运行编译后的输出                   |
| 运行器控制              | 按工作流需要启动或跳过浏览器运行器                     |
| 快速手动 QA             | 在发布前快速验证可打包行为                             |

> `preview` 是仅运行模式。当 `dist/<browser>` 输出存在时优先使用它。你也可以把它指向另一个已包含 `manifest.json` 的未打包扩展文件夹。


如果你省略路径,Extension.js 会使用当前工作目录。

## `preview` 如何选择要运行的内容

`preview` 按顺序检查以下位置:

1. 所选浏览器目标的 `dist/<browser>`。
2. 提供的项目路径或当前工作目录。

该文件夹需要包含一个带 `manifest.json` 的未打包扩展。是否在同一命令中运行过构建并不重要。

## 参数与标志

| 标志                        | 别名               | 它的作用                                                                       | 默认值                 |
| --------------------------- | ------------------ | ------------------------------------------------------------------------------ | ---------------------- |
| `[path]`                    | -                  | 从项目路径预览已构建的扩展。                                                   | `process.cwd()`        |
| `--browser <browser>`       | -                  | 浏览器/引擎目标(`chromium`、`chrome`、`edge`、`firefox` 等)。                | `chromium`             |
| `--profile <path\|boolean>` | -                  | 浏览器配置文件路径或布尔配置文件模式。                                         | 全新配置文件           |
| `--chromium-binary <path>`  | -                  | [自定义 Chromium 系二进制路径](/docs/browsers/running-other-browsers)。         | 系统默认               |
| `--gecko-binary <path>`     | `--firefox-binary` | [自定义 Gecko 系二进制路径](/docs/browsers/running-other-browsers)。            | 系统默认               |
| `--starting-url <url>`      | -                  | 在启动的浏览器中作为起始 URL。                                                 | 未设置                 |
| `--no-browser`              | -                  | 跳过浏览器启动。                                                               | 启用浏览器启动         |
| `--port <port>`             | -                  | 启用运行器时使用的运行器/DevTools 端口。使用 `0` 由操作系统分配。              | `8080`                 |
| `--host <host>`             | -                  | 开发服务器绑定的主机。Docker/dev 容器使用 `0.0.0.0`。                          | `127.0.0.1`            |
| `--extensions <list>`       | -                  | 逗号分隔的伴随扩展或商店 URL。                                                 | 未设置                 |
| `--author`                  | `--author-mode`    | 启用维护者诊断。                                                               | 禁用                   |

## 自动化元数据

`preview` 将就绪元数据写入:

- `dist/extension-js/<browser>/ready.json`

对于 `--no-browser` 流程,它提供确定的命令状态:

- 命令初始化期间为 `starting`
- 仅运行验证完成后为 `ready`
- 缺少所需输出或启动失败时为 `error`
- `runId` 和 `startedAt` 用于在脚本/代理中关联会话

`preview` 不提供 `--wait` 闸门标志。对于 `preview` 自动化,直接消费 `ready.json`。

## 行为说明

- `preview` 是仅运行模式,绝不会编译项目。
- `preview` 优先使用已有的构建输出(`dist/<browser>`),但也可以回退到另一个未打包扩展根目录。
- `preview` 不运行监听模式或热模块替换(HMR)。
- `preview` 不支持源代码检视标志;请用 `dev` 实现该工作流。
- 对于脚本/代理,依赖 `ready.json`,避免解析终端输出。

## 最佳实践

- 在测试新鲜的生产产物时,先运行 `build` 再运行 `preview`。
- 当你的未打包扩展不在默认项目输出位置时,传入项目路径参数。
- 使用 `--browser` 在打包前跨目标验证行为。
