`create` 会根据所选模板生成文件、配置和启动脚本,并可选择安装依赖。

## 何时使用 `create`

- 从零开始一个新扩展。
- 快速搭建多个概念验证 (POC) 想法。
- 用一致的模板默认值标准化团队成员的上手体验。

## Create 命令能力

| 能力             | 它能提供什么                                                |
| ---------------- | ----------------------------------------------------------- |
| 模板脚手架       | 从官方模板出发,获得一个就绪的项目结构                      |
| 依赖安装         | 可在脚手架后选择安装所需依赖                                |
| 路径灵活性       | 按项目名或显式文件夹路径创建                                |
| 快速上手         | 从空文件夹快速过渡到可运行的扩展                            |

## 用法

<CodeGroup>

```bash npm
npx extension@latest create <extension-name|extension-path> [options]
```

```bash pnpm
pnpx extension@latest create <extension-name|extension-path> [options]
```

```bash yarn
yarn dlx extension@latest create <extension-name|extension-path> [options]
```

```bash bun
bunx extension@latest create <extension-name|extension-path> [options]
```

```bash bun
bunx extension@latest create <extension-name|extension-path> [options]
```

</CodeGroup>

## 参数与选项

| 选项                  | 别名  | 作用                                       | 默认值       |
| --------------------- | ----- | ------------------------------------------ | ------------ |
| `[path or name]`      | -     | 要创建的项目文件夹/名称。                  | 必填         |
| `--template <name>`   | `-t`  | 设置模板别名。                             | `javascript` |
| `--install [boolean]` | -     | 在脚手架完成后安装依赖。                   | `false`      |

如果你想使用默认的 JavaScript 启动模板,直接省略 `--template` 即可。仅当需要使用 [官方示例](https://github.com/extension-js/examples/tree/main/examples) 中的其他技术栈时,才添加 `--template=<slug>`。

`init` 是同一默认启动模板的别名。只有当示例仓库中确实存在名为 `default` 的文件夹时,才使用字面值 `default`。

## 共享的全局选项

## 最佳实践

- 从匹配你 UI/运行时需求的模板开始,减少后续设置上的偏差。
- 第一次运行保持精简,在确认基本命令流程之后再添加额外工具。
