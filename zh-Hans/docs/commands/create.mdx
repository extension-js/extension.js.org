`create` 会按所选模板生成文件、配置以及启动脚本，并可选地安装依赖。

## 何时使用 `create`

- 从零开始一个新扩展。
- 快速搭建多个概念验证（PoC）。
- 用统一的模板默认值，标准化团队成员的上手流程。

## Create 命令能力

| 能力                 | 你能得到什么                                            |
| -------------------- | ------------------------------------------------------- |
| 模板脚手架           | 基于官方模板，搭配就绪的项目结构                        |
| 依赖安装             | 脚手架完成后可选安装所需依赖                            |
| 路径灵活性           | 既可以按项目名创建，也可以按显式目录路径创建            |
| 顺畅的上手           | 从空目录快速走到可运行的扩展                            |

## 用法

<CodeGroup>

```bash npm
npx extension@latest create <extension-name|extension-path> [options]
```

```bash pnpm
pnpx extension@latest create <extension-name|extension-path> [options]
```

```bash yarn
yarn dlx extension@latest create <extension-name|extension-path> [options]
```

```bash bun
bunx extension@latest create <extension-name|extension-path> [options]
```

```bash bun
bunx extension@latest create <extension-name|extension-path> [options]
```

</CodeGroup>

## 参数与 flag

| Flag                  | 别名  | 作用                                | 默认值       |
| --------------------- | ----- | ----------------------------------- | ------------ |
| `[path or name]`      | -     | 要创建的项目目录 / 名称。           | 必填         |
| `--template <name>`   | `-t`  | 设置模板 slug。                     | `javascript` |
| `--install [boolean]` | -     | 脚手架完成后安装依赖。              | `false`      |

如果你想使用默认的 JavaScript 起点，直接省略 `--template`。当你需要选择[官方示例](https://github.com/extension-js/examples/tree/main/examples)中的其他技术栈时，再加 `--template=<slug>`。

slug `init` 是同一个默认起点的别名。只有当 examples 仓库里确实存在某个名为 `default` 的文件夹时，才使用字面量 slug `default`。

## 共享的全局选项

## 最佳实践

- 选择与你 UI / 运行时需求匹配的模板，减少配置漂移。
- 让第一次运行保持简单，验证基础命令流程后再追加额外工具链。
