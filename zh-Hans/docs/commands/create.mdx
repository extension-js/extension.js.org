`create` 会为所选模板搭建文件、配置和启动脚本,并可选地安装依赖。

## 何时使用 `create`

- 从零开始一个新扩展。
- 快速搭建多个概念验证。
- 用一致的模板默认值,为团队成员提供标准化的上手体验。

## Create 命令能力

| 能力                 | 它能带给你什么                                              |
| -------------------- | ----------------------------------------------------------- |
| 模板脚手架           | 以官方模板和现成的项目结构起步                              |
| 依赖安装             | 脚手架后可选地安装所需依赖                                  |
| 路径灵活性           | 可按项目名或显式文件夹路径创建                              |
| 快速上手             | 从空文件夹快速到可运行的扩展                                |

## 用法

<CodeGroup>

```bash npm
npx extension@latest create <extension-name|extension-path> [options]
```

```bash pnpm
pnpx extension@latest create <extension-name|extension-path> [options]
```

```bash yarn
yarn dlx extension@latest create <extension-name|extension-path> [options]
```

```bash bun
bunx extension@latest create <extension-name|extension-path> [options]
```

```bash bun
bunx extension@latest create <extension-name|extension-path> [options]
```

</CodeGroup>

## 参数与标志

| 标志                  | 别名 | 它的作用                                  | 默认值        |
| --------------------- | ---- | ----------------------------------------- | ------------- |
| `[path or name]`      | -    | 要创建的项目文件夹/名称。                 | 必填          |
| `--template <name>`   | `-t` | 设置模板别名。                            | `javascript`  |
| `--install [boolean]` | -    | 在脚手架完成后安装依赖。                  | `false`       |

如果你想使用默认的 JavaScript 起始模板,完全省略 `--template` 即可。仅当你想使用[官方示例](https://github.com/extension-js/examples/tree/main/examples)中其他技术栈时,才添加 `--template=<slug>`。

别名 `init` 等价于同一个默认起始模板。只有当 examples 仓库中真的存在 `default` 文件夹时,才使用字面别名 `default`。

## 共享的全局选项

同样支持[全局标志](/docs/workflows/global-flags)。

## 最佳实践

- 从契合你 UI/运行时需求的模板起步,减少配置漂移。
- 让首次运行保持小而清晰,在验证基础命令流程后再添加额外工具。
