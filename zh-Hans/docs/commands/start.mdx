当你希望在一条命令中完成生产构建并立即启动浏览器时,使用 `start`。

`start` 命令先运行生产构建,然后使用与 `preview` 相同的流程启动构建好的扩展。

## 何时使用 `start`

- 编译完成后立刻手动验证生产行为。
- 复现监听模式与生产输出之间的运行时差异。
- 在本地进行类生产检查,而不是先 `build` 再 `preview` 分两步。

## Start 命令能力

| 能力                     | 它能带给你什么                                              |
| ------------------------ | ----------------------------------------------------------- |
| 构建 + 启动 工作流       | 一步运行生产编译并启动浏览器                                |
| 目标选择                 | 直接在选定的浏览器或引擎目标中启动                          |
| 运行器控制               | 在只需构建验证时跳过浏览器启动                              |
| 类生产验证               | 检查真实的编译输出而不是监听模式状态                        |

## 与其他命令的差异

- `dev`:开发服务器 + 热模块替换(HMR)/监听模式
- `build`:仅生产构建
- `preview`:不构建直接启动一个已有的扩展
- `start`:依次执行 `build` + `preview`

## 自动化元数据

`start` 将就绪元数据写入:

- `dist/extension-js/<browser>/ready.json`

这在使用 `--no-browser` 进行自动化时很有用:

- 在启动外部运行器前等待 `status: "ready"`。
- 将 `status: "error"` 视为确定性的失败信号。
- 使用 `runId` 和 `startedAt` 关联某次具体的运行时会话。

### `--no-browser` 与就绪同步

`--no-browser` 仅禁用浏览器启动。它不会阻塞外部运行器直到生产构建完成。

对于面向生产的 Playwright、持续集成(CI)和 AI 工作流:

1. 将 `extension start --no-browser` 作为生产者进程运行。
2. 将 `extension start --wait --browser=<browser>` 作为就绪闸门运行。
3. 只在 `status: "ready"` 之后再启动外部浏览器自动化。

`--wait` 在 `error`/超时时以非零状态退出,并会忽略来自已死亡进程(`pid` 已不再存活)的过期契约。
如果你在同一次调用中同时传入 `--wait` 和 `--no-browser`,`--wait` 优先。此时命令在 wait-only 模式下运行。

## 行为说明

- `start` 不运行开发服务器,也不提供热模块替换(HMR)或监听模式。
- `start` 面向生产模式;迭代式本地开发请使用 `dev`。
- `start` 不支持源代码检视选项。请使用 `dev --source ...` 实现该工作流。
- 对于机器消费者,解析 `dist/extension-js/<browser>/ready.json` 而不是终端文本。

## 后续步骤

- 用 [`dev`](/docs/commands/dev) 快速迭代。
- 用 [`preview`](/docs/commands/preview) 启动已有的构建输出。
