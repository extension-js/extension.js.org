当你希望一条命令完成「生产构建 + 立即启动浏览器」时使用 `start`。

`start` 命令会先运行生产构建，然后使用与 `preview` 相同的流程启动构建后的扩展。

## 何时使用 `start`

- 在编译之后手动校验生产行为。
- 复现监听模式产物与生产产物之间的运行时差异。
- 在本地做一次类生产校验，而不必先 `build` 再 `preview`。

## Start 命令能力

| 能力                       | 你能得到什么                                              |
| -------------------------- | --------------------------------------------------------- |
| 构建 + 启动工作流          | 一步完成生产编译与浏览器启动                              |
| 目标选择                   | 直接在所选浏览器或引擎目标上启动                          |
| 运行器控制                 | 当你只需要校验构建时，跳过浏览器启动                      |
| 类生产校验                 | 检查真实编译产物，而不是监听模式的状态                    |

## 与其他命令的区别

- `dev`：开发服务器 + 热模块替换 (HMR) / 监听模式
- `build`：仅生产构建
- `preview`：在不构建的情况下启动已有的构建产物
- `start`：依次执行 `build` + `preview`

## 自动化元数据

`start` 会把就绪元数据写入：

- `dist/extension-js/<browser>/ready.json`

在使用 `--no-browser` 的自动化场景中非常有用：

- 在启动外部运行器之前，等待 `status: "ready"`。
- 把 `status: "error"` 当作确定的失败信号处理。
- 用 `runId` 与 `startedAt` 关联具体的运行时会话。

### `--no-browser` 与就绪同步

`--no-browser` 只会禁用浏览器启动，并不会在生产构建完成之前阻塞外部运行器。

对面向生产的 Playwright、持续集成 (CI) 与 AI 工作流：

1. 把 `extension start --no-browser` 作为生产者进程启动。
2. 把 `extension start --wait --browser=<browser>` 作为就绪门控运行。
3. 仅在 `status: "ready"` 之后再启动外部浏览器自动化。

`--wait` 遇到 `error` / 超时会以非零状态码退出，并忽略来自已死进程的过期契约（`pid` 已不再存活）。
如果你在同一次调用中同时传入 `--wait` 与 `--no-browser`，`--wait` 优先。命令会以仅等待模式运行。

## 行为说明

- `start` 不会运行开发服务器，也不提供热模块替换 (HMR) 或监听模式。
- `start` 面向生产模式；本地迭代开发请使用 `dev`。
- `start` 不支持源码检查选项。该工作流请使用 `dev --source ...`。
- 对机器消费者，请解析 `dist/extension-js/<browser>/ready.json`，而不是终端文本。
