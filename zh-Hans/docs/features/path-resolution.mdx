按你习惯的方式书写路径，让 Extension.js 把它们规范化为运行时安全的输出路径。

当你移动文件、重命名入口或切换浏览器目标时，让运行时路径继续工作。Extension.js 会跨 manifest 字段与扩展 API 使用，重写所支持的路径。编译后的资源能在 `dist/<browser>` 中正确解析。

## 工作原理

路径解析在两个地方运行：

- **manifest 编译管线**：把 manifest 中声明的路径解析到产出的资源。
- **针对 JS/TS 的路径解析插件**：重写传递给支持的扩展 API（例如 `runtime.getURL`、`tabs.update`、`scripting.*`、`action.*`、`sidePanel.*`）的静态路径字面量。

## 支持的输入风格

Extension.js 会按以下规则规范化常见的路径输入：

| 输入风格                         | 示例                                                | 规范化后的输出                                                       |
| -------------------------------- | --------------------------------------------------- | -------------------------------------------------------------------- |
| 公共根目录（绝对）               | `/public/icons/icon.png`                            | `icons/icon.png`                                                     |
| 公共根目录（相对）               | `public/icons/icon.png`、`./public/icons/icon.png` | `icons/icon.png`                                                     |
| 根绝对路径                       | `/foo/bar.js`                                       | `foo/bar.js`                                                         |
| 根下的特殊文件夹                 | `/pages/popup.njk`                                  | `pages/popup.html`                                                   |
| 特殊文件夹                       | `pages/popup.html`、`scripts/content.tsx`          | `pages/popup.html`、`scripts/content.js`                            |
| 从作者文件出发的父级相对路径     | `../scripts/content.ts`                             | `scripts/content.js`（解析到项目根的特殊文件夹时）                  |

`pages/` 与 `scripts/` 的扩展名映射包括：

- `.ts`、`.tsx`、`.js`、`.jsx` → `.js`
- `.njk`、`.nunjucks`、`.html` → `.html`
- `.scss`、`.sass`、`.less`、`.css` → `.css`

## Extension.js 不会重写的情况

为避免误判，Extension.js 会有意跳过下列输入：

- `http://` 与 `https://` URL
- `data:` URL
- `chrome://` URL
- `moz-extension://` URL
- 通配符模式（`*`、`?`、`{}`、`[]`）
- 构建期无法安全解析的非静态/动态表达式

## manifest 输出示例

假设目标浏览器为 `chrome`：

| manifest 字段                  | 输入路径                    | 输出路径                       |
| ------------------------------ | --------------------------- | ------------------------------- |
| `action.default_popup`         | `/src/popup.html`           | `action/index.html`             |
| `background.page`              | `/src/background.html`      | `background/index.html`         |
| `background.service_worker`    | `/src/background.ts`        | `background/service_worker.js`  |
| `browser_action.default_popup` | `/src/popup.html`           | `action/index.html`             |
| `chrome_url_overrides.newtab`  | `/src/newtab.html`          | `newtab/index.html`             |
| `content_scripts.js`           | `/src/content_script.ts`    | `content_scripts/content-0.js`  |
| `content_scripts.css`          | `/src/content_style.css`    | `content_scripts/content-0.css` |
| `devtools_page`                | `/src/devtools.html`        | `devtools/index.html`           |
| `options_ui.page`              | `/src/options.html`         | `options/index.html`            |

| `side_panel.default_path`      | `/src/sidepanel.html`       | `sidebar/index.html`            |

Extension.js 会把所有编译产物输出到 `dist/<browser>` 下。

## 引用输出文件

对于 `chrome.runtime.getURL()` 这样的浏览器 API，优先使用稳定、输出根目录形式的路径：

```javascript
const iconUrl = chrome.runtime.getURL("/icons/icon.png");
console.log(iconUrl);
```

输出：`chrome-extension:<extension-id>/icons/icon.png`

## 诊断与安全检查

当解析器无法把引用路径解析到打包后的资源时（例如嵌套的 `src/pages/...`，或重写后缺失的 `public/` 文件），它会发出警告。  
Extension.js 在每次文件转换中会对警告做去重，以减少噪音。

## 重要的路径规则

- 起始的 `/` 表示扩展输出根目录，而不是文件系统根目录。
- `public/...` 与 `/public/...` 都会规范化为输出根目录下的资源。
- `pages/` 与 `scripts/` 是带有扩展名映射规则的特殊文件夹。
- Extension.js 会保留操作系统的绝对路径，而不是把它们当作扩展输出路径处理。
- 当无法安全确定最终打包路径时，Extension.js 会跳过动态表达式。

## 最佳实践

- 把 `pages/`、`scripts/` 与 `public/` 保留在项目根目录，以获得可预期的重写效果。
- 尽可能给扩展 API 路径使用静态字面量；Extension.js 可能跳过动态表达式。
- 把 `/public/...` 当作输出根目录下的资源，避免在运行时假设源相对路径。
- 在 `dev` 期间认真对待警告——它们通常预示着打包前的路径不匹配。
