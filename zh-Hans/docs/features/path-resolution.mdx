按你自然书写的方式写路径,让 Extension.js 把它们规范化为运行时安全的输出路径。

在你移动文件、重命名入口或更换浏览器目标时,让运行时路径仍然可用。Extension.js 会在 manifest 字段与扩展 API 使用之间统一重写支持的路径,让编译后的资源在 `dist/<browser>` 内正确解析。

## 工作原理

路径解析在两个地方运行:

- **manifest 编译流水线**:把 manifest 中声明的路径解析为输出资源。
- **JS/TS 路径解析插件**:重写传入受支持扩展 API 的静态路径字面量(例如 `runtime.getURL`、`tabs.update`、`scripting.*`、`action.*`、`sidePanel.*`)。

## 支持的输入风格

Extension.js 将常见的路径输入规范化如下:

| 输入风格                       | 示例                                                | 规范化输出                                                       |
| ------------------------------ | --------------------------------------------------- | ---------------------------------------------------------------- |
| 公共根目录(绝对路径)         | `/public/icons/icon.png`                            | `icons/icon.png`                                                 |
| 公共根目录(相对路径)         | `public/icons/icon.png`、`./public/icons/icon.png`  | `icons/icon.png`                                                 |
| 根绝对路径                     | `/foo/bar.js`                                       | `foo/bar.js`                                                     |
| 根级特殊文件夹                 | `/pages/popup.njk`                                  | `pages/popup.html`                                               |
| 特殊文件夹                     | `pages/popup.html`、`scripts/content.tsx`           | `pages/popup.html`、`scripts/content.js`                         |
| 相对于作者文件的父级路径       | `../scripts/content.ts`                             | `scripts/content.js`(当解析到项目根级特殊文件夹时)             |

对于 `pages/` 与 `scripts/` 的扩展名映射包括:

- `.ts`、`.tsx`、`.js`、`.jsx` → `.js`
- `.njk`、`.nunjucks`、`.html` → `.html`
- `.scss`、`.sass`、`.less`、`.css` → `.css`

## Extension.js 不会重写的情形

为避免误判,Extension.js 有意跳过以下输入:

- `http://` 与 `https://` URL
- `data:` URL
- `chrome://` URL
- `moz-extension://` URL
- 通配符模式(`*`、`?`、`{}`、`[]`)
- 构建无法安全解析的非静态/动态表达式

## Manifest 输出示例

假设目标浏览器为 `chrome`:

| Manifest 字段                  | 输入路径                   | 输出路径                        |
| ------------------------------ | -------------------------- | ------------------------------- |
| `action.default_popup`         | `/src/popup.html`          | `action/index.html`             |
| `background.page`              | `/src/background.html`     | `background/index.html`         |
| `background.service_worker`    | `/src/background.ts`       | `background/service_worker.js`  |
| `browser_action.default_popup` | `/src/popup.html`          | `action/index.html`             |
| `chrome_url_overrides.newtab`  | `/src/newtab.html`         | `newtab/index.html`             |
| `content_scripts.js`           | `/src/content_script.ts`   | `content_scripts/content-0.js`  |
| `content_scripts.css`          | `/src/content_style.css`   | `content_scripts/content-0.css` |
| `devtools_page`                | `/src/devtools.html`       | `devtools/index.html`           |
| `options_ui.page`              | `/src/options.html`        | `options/index.html`            |
| `sandbox.pages`                | `/src/sandbox.html`        | `sandbox/page-0.html`           |
| `side_panel.default_path`      | `/src/sidepanel.html`      | `sidebar/index.html`            |
| `sidebar_action.default_panel` | `/src/sidebar_panel.html`  | `sidebar/index.html`            |

并非每个 manifest 字段都映射到同名源文件的目录。Extension.js 基于浏览器对外的扩展契约来确定输出路径,而不是基于你原始的书写路径。

Extension.js 会把所有编译产物输出到 `dist/<browser>`。

## 引用输出文件

对于 `chrome.runtime.getURL()` 等浏览器 API,推荐使用稳定的输出根路径:

```javascript
const iconUrl = chrome.runtime.getURL("/icons/icon.png");
console.log(iconUrl);
```

输出:`chrome-extension:<extension-id>/icons/icon.png`

## 诊断与安全检查

当解析器无法把引用路径解析到已打包资源时(例如重写后的嵌套 `src/pages/...` 或缺失的 `public/` 文件),会向你发出警告。  
Extension.js 会按每次文件转换 pass 去重警告,以减少噪音。

## 重要的路径规则

- 起始的 `/` 表示扩展输出根目录,不是文件系统根目录。
- `public/...` 与 `/public/...` 会规范化为输出根资源。
- `pages/` 与 `scripts/` 是带扩展名映射规则的特殊文件夹。
- Extension.js 会保留绝对的操作系统文件路径,不会把它们当作扩展输出路径处理。
- 当无法安全确定最终打包路径时,Extension.js 会跳过动态表达式。

## 最佳实践

- 把 `pages/`、`scripts/` 与 `public/` 保留在项目根目录,以获得可预测的重写。
- 在扩展 API 路径上尽量使用静态字面量;Extension.js 可能会跳过动态表达式。
- 把 `/public/...` 视为输出根资源,避免在运行时假设源相对路径。
- 在 `dev` 期间认真对待警告,它们通常意味着打包前的路径不匹配。
