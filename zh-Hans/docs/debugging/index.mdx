浏览器扩展难以调试，是因为有意思的状态藏在不易触及的地方：一个会进入空闲的 MV3 service worker、一个隔离的内容脚本世界、一个失去焦点就立刻关闭的 popup。Extension.js 在你正在运行的开发会话中打开一个小小的 **本地控制通道**，让你（或一个 AI 智能体、或一个 CI 任务）能够直接到达这些上下文 —— 读取它们打出的日志、查看它们渲染出的内容、调用它们，并触发用户会触发的事件。

它是本地的，无需账号，观察始终免费。任何会 _改变_ 状态的操作都按会话显式启用。

## 启用方式

使用所需的控制标志启动开发会话：

```bash
# 只读模式始终开启。加上 control 才能对扩展执行操作：
pnpm extension dev --allow-control

# 再加上 eval 才能在某个上下文中运行任意表达式：
pnpm extension dev --allow-control --allow-eval
```

下面的每一个操作都针对该会话，并以同样的方式定位上下文 —— 不管你是在读取还是在执行操作。

## 你可以做什么

| 命令                                       | 作用                                                                                                                | 门控              |
| ------------------------------------------ | ------------------------------------------------------------------------------------------------------------------- | ----------------- |
| `extension logs`                           | 在一个有序时间线中流式输出每个上下文（SW、content、popup、options、sidebar、devtools）的日志                          | —                 |
| `extension inspect`                        | 读取某个面注入后的实时 DOM                                                                                          | —                 |
| `extension storage get \| set`             | 读取或写入你的扩展的 `chrome.storage`                                                                               | `--allow-control` |
| `extension reload`                         | 重新加载扩展或某个标签页                                                                                            | `--allow-control` |
| `extension open <popup\|options\|sidebar>` | 打开扩展的某个面                                                                                                    | `--allow-control` |
| `extension open action`                    | [触发工具栏 action](/docs/debugging/trigger-actions-and-commands) —— 打开其 popup，或重放 `onClicked`                | `--allow-control` |
| `extension open command --name <cmd>`      | [重放某个键盘快捷键命令](/docs/debugging/trigger-actions-and-commands)                                                | `--allow-control` |
| `extension eval "<expr>" --context <c>`    | 在某个上下文中对表达式求值并返回结果                                                                                | `--allow-eval`    |

## 定位一个上下文

读取与操作共用同一套词汇 —— 你说出面的名称，Extension.js 会根据它正在跟踪的会话来解析。

| 地址                                                | 含义                                                |
| --------------------------------------------------- | --------------------------------------------------- |
| `--context background`                              | service worker（或 MV2 的后台页面）                 |
| `--context popup` / `options` / `sidebar`           | 如果已打开，则是那个扩展面                          |
| `--context content --url "https://shop.example/*"`  | 在匹配页面上的隔离内容世界                          |
| `--context content --tab 7`                         | 指定标签页中的内容世界                              |

## 示例：我的后台处理器到底有没有跑？

```bash
pnpm extension eval "globalThis.__lastSyncAt" --context background --allow-eval
```

```json
{ "ok": true, "value": 1718000000000 }
```

你的表达式触发的任何 `console.log` 都会同时通过 `extension logs` 流出，并按序号相关联 —— 因此你能 _同时_ 看到返回值和副作用。

## 跨浏览器支持

Extension.js 通过 **浏览器内伴随程序** 而非 Chrome DevTools Protocol 进行调试，因此核心循环可以在 **Chrome 和 Firefox** 上都触达你自己的面 —— 以前那道“Firefox 用 RDP，不支持”的墙在这些工具上已经被推倒。

| 能力                                                                              | Chromium | Firefox      |
| --------------------------------------------------------------------------------- | -------- | ------------ |
| `logs`、`inspect`、`eval`、`storage`、`reload`、`open`（含 `action`/`command`）   | ✓        | ✓（已验证）  |
| `inspect --deep-dom`（关闭式 shadow root）                                        | ✓        | —（使用 CDP）|
| `extension_list_extensions`（MCP 工具）                                           | ✓        | —（使用 CDP）|

（`extension_list_extensions` 是一个 MCP 工具而非 CLI 动词 —— 它通过 DevTools Protocol 连接，因此仅限 Chromium。）

## 安全

这些门控是有意为之，而非官僚主义：

- **观察无需任何门控。** 读取日志和 DOM 始终可用。
- **有边界的操作需要 `--allow-control`。** `storage`、`reload` 和 `open` 会改变状态，因此按会话显式启用。
- **`eval` 需要 `--allow-eval` _和_ 一个按会话生成的令牌。** 该令牌写入 `dist/` 之外的 `0600` 文件，因此不会进入构建产物 —— 本地的随机进程无法悄悄驱动你的 service worker。
- **不会出现在生产环境。** 控制通道仅在 `dev`/`preview` 期间存在；它依赖于一个不会出现在已构建包中的端口。

## 与 AI 智能体协作

同样的操作通过 [`@extension.dev/mcp`](/docs/integrations/chrome-devtools-mcp) 以 MCP 工具形式暴露（`extension_logs`、`extension_eval`、`extension_storage`、`extension_reload`、`extension_open`、`extension_list_extensions`）。门控完全相同 —— 助手可以自由观察，但只有在你为该会话启用后才能执行操作。
