你无法在 CI 中点击工具栏图标,而没有弹窗的 action 只有在其 `onClicked` 触发时才会做事。Extension.js 让你可以直接触发这些处理程序 —— 工具栏 action 和键盘快捷键命令 —— 让你能以无头方式调用它们,并对它们的行为做断言。

这之所以能工作,是因为 Extension.js 在构建时捕获你的 `chrome.action.onClicked` 和 `chrome.commands.onCommand` 监听器(开发运行时在你的后台代码之前加载),并按需重放它们。没有窗口、没有鼠标、不会抖动。

<Note>
  需要使用 `--allow-control` 启动的开发会话 —— 参见 [调试概览](/docs/debugging)。
</Note>

## 触发工具栏 action

```bash
pnpm extension open action --allow-control --output json
```

如果该 action 有 `default_popup`,这会打开它。如果没有,它会重放 `onClicked` 监听器:

```json
{
  "ok": true,
  "value": { "triggered": "onClicked", "listeners": 1, "gesture": false }
}
```

## 触发键盘命令

按名称重放任何 `chrome.commands.onCommand` 处理程序 —— 快捷键本身永远不会被按下:

```bash
pnpm extension open command --name toggle-theme --allow-control --output json
```

```json
{
  "ok": true,
  "value": {
    "triggered": "command",
    "command": "toggle-theme",
    "listeners": 1,
    "gesture": false
  }
}
```

## 唯一的注意事项:没有用户手势

重放会调用你的处理程序,但它 **不是** 真实的用户点击,因此浏览器不会附加用户手势。实际后果是:

<Warning>
  真实的工具栏点击会授予临时的 **`activeTab`**,并满足受手势门控的 API
  (`chrome.permissions.request`、交互式
  `chrome.identity.getAuthToken`)。重放不会。所以依赖
  `activeTab` 的处理程序 —— 比如对活动标签页的 `chrome.scripting.executeScript`、
  `chrome.tabs.captureVisibleTab` —— 行为会与真实点击不同。
</Warning>

Extension.js 在这点重要时会告知你:结果会报告 `gesture: false`,并在你的清单声明了 `activeTab` 时添加 `warning`:

```json
{
  "ok": true,
  "value": {
    "triggered": "onClicked",
    "listeners": 1,
    "gesture": false,
    "warning": "replayed without a user gesture: activeTab is NOT granted, so APIs that depend on it (scripting on the active tab, captureVisibleTab) behave differently than a real click"
  }
}
```

对于大多数处理程序逻辑 —— 切换状态、写入存储、打开标签页、消息传递 —— 重放正是你想要的:确定性且可脚本化。仅在你确实需要手势语义时,才使用下面的真实点击。

## 需要真正的手势点击?

真实的 action 点击会授予 `activeTab` —— 已在 Chrome 149 上确认。重放故意 **不** 携带该手势(因此依赖 `activeTab` 的处理程序行为不同)。对于罕见的需要真实手势语义的情况,请使用 [chrome-devtools-mcp](https://github.com/ChromeDevTools/chrome-devtools-mcp) 的 `trigger_extension_action`(仅 Chromium),它通过受支持的 pipe 传输由 Puppeteer 驱动真实调用。

我们仍然推荐 [与 Extension.js 并行运行它](/docs/integrations/chrome-devtools-mcp):让它负责真实手势的浏览器驱动,而 Extension.js 负责你用于其他一切的、构建感知的、跨浏览器、无头的重放。

## 在 CI 中:断言你的 action 处理程序已运行

启动一个会话,触发 action,并对副作用进行断言 —— 没有浏览器 UI,没有 Playwright。这就是 GitHub Actions 作业中的完整循环:

```yaml
- run: pnpm extension dev --browser=chromium --allow-control --wait --wait-format=json
- run: pnpm extension open action --browser=chromium --output json
- run: |
    pnpm extension storage get --key lastClickedAt --browser=chromium --output json \
      | node -e 'process.exit(JSON.parse(require("fs").readFileSync(0)).value?.lastClickedAt ? 0 : 1)'
```

因为触发是确定性且无头的,所以它是一个可靠的门控 —— Extension.js 自己的 `smoke:open-action` 检查也走相同的路径,它会触发 action _以及_ 一个命令,然后读取 `chrome.storage` 来证明每个处理程序确实执行了。

参见 [CI 模板](/docs/workflows/ci-templates#test-handlers-without-a-browser-ui) 了解完整流水线。

## 跨浏览器

`open action` 和 `open command` 构建于浏览器内伴侣之上,因此它们可以在 **Chrome 和 Firefox** 上运行 —— 它们只触碰 `addListener`,没有任何引擎特定的东西,并已在两者上验证。同一份 `background.service_worker` 源也适用于 Firefox:Extension.js 会将其翻译为 Firefox 构建的 `background.scripts` 事件页面。(真正的手势点击仅限 Chromium,位于 chrome-devtools-mcp 中 —— 见上文。)
