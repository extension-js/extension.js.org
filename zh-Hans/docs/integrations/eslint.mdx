在问题进入运行时之前，让扩展代码质量保持可预期。ESLint 运行在构建管线之外。请从你的脚本或持续集成（CI）中运行它。

## 何时适合使用 ESLint

- 你希望在团队协作中保持一致的代码规范。
- 你希望用 CI 检查尽早捕获常见的 JS/TS 错误。
- 你希望在发布前强制执行扩展特定的质量规则。

## ESLint 能力

| 能力                              | 你能得到什么                                                            |
| --------------------------------- | ----------------------------------------------------------------------- |
| JavaScript 与 TypeScript lint     | 捕获常见错误并强制项目规范                                              |
| Flat config 支持                  | 在新项目中使用现代的 `eslint.config.mjs`                                |
| CI 友好                           | 在本地脚本与 Pull Request 流水线中运行 lint 检查                        |
| 扩展特定的规则集                  | 在团队需要更严格检查时添加浏览器扩展相关的插件/规则                     |

## 在已有扩展中使用

安装核心依赖：

<PackageManagerTabs command="install -D eslint @eslint/js globals typescript-eslint" />

创建 `eslint.config.mjs`：

```js
import globals from "globals";
import pluginJs from "@eslint/js";
import tseslint from "typescript-eslint";

export default [
  { files: ["**/*.{js,mjs,cjs,ts,tsx,jsx}"] },
  {
    languageOptions: {
      globals: globals.browser,
    },
  },
  pluginJs.configs.recommended,
  ...tseslint.configs.recommended,
];
```

从你的脚本中运行 lint 命令：

<CodeGroup>

```bash npm
npm run lint
```

```bash pnpm
pnpm lint
```

```bash yarn
yarn lint
```

```bash bun
bun run lint
```

```bash bun
bun run lint
```

</CodeGroup>

## 最佳实践

- 把 lint 规则纳入源码管理，并在 CI 中执行。
- 使用 lint-staged 或 pre-commit 钩子获得更快的本地反馈。
- 仅在需要时添加框架特定的插件（例如 React）。
