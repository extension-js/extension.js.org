以最少的手工操作保持扩展代码库的格式化一致。Prettier 通过脚本、编辑器集成和持续集成(CI)运行。它不在 Extension.js 构建流水线中运行。

## 何时适合使用 Prettier

- 你希望本地和 CI 工作流中拥有确定性的格式化。
- 你在接入贡献者,需要低摩擦的样式一致性。
- 你希望 ESLint 专注于正确性,而 Prettier 处理格式化。

## Prettier 能力

| 能力                       | 提供给你的功能                                            |
| -------------------------- | --------------------------------------------------------- |
| 一致的格式化               | 为 JS、TS、JSON、CSS 和 Markdown 提供一种共享样式         |
| 脚本和编辑器工作流         | 在本地保存时格式化并在 CI 中强制执行                      |
| 减少 lint 冲突             | 与 `eslint-config-prettier` 配对以避免规则重叠            |
| 低维护的默认值             | 以最少配置采用稳定基线                                    |

## 在现有扩展中使用

安装 Prettier:

<PackageManagerTabs command="install -D prettier" />

创建一个配置文件(例如 `.prettierrc`):

```json
{
  "semi": false,
  "singleQuote": true,
  "trailingComma": "none",
  "printWidth": 80,
  "tabWidth": 2,
  "arrowParens": "always"
}
```

从脚本或手动运行 Prettier:

<CodeGroup>

```bash npm
npm run format
```

```bash pnpm
pnpm format
```

```bash yarn
yarn format
```

```bash bun
bun run format
```

```bash bun
bun run format
```

</CodeGroup>

如果尚无 format 脚本:

## 与 ESLint 集成

为避免在 ESLint 中出现格式化规则冲突,请安装:

<PackageManagerTabs command="install -D eslint-config-prettier" />

然后在 ESLint 配置链中添加 `eslint-config-prettier`,让 ESLint 把格式化关注点交给 Prettier。

## 最佳实践

- 在编辑器和 CI 中自动执行格式化,而不是手动操作。
- 在 CI 中使用 `--check`,在本地使用 `--write`。
- 在更大型仓库中,在提交前格式化暂存文件。
