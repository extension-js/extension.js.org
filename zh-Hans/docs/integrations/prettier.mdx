以最小的手动工作量在扩展代码库中保持一致的格式化。Prettier 通过脚本、编辑器集成和持续集成(CI)运行。它不在 Extension.js 构建管道中运行。

## 何时适合使用 Prettier

- 你希望在本地和 CI 工作流中有确定性的格式化。
- 你正在引入贡献者,需要低阻力的风格一致性。
- 你希望 ESLint 专注于正确性,而 Prettier 处理格式化。

## Prettier 能力

| 能力                       | 它能为你提供什么                                              |
| -------------------------- | ------------------------------------------------------------- |
| 一致的格式化               | 为 JS、TS、JSON、CSS 和 Markdown 提供一种共享风格             |
| 脚本和编辑器工作流         | 在本地保存时格式化,在 CI 中强制执行                          |
| 减少 lint 冲突             | 与 `eslint-config-prettier` 搭配以避免规则重叠                |
| 低维护的默认值             | 以最少的配置采用稳定的基线                                    |

## 与现有扩展一起使用

安装 Prettier:

<PackageManagerTabs command="install -D prettier" />

创建一个配置文件(例如 `.prettierrc`):

```json
{
  "semi": false,
  "singleQuote": true,
  "trailingComma": "none",
  "printWidth": 80,
  "tabWidth": 2,
  "arrowParens": "always"
}
```

从脚本或手动运行 Prettier:

<CodeGroup>

```bash npm
npm run format
```

```bash pnpm
pnpm format
```

```bash yarn
yarn format
```

```bash bun
bun run format
```

```bash bun
bun run format
```

</CodeGroup>

如果还没有 format 脚本:

## 与 ESLint 集成

为避免 ESLint 中的格式化规则冲突,安装:

<PackageManagerTabs command="install -D eslint-config-prettier" />

然后在你的 ESLint 配置链中添加 `eslint-config-prettier`,让 ESLint 将格式化交给 Prettier 处理。

## 最佳实践

- 让格式化在编辑器和 CI 中自动进行,而不是手动。
- 在 CI 中使用 `--check`,在本地使用 `--write`。
- 在较大的仓库中,在 pre-commit 时格式化已暂存的文件。
