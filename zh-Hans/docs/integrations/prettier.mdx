以最小的手动成本让扩展代码库保持一致的格式。Prettier 通过脚本、编辑器集成与持续集成（CI）运行。它不运行在 Extension.js 的构建管线里。

## 何时适合使用 Prettier

- 你希望在本地与 CI 工作流中获得确定性的格式化。
- 你正在引入新贡献者，需要低摩擦的风格一致性。
- 你希望 ESLint 专注于正确性，由 Prettier 负责格式化。

## Prettier 能力

| 能力                       | 你能得到什么                                              |
| -------------------------- | --------------------------------------------------------- |
| 一致的格式                 | 让 JS、TS、JSON、CSS、Markdown 共用一种风格                |
| 脚本与编辑器工作流         | 在本地保存时自动格式化，并在 CI 中强制执行                |
| 减少 lint 冲突             | 搭配 `eslint-config-prettier` 避免规则重叠                |
| 低维护成本的默认值         | 用极少的配置采用一套稳定基线                              |

## 在已有扩展中使用

安装 Prettier：

<PackageManagerTabs command="install -D prettier" />

创建一个配置文件（例如 `.prettierrc`）：

```json
{
  "semi": false,
  "singleQuote": true,
  "trailingComma": "none",
  "printWidth": 80,
  "tabWidth": 2,
  "arrowParens": "always"
}
```

通过脚本运行 Prettier，或者手动运行：

<CodeGroup>

```bash npm
npm run format
```

```bash pnpm
pnpm format
```

```bash yarn
yarn format
```

```bash bun
bun run format
```

```bash bun
bun run format
```

</CodeGroup>

如果还没有 format 脚本：

## 与 ESLint 集成

为避免 ESLint 中的格式化规则冲突，请安装：

<PackageManagerTabs command="install -D eslint-config-prettier" />

然后在 ESLint 的配置链中加入 `eslint-config-prettier`，让 ESLint 把格式化相关的事交给 Prettier。

## 最佳实践

- 让格式化在编辑器与 CI 中自动进行，而不是手动操作。
- 在 CI 中使用 `--check`，在本地使用 `--write`。
- 在更大的仓库中，提交前格式化暂存文件。
