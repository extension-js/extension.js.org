[chrome-devtools-mcp](https://github.com/ChromeDevTools/chrome-devtools-mcp) 让 AI 代理可以直接控制一个活跃的 Chrome 实例,最近的版本还加入了扩展工具(`install`、`list`、`reload`、`trigger` 和 `uninstall`,以及 service-worker 日志)。Extension.js MCP 服务器(`@extension.dev/mcp`)覆盖了相同的扩展能力,但是感知构建的并且跨浏览器。

它们是互补的,而不是竞争关系。两者都运行:让 chrome-devtools-mcp 驱动 Chrome 本身,让 Extension.js 拥有所有依赖你项目的内容——构建、热重载、生成的 manifest、存储、跨上下文的日志,以及在 Firefox、Edge 和 Safari 上相同的工作流。

## 何时使用哪个

| 任务                                                              | 使用                                                       |
| ----------------------------------------------------------------- | ---------------------------------------------------------- |
| 构建、开发服务器、热重载、生产打包                                | Extension.js(`extension_dev`、`extension_build`)          |
| 读取生成的、跨浏览器的 manifest                                   | Extension.js(`extension_manifest_validate`)               |
| 以可脚本化方式回放工具栏 action 处理器,无需 popup                | Extension.js(`extension_open` 配 `surface: \"action\"`)   |
| 回放键盘快捷键命令处理器                                          | Extension.js(`extension_open` 配 `surface: \"command\"`)  |
| 在开发浏览器中列出具有实时上下文的扩展                            | Extension.js(`extension_list_extensions`)                 |
| 跨 SW、内容脚本、popup、options、sidebar 的日志                   | Extension.js(`extension_logs`)                            |
| 读/写 `chrome.storage`,在任意上下文中求值                        | Extension.js(`extension_storage`、`extension_eval`)       |
| 在 Firefox、Edge 或 Safari 上使用相同流程                         | Extension.js(原生跨浏览器)                                |
| 驱动任意网页,点击/滚动,填写表单                                | chrome-devtools-mcp                                        |
| 性能跟踪、Lighthouse 审计、网络检查                              | chrome-devtools-mcp                                        |
| 检查你没有构建过的扩展(没有项目)                               | chrome-devtools-mcp                                        |
| 通过**真实手势**触发 action(获得 `activeTab`)                  | chrome-devtools-mcp(`trigger_extension_action`)           |

一条有用的经验法则:如果任务涉及你的**项目**(源代码、构建、manifest、重载),使用 Extension.js。如果任务通用地涉及**浏览器**(页面、网络、性能),使用 chrome-devtools-mcp。

## 同时安装两个服务器

Extension.js 工具以 `extension_*` 命名空间,所以它们绝不会与 chrome-devtools-mcp 工具冲突。熟悉其中一个的代理可以迁移到另一个。

<CodeGroup>

```bash Claude Code
# Extension.js — 感知构建的扩展控制
claude mcp add extension-dev npx @extension.dev/mcp

# chrome-devtools-mcp — 原始 Chrome 调试(扩展工具选择性启用)
claude mcp add chrome-devtools npx chrome-devtools-mcp@latest --categoryExtensions
```

```json Claude Desktop (claude_desktop_config.json)
{
  "mcpServers": {
    "extension-dev": {
      "command": "npx",
      "args": ["@extension.dev/mcp"]
    },
    "chrome-devtools": {
      "command": "npx",
      "args": ["chrome-devtools-mcp@latest", "--categoryExtensions"]
    }
  }
}
```

```json Cursor (.cursor/mcp.json)
{
  "mcpServers": {
    "extension-dev": {
      "command": "npx",
      "args": ["@extension.dev/mcp"]
    },
    "chrome-devtools": {
      "command": "npx",
      "args": ["chrome-devtools-mcp@latest", "--categoryExtensions"]
    }
  }
}
```

</CodeGroup>

<Note>
  chrome-devtools-mcp 的扩展工具(`--categoryExtensions`)目前需要管道连接。
  在 Chrome 149 之前,通过 WebSocket 端点(`browserUrl` / `wsEndpoint`)
  附加到一个已经运行的浏览器对扩展类别不受支持。Extension.js 连接到它为
  你的开发会话启动的浏览器,所以它的扩展工具今天就能用。
</Note>

## 能力映射

每个 chrome-devtools-mcp 扩展动词都有 Extension.js 的对应项,以及围绕它的构建平台。

| chrome-devtools-mcp        | Extension.js                              | 说明                                                                                                                                                                                                          |
| -------------------------- | ----------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `install_extension`        | `extension_dev`、`extension_preview`      | 加载与真实的构建 + 热重载绑定,而非静态文件夹                                                                                                                                                                 |
| `list_extensions`          | `extension_list_extensions`               | 列出具有实时上下文的扩展;通过 Extensions 域只读                                                                                                                                                              |
| `reload_extension`         | `extension_reload`                        | 感知重建;重新加载后台或标签页                                                                                                                                                                                |
| `trigger_extension_action` | `extension_open`(`surface: \"action\"`)  | 跨浏览器、**无手势**的 `onClicked` 处理器可移植回放(参见注意事项)。对于真实手势点击(activeTab),使用 chrome-devtools-mcp 的 `trigger_extension_action`                                                    |
| —                          | `extension_open`(`surface: \"command\"`) | 回放 `chrome.commands.onCommand` 键盘快捷键处理器——chrome-devtools-mcp 无对应                                                                                                                                |
| `uninstall_extension`      | 由开发会话生命周期管理                    | Extension.js 通过 `dev` / `preview` 拥有加载/卸载                                                                                                                                                             |
| service-worker 日志        | `extension_logs`                          | 一条贯穿所有上下文的有序时间线,带 `follow`                                                                                                                                                                   |

### action / command 触发是如何工作的——以及它们唯一的注意事项

chrome-devtools-mcp 通过点击真实的工具栏按钮来触发 action,这需要可见的窗口和真实的用户手势。Extension.js 采用了不同的方法:它在构建时捕获扩展的 `chrome.action.onClicked`(以及 `chrome.commands.onCommand`)监听器,并在按需时**回放**它们。这让触发可脚本化、可复现——这是代理测试的自然模式——而且因为它只接触 `addListener`,所以它在 **Chromium 和 Firefox** 上都能工作(两者均已验证)。单个 `background.service_worker` 源也能在 Firefox 上工作——Extension.js 为 Firefox 目标把它转换为 `background.scripts` 事件页面,因为 Firefox 不运行 service-worker 后台。

<Warning>
  回放**没有用户手势**地调用你的处理器。真实的工具栏点击会授予临时
  `activeTab` 权限并满足受手势门控的 API(`chrome.permissions.request`、
  交互式 `identity.getAuthToken`);回放则不会。所以,依赖 `activeTab`
  的处理器(例如对活动标签页执行 `chrome.scripting.executeScript`
  或 `captureVisibleTab`)的行为会与真实点击不同。结果包含 `gesture:
  false`,当你的 manifest 声明 `activeTab` 时还会有 `warning`。
  当你需要真实点击保真度时,请使用 chrome-devtools-mcp 的
  `trigger_extension_action`(见下文)。
</Warning>

### 需要真实手势点击?使用 chrome-devtools-mcp

两者是互补的,而不是竞争关系:

- **`extension_open`(`surface: \"action\"`)** — 日常路径。回放处理器,跨浏览器、无头、无手势(不授予 `activeTab`)。把它用于绝大多数处理器测试。
- **chrome-devtools-mcp `trigger_extension_action`** — 通过 Puppeteer(经由受支持的管道传输)进行的_真实_ action 调用,所以 `activeTab` 会像用户点击一样被授予。仅限 Chromium。仅在你的处理器确实依赖手势时使用。

由于本页已经推荐同时运行两个服务器,就让 chrome-devtools-mcp 拥有真实手势的浏览器驱动,而让 Extension.js 拥有感知构建的、跨浏览器的回放——无需重复。

参见 [重载与 HMR](/docs/features/reload-and-hmr)。

### 构造上的安全

回放包装**只注入到你的开发构建中**——代理桥接由一个仅在 `extension dev`/`preview` 期间存在的控制端口门控,生产构建中不添加任何内容。`addListener` 包装会透明地委托给原始实现,所以你的扩展在有或没有桥接时行为完全相同。
