在扩展界面之间保持样式表质量一致,并尽早捕获 CSS 问题。Stylelint 通过脚本或持续集成(CI)运行,不在 Extension.js 的 dev/构建管道中运行。

## 何时适合使用 Stylelint

- 你在多个扩展界面之间维护共享的 CSS/Sass/Less。
- 你希望在 pull request 合并前进行样式质量检查。
- 你在团队工作流中需要一致的样式表约定。

## Stylelint 能力

| 能力                  | 它能为你提供什么                                       |
| --------------------- | ------------------------------------------------------ |
| CSS 质量检查          | 捕获无效模式并强制执行样式约定                         |
| 预处理器支持          | 在一个工作流中检查 CSS、Sass 和 Less                   |
| 脚本和 CI 集成        | 在合并和发布前运行样式检查                             |
| 与格式化器对齐        | 与 Prettier 结合以减少样式工具冲突                     |

## 与现有扩展一起使用

安装 Stylelint 依赖:

<PackageManagerTabs command="install -D stylelint stylelint-config-standard-scss" />

### Stylelint 配置

在项目根目录创建 Stylelint 配置文件(例如 `.stylelintrc.json`):

```json
{
  "extends": "stylelint-config-standard-scss",
  "rules": {
    "color-no-invalid-hex": true,
    "block-no-empty": true
  }
}
```

### 运行 Stylelint

<CodeGroup>

```bash npm
npm run lint:styles
```

```bash pnpm
pnpm lint:styles
```

```bash yarn
yarn lint:styles
```

```bash bun
bun run lint:styles
```

```bash bun
bun run lint:styles
```

</CodeGroup>

## 与 Prettier 集成

将 Stylelint 与 Prettier 集成,以同时格式化和检查样式。安装以下依赖:

<PackageManagerTabs command="install -D stylelint-prettier prettier" />

然后,更新你的 `.stylelintrc.json` 文件以包含 Prettier:

```json
{
  "extends": [
    "stylelint-config-standard-scss",
    "stylelint-prettier/recommended"
  ]
}
```

这能让样式格式化和检查保持一致。

## 最佳实践

- 保留专门的 `lint:styles` 命令并在 CI 中运行。
- 在 monorepo 中跨扩展包使用一份共享配置。
- 将 Stylelint 与 Prettier 搭配,让样式格式化保持确定性。

## 检测说明

Extension.js 可以检测 Stylelint 配置文件用于诊断/报告,但它不会将 Stylelint 注入到构建管道中。
