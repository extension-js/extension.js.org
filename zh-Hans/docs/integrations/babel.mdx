只有当你需要 SWC（Speedy Web Compiler）覆盖不到的转换或插件时才引入 Babel。把 SWC 当作默认管线，再把 Babel 限定在确实需要它的文件上。

## 何时适合用 Babel

- 你依赖只在 Babel 上才有的插件或 preset。
- 你正在把一个重度依赖 Babel 的既有项目逐步迁移过来。
- 你希望对特定文件类型（例如 MDX）做有针对性的转换。

## Babel 能力

| 能力                                          | 你能得到什么                                                                                   |
| --------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| 定向转换                                      | 仅为特定文件类型（例如 MDX）跑 Babel，不需要替换整个构建管线                                   |
| 接入插件生态                                  | 复用既有 Web 项目里只在 Babel 上才有的 preset/plugin                                           |
| 渐进式迁移                                    | 以更小的步子把遗留的 Babel 扩展项目迁移到 Extension.js                                         |
| Rspack（Extension.js 打包器）集成             | 通过 `extension.config.*` 的 `config` 钩子配置 Babel                                           |

## 配置示例

安装 Babel 依赖：

<PackageManagerTabs command="install -D @babel/core @babel/preset-env babel-loader" />

创建 `babel.config.json`：

```json
{
  "presets": ["@babel/preset-env"]
}
```

在 `extension.config.js` 中仅为 MDX 接入 Babel：

```js
export default {
  config: (config) => {
    config.module.rules.push({
      test: /\.mdx$/,
      use: ["babel-loader", "@mdx-js/loader"],
    });
    return config;
  },
};
```

## 仅在必要时为 JS/TS 替换 SWC

只有当你确实需要让 Babel 直接处理 JS/TS 入口文件时，才使用这种模式：

```js
export default {
  config: (config) => {
    config.module.rules = [
      {
        test: /\.[jt]sx?$/,
        exclude: /node_modules/,
        use: [{ loader: "babel-loader" }],
      },
      ...config.module.rules.filter(
        (rule) => !String(rule?.test || "").includes("[jt]sx"),
      ),
    ];
    return config;
  },
};
```

## 最佳实践

- 优先做有针对性的 Babel 使用，而不是替换默认的 JS/TS 管线。
- 让 Babel 规则保持显式，使 loader 顺序可预期。
- 仅在迁移或插件兼容性需要时复用 Babel。
