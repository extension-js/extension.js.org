通过先检查最常见的失败原因来快速解开本地开发的阻塞。在深入调试之前，请将本页面作为实用的分诊路径。

## 故障排查能力

| 领域               | 它能帮助你验证什么                                          |
| ------------------ | ----------------------------------------------------------- |
| 重启诊断           | 结构性改动是否需要重启                                      |
| 依赖配置           | 当前技术栈是否缺少可选工具                                  |
| 浏览器启动         | 目标/浏览器二进制配置是否有效                               |
| 构建/预览流程      | 预览所需的输出是否存在并与目标匹配                          |
| Manifest 路径      | 引用的文件是否能正确解析                                    |

## 快速分诊流程

1. 用一个目标复现（`--browser=chromium`）。
2. 移除自定义配置文件和二进制标志。
3. 确认问题是出现在 `dev`、`build` 还是两者中。
4. 一次只修一类错误（重启、依赖、路径或启动）。

## 快速诊断决策树

| 如果这个失败...           | 先检查                                                | 下一步操作                                                                                  |
| ------------------------- | ----------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| 扩展无法加载              | 构建和 manifest 错误                                  | 验证引用的文件并重新运行 `dev`                                                              |
| 浏览器无法启动            | `--browser` + 二进制路径                              | 移除自定义二进制/配置文件标志并重试                                                         |
| 改动未更新                | 更新类型（热模块替换 vs 需要重启）                    | 查看 [Dev 更新行为](/docs/workflows/dev-update-behavior)，必要时重启                        |
| `preview` 失败            | 现有的 `dist/<browser>` 输出                          | 先运行 `extension build --browser=<target>`                                                 |
| 运行时仍然有问题          | 最小可复现配置                                        | 缩小范围并捕获准确的命令 + 错误                                                             |

## 常见问题

### 需要重启的诊断

如果你看到需要重启的错误，请停止并重新运行 `extension dev`。

典型触发条件：

- Manifest 入口点列表变更。
- 在 `pages/` 或 `scripts/` 中添加/移除文件。
- HTML 中结构性的 script/style 入口变更。

完整对照表请参见 [Dev 更新行为](/docs/workflows/dev-update-behavior)。

### 缺少可选依赖

CLI 会按需安装一些集成（例如 Vue/Preact 刷新工具、样式预处理器）。

如果 CLI 提示安装可选依赖：

1. 允许安装。
2. 重启 `extension dev`。
3. 依赖安装完成后再运行一次。

### 浏览器二进制问题

如果目标浏览器启动失败：

- 验证自定义二进制路径（`--chromium-binary` / `--gecko-binary`）。
- 确认浏览器目标与提供的二进制兼容。
- 回退到默认的 `chromium` 目标以隔离配置问题。

### 构建可以运行但预览失败

`preview` 期望存在已有的构建输出。

- 运行 `extension build --browser=<target>`。
- 然后运行 `extension preview --browser=<target>`。

### Manifest 引用错误

如果构建报告缺少 manifest 文件：

- 验证路径是否相对于 manifest 位置。
- 验证开头 `/` 的使用（public 根语义）。
- 确保文件确实存在于源/public 位置。

### Docker、开发容器和 Codespaces

当在容器内运行时，dev 服务器默认绑定到 `127.0.0.1`，因此主机无法访问。

- 传入 `--host 0.0.0.0` 以绑定到所有接口。
- 如果 `8080` 被占用，请使用 `--port 0` 让操作系统分配可用端口。
- 如果在持续集成（CI）容器中浏览器连接缓慢或不稳定，请使用[浏览器传输调优变量](/docs/features/environment-variables#browser-transport-tuning-variables)来增加连接超时。

```bash
extension dev --host 0.0.0.0 --port 0
```

### `scripts/` 文件夹中的 Node.js 脚本

如果构建失败并报错 `scripts/ is a reserved folder in Extension.js`，说明你在 `scripts/` 特殊文件夹中放了一个 Node.js 文件。Extension.js 会用一个浏览器内容脚本运行时包裹 `scripts/` 中的每个文件。仅用于 Node.js 的文件在此上下文中会失败。

将该文件移动到项目根目录下的其他文件夹（例如 `bin/`、`tools/` 或 `ci-scripts/`）。详见[特殊文件夹](/docs/features/special-folders#nodejs-scripts-are-not-allowed-in-scripts)。

### 修复后仍然受阻

如果问题依然存在：

- 清除临时假设并从一个干净的终端会话开始复现。
- 缩小到仍能复现失败的最小 manifest/入口点用例。
- 捕获准确的命令 + 错误输出以便提交 issue。

## 调试检查清单

- 先用单一浏览器目标复现（`--browser=chromium`）。
- 在不带自定义配置文件/二进制标志的情况下复现。
- 检查问题是否同时出现在 `dev` 和 `build` 中。
- 在修改 manifest + 入口点文件时一次只做一个变更。
