通过可重复的端到端测试验证扩展在不同浏览器上的行为。

Extension.js 项目可以使用 Playwright 在持续集成（CI）与本地环境中测试扩展流程、UI 渲染以及集成行为。

## Playwright 测试能力

| 能力                       | 它为你提供的内容                                                      |
| -------------------------- | --------------------------------------------------------------------- |
| 跨浏览器运行时检查         | 在 Chromium 与 Firefox 引擎上验证核心流程                             |
| UI 与交互覆盖              | 在真实浏览器上下文中测试弹窗/选项页/内容脚本行为                      |
| CI 友好的报告              | 为失败的测试捕获 trace/截图/录像                                      |
| 回归保障                   | 捕获单元测试通常无法发现的集成 Bug                                    |

## 为什么使用

- 捕获单元测试无法发现的运行时回归。
- 在真实浏览器引擎上验证扩展行为。
- 在发布前验证多浏览器变更。

## 典型配置

在项目中安装 Playwright 测试依赖：

<PackageManagerTabs command="install -D @playwright/test" />

创建 `playwright.config.ts` 并定义浏览器项目及报告。

## 推荐基线

```ts
import { defineConfig, devices } from "@playwright/test";

export default defineConfig({
  testDir: "e2e",
  retries: process.env.CI ? 2 : 0,
  reporter: [["html"], ["list"]],
  projects: [
    { name: "chromium", use: { ...devices["Desktop Chrome"] } },
    { name: "firefox", use: { ...devices["Desktop Firefox"] } },
  ],
});
```

## 自动化契约（推荐）

为了实现确定性的自动化，不要解析终端文本。请使用 Extension.js 生成的元数据文件：

- `dist/extension-js/<browser>/ready.json`
- `dist/extension-js/<browser>/events.ndjson`（用于 watch/rebuild 事件的换行分隔 JSON）

`ready.json` 包含面向脚本/智能体的稳定字段：

- `status`：`starting` | `ready` | `error`
- `command`：`dev` | `start` | `preview`
- `browser`
- `distPath`
- `manifestPath`
- `port`
- `pid`
- `runId`
- `startedAt`
- `compiledAt`
- `errors`

将该契约作为就绪状态与失败信息的权威来源。

## 规范的 Playwright 流程（AI 友好）

1. 以无浏览器模式启动 Extension.js。
2. 等待 `ready.json` 报告 `status: "ready"`。
3. 使用来自 `distPath` 的扩展产物启动 Playwright。
4. 运行测试并关闭。

### 开发模式与测试模式

- `dev` 用于 watch 模式迭代（`extension dev --no-browser` + `extension dev --wait`）。
- `start` 用于类生产检查（`extension start --no-browser` + `extension start --wait`）。

### 重要区别：运行模式与就绪门禁

- `--no-browser` 是 **运行模式**：它在不启动浏览器的情况下启动扩展管线。
- 就绪门禁（`extension dev --wait --browser=<browser>`）是 **同步步骤**，告诉 Playwright 扩展何时就绪。

`--no-browser` 会产出构建结果。等待步骤会在测试继续前确认就绪。如果你的环境无法运行第二个 CLI 进程，请直接轮询 `ready.json`。

使用如下双进程模式：

1. 进程 A：`extension dev --no-browser`
2. 进程 B：`extension dev --wait --browser=<browser> --wait-format=json`
3. 仅在等待步骤成功退出后才启动 Playwright

面向生产的变体：

1. 进程 A：`extension start --no-browser`
2. 进程 B：`extension start --wait --browser=<browser> --wait-format=json`
3. 仅在等待步骤成功退出后才启动 Playwright

```ts
import { spawn } from "node:child_process";
import { chromium } from "@playwright/test";

const browserName = "chromium";

const child = spawn(
  "pnpm",
  ["extension", "dev", "--no-browser", `--browser=${browserName}`],
  {
    stdio: "inherit",
    env: process.env,
  },
);

async function waitForReady() {
  return await new Promise<any>((resolve, reject) => {
    let stdout = "";
    let stderr = "";
    const wait = spawn(
      "pnpm",
      [
        "extension",
        "dev",
        "--wait",
        `--browser=${browserName}`,
        "--wait-timeout=60000",
        "--wait-format=json",
      ],
      { stdio: ["ignore", "pipe", "pipe"], env: process.env },
    );
    wait.stdout.on("data", (chunk) => (stdout += chunk.toString()));
    wait.stderr.on("data", (chunk) => (stderr += chunk.toString()));
    wait.on("error", reject);
    wait.on("close", (code) => {
      if ((code ?? 1) !== 0) {
        reject(
          new Error(stderr || `wait command failed with code ${String(code)}`),
        );
        return;
      }
      const payload = JSON.parse(stdout.trim());
      resolve(payload.results[0]);
    });
  });
}

const ready = await waitForReady();

const context = await chromium.launchPersistentContext("", {
  headless: false,
  args: [
    `--disable-extensions-except=${ready.distPath}`,
    `--load-extension=${ready.distPath}`,
  ],
});

// run tests using context/pages...

await context.close();
child.kill("SIGTERM");
```

## 扩展的实用建议

- 让测试夹具保持确定性；扩展启动可能对配置文件状态敏感。
- 优先在扩展 UI 条件上使用显式等待，而非固定的超时。
- 在 CI 中同时运行 Chromium 和 Firefox 项目，以获得跨引擎的可信度。
- 在失败时捕获 trace/截图/录像，加速调试。
- 优先使用 `ready.json`/`events.ndjson`，而不是解析 stdout，以提高机器可靠性。

## 常见陷阱

- 依赖固定超时而非基于状态的等待
- 在 CI 中只运行一个浏览器目标
- 失败运行未上传产物
- 测试与本地特有的配置或环境假设耦合
