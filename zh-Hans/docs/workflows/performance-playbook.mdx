扩展中的性能回归通常来自少数固定的位置。最常见的是 service worker 冷启动（初始启动延迟）、过重的内容脚本注入、过大的 UI 打包，以及媒体密集的 web-accessible 资源。

下面的检查清单为每类问题映射了具体的修复方案。

## 性能优化能力

| 区域            | 它能帮助你优化什么                                                    |
| --------------- | --------------------------------------------------------------------- |
| 内容脚本        | 注入开销、DOM 工作时机以及重新挂载开销                                |
| Service worker  | 启动路径、事件处理开销以及冷启动行为                                  |
| UI 界面         | 初始渲染体积与可选功能的延迟加载                                      |
| 资源            | 打包体积、图标/媒体大小以及 web-accessible 资源占用                   |
| 运维            | 持续集成（CI）检查与运行时回归追踪                                    |

## 快速优化流程

1. 在一个目标浏览器上测量首次运行行为。
2. 削减内容脚本和后台处理器中的同步启动工作。
3. 将可选 UI 功能延迟到初始渲染之后再加载。
4. 重新检查构建产物大小，并对关键流程进行冒烟测试。

## 内容脚本

- 保持入口文件精简，并推迟非关键工作。
- 避免在 `document_start` 阶段进行繁重的同步 DOM 扫描。
- 限定观察者与事件监听器的作用域；在重新挂载/销毁时清理。
- 将可复用逻辑拆分到共享模块中，但避免脆弱的动态导入模式。

## 后台/service worker

- 启动时最小化工作量；待事件到来时再延迟初始化。
- 在安全的情况下缓存稳定的计算状态。
- 避免在事件处理器中执行长时间的同步任务。
- 在活跃开发期间关注 service worker 依赖的变动。

## UI 界面（弹窗/选项页/新标签页/侧边栏）

- 保持初始渲染路径轻量。
- 按需加载大型可选 UI 功能。
- 仅在本地调试会话中确有需要时使用框架开发工具。
- 让样式保持模块化，避免庞大的全局 CSS 负担。

## 资源与素材

- 根据目标用途优化图标/图片大小。
- 限制 web-accessible 资源（WAR）只暴露必需的素材。
- 让公开资源保持有意；移除陈旧文件。
- 定期核查生成的 `dist/<browser>` 输出大小。

## 性能预算

生产构建（`build`）在打包超过其类别大小预算时会发出 **警告**。预算只用于警告 —— 永远不会让构建失败 —— 并在生产模式下默认启用。

| 类别                                                    | 默认预算       | 原因                                                       |
| ------------------------------------------------------- | -------------- | ---------------------------------------------------------- |
| 内容脚本                                                | 512 KiB        | 在每次匹配的页面导航时注入。                               |
| Service worker                                          | 512 KiB        | 每次事件都会从冷启动唤醒；过大的打包会拖慢启动。           |
| 页面 / UI（弹窗、选项页、侧边栏、devtools、新标签页） | 1 MiB          | 按需打开。                                                 |

可以在 [`extension.config.js`](/docs/features/extension-configuration) 中通过 `perfBudgets` 覆盖任何类别（值以字节为单位）：

```js
export default {
  perfBudgets: {
    "content-script": 768 * 1024,
    "service-worker": 512 * 1024,
    page: 2 * 1024 * 1024,
  },
};
```

## 运维检查

- 在 CI 中运行多浏览器构建检查。
- 跟踪端到端（E2E）运行时长随时间的回归信号。
- 为关键扩展流程（安装、打开 UI、内容脚本激活）添加冒烟测试。

## 常见性能陷阱

- 在每个匹配页面上加载的大型内容脚本打包
- 在 `document_start` 阶段没有守卫条件就做大量工作
- service worker 处理器中执行同步的非必要初始化
- UI 界面发送大量全局 CSS/JS，却只用到部分功能
