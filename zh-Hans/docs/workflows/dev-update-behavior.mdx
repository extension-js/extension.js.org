清楚哪些变化能立即生效、哪些需要重启，让迭代保持快速。Extension.js 会为每次文件变更选择最轻量且安全的更新模式。

## 更新模式

| 模式                         | 发生了什么                                                   | 典型影响                                                  |
| ---------------------------- | ------------------------------------------------------------ | --------------------------------------------------------- |
| 热模块替换（HMR）            | 模块/样式更新就地生效                                       | 迭代最快，无需完整重新拉起扩展                            |
| `硬重载`                     | 目标浏览器重新加载扩展运行时                                | 扩展上下文重启，但 dev 进程继续运行                       |
| `需要重启`                   | 编译期诊断要求你重启 `extension dev`                        | 入口结构性变化时必需                                      |

## 常见示例

| 变更                                              | 预期行为                                  |
| ------------------------------------------------- | ----------------------------------------- |
| 编辑 content script/组件代码                      | HMR/重新挂载路径（如支持）               |
| 编辑被 content script/页面使用的样式导入          | 走 HMR 风格的更新路径（依上下文而定）    |
| 编辑 `manifest.json` 的值（非结构性）             | 硬重载流程                                |
| 添加/移除 manifest 入口引用                       | 需要重启                                  |
| 添加/移除 `pages/` 或 `scripts/` 入口             | 需要重启                                  |
| 编辑 `_locales/*/messages.json`                   | 硬重载流程                                |

## 当需要重启时怎么办

当 Extension.js 报出 `Restart required` 时，请先停止，然后重新运行 `extension dev` 再继续。

典型触发：

- 添加或移除 manifest 入口引用
- 添加或移除作为扩展入口使用的 `pages/` 或 `scripts/` 文件
- HTML 入口中的脚本/样式入口出现结构性变化

## 实用建议

- 在活跃的 dev 会话中优先修改已有模块/资源，以获得最快反馈。
- 把结构性的 manifest 变更（入口列表编辑）批量处理，做到只重启一次。
- 把"需要重启"的诊断视为权威结论；不重启就继续，可能让运行时停留在过期状态。
