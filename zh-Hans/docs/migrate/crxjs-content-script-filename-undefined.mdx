你在一个 CRXJS 扩展项目上运行 `vite build` 时得到了:

```text
[crx:manifest-post] Content script fileName is undefined: "src/content.ts"
```

构建失败,清单从未被写入,错误中的路径是你项目中确实存在的文件。本页解释为什么会发生、可以让 CRXJS 继续工作的解决办法,以及结构性的修复方案。

## 这个错误意味着什么

`@crxjs/vite-plugin` 分两遍构建你的扩展。在 `generateBundle` 期间,`crx:manifest-post` 步骤遍历清单中每个 `content_scripts` 条目,并向打包器询问为每个源文件发出的 chunk 的输出文件名。当打包器没有为该源路径返回 chunk 时,查找返回 `undefined`,插件就抛出错误。

所以这个错误不是"你的文件不存在"。它意味着插件和打包器对你内容脚本所发出的 chunk 的命名或键不一致。

## 为什么 Vite 8 会触发它

Vite 8 将默认打包器从 Rollup 替换为 [Rolldown](https://vite.dev/blog/announcing-vite8)。CRXJS 是针对 Rollup 的 chunk 发出模型编写的,其内容脚本解析依赖于在 Rolldown 下行为不同的内部细节。在特定设置下,这个错误以前就存在(在 [crxjs/chrome-extension-tools#883](https://github.com/crxjs/chrome-extension-tools/issues/883) 中被跟踪),但 Vite 8 的默认值让它从边缘场景变成了常见场景。

## 让 CRXJS 继续工作的解决办法

按顺序尝试:

1. **将 Vite 固定为 7。** Vite 7 仍使用 Rollup,CRXJS 能理解它:

   ```bash
   npm install vite@^7 --save-dev --save-exact
   ```

   这能让你今天解除阻塞,但代价是落后于 Vite。

2. **将 `@crxjs/vite-plugin` 更新到最新版本。** 项目一直在逐步添加 Rolldown 兼容性。在固定任何东西之前,请查看 [release 页面](https://github.com/crxjs/chrome-extension-tools/releases) 和上述 issue 了解当前支持状态。

3. **检查清单如何引用脚本。** 查找按路径作为键。让 `content_scripts.js` 条目与你项目根目录的路径风格保持一致(相对路径,不带前导斜杠),并确认同一个文件没有在别处通过 `?script` 后缀被动态导入。

如果你的设置上述方法都不奏效,剩下的选项就是不再依赖 Vite 插件层。

## 结构性修复

根本问题是架构性的。CRXJS 在通用打包器的插件 API 之上重建扩展语义(清单连接、内容脚本发出、HMR),因此每次主要打包器变更都可能破坏它。Extension.js 是一个 [浏览器扩展框架](/docs/compare/extension-js-vs-wxt),其中清单是唯一来源,而打包器是你永远不需要连接的内部细节:

```bash
npx extension@latest dev
```

- 你的 `manifest.json` 直接声明 `content/index.ts`。没有插件去解析 chunk 名称;框架编译清单指向的任何内容。
- 没有 `vite.config.ts`、没有 `manifest.config.ts`、也没有需要保持对齐的插件版本矩阵。
- 同一套源码输出到 Chrome、Firefox 和 Edge,每个面都有自己的重载行为。

你的 React、Vue 或 Svelte 组件以及 `chrome.*` 调用可以原样延续。完整步骤大约需要十分钟:[从 CRXJS 迁移到 Extension.js](/docs/migrate/from-crxjs)。
