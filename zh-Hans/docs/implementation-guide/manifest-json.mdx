通过把 `manifest.json` 作为入口、资源与浏览器特定行为的唯一事实来源,让扩展构建保持可预测。

Extension.js 会编译你的 manifest,并筛选带浏览器前缀的字段。它会重写运行时路径,校验被引用的文件,并为每个目标浏览器产出可立即加载的 manifest。

## Manifest 能力

| 能力                | 你可以获得什么                                              |
| ----------------- | ---------------------------------------------------- |
| 浏览器特定字段过滤         | 保留单一 manifest 文件,同时输出针对各目标的产物                       |
| 路径规范化             | 自动解析运行时安全的输出路径                                       |
| 引用校验              | 当 HTML/脚本/CSS/JSON/图标文件缺失时及早失败                      |
| 按目标输出             | 在 `dist/<browser>` 中按浏览器目标生成 manifest 产物              |

## Extension.js 在哪里读取 manifest

- `src/manifest.json`(存在时优先使用)
- 项目根目录的 `manifest.json`

Extension.js 不会把 `public/manifest.json` 用作源 manifest。

当项目存在 `package.json` 时,Extension.js 会忽略 `public/manifest.json`,以防止被复制的静态资源覆盖 Extension.js 生成的 manifest。

## Extension.js 对它做了什么

在开发/构建期间,manifest 流水线会:

1. 从你的源文件输出 manifest 资源。
2. 为当前激活的浏览器目标筛选带浏览器前缀的键。
3. 对扩展产物应用 manifest 覆盖/路径规范化。
4. 校验被引用的文件(HTML/脚本/CSS/图标/JSON),并在缺失时尽早失败。

## 一份 manifest,多种浏览器

带浏览器前缀的键让你能在保留单一 manifest 的同时,实现浏览器特定行为:

- `chromium:*`、`chrome:*`、`edge:*`
- `firefox:*`、`gecko:*`

这些前缀可应用于顶层键和嵌套的 manifest 字段。

示例:

- `chromium:key`
- `background.firefox:scripts`
- `background.chromium:service_worker`

## 权限设计

manifest 也是扩展声明自身能力的地方。Extension.js 会编译 manifest,但你仍需要良好的权限设计。

- 让 `permissions` 保持小而精。
- 让 `host_permissions` 限制在功能允许的最窄范围内。
- 尽可能把非核心能力移到 `optional_permissions` 或 `optional_host_permissions`。
- 每次内容脚本 matches 或后台能力变化时,审视权限范围。

权限策略请参阅 [权限与 host 权限](/docs/implementation-guide/permissions-and-host-permissions)。

## 输出行为

在需要时,Extension.js 会把 manifest 路径重写到可预测的输出位置。两个重要示例:

- `background.service_worker` 变为 `background/service_worker.js`
- `side_panel.default_path` 变为 `sidebar/index.html`

Extension.js 还会按 manifest 入口索引规范化内容脚本:

- `content_scripts/content-0.js`
- `content_scripts/content-0.css`

被发出的这些路径正是浏览器实际加载的路径。在编写源代码时使用源路径,让 Extension.js 在输出时重写它们。

## 开发期行为

- 当 `manifest.json` 变化时,Extension.js 会重新编译并触发扩展硬重载流程。
- 如果 manifest 入口结构发生变化(例如脚本列表变化),Extension.js 可能需要重启开发服务器。
- 被 manifest 字段引用的文件缺失会导致编译失败,并给出以 manifest 为焦点的错误。

### 变更结果矩阵

| Manifest 变更类型                          | 典型结果                       |
| ------------------------------------- | -------------------------- |
| 更新非结构性的值(例如描述/权限元数据)                   | 硬重载流程                       |
| 更新仍能干净解析的资源路径值                          | 重新编译 + 硬重载流程                |
| 在 manifest 中新增/移除脚本或页面入口                | 需要重启                        |
| 引入不合法/缺失的被引用文件                          | 构建错误(先修复,再重新运行)             |

## 最佳实践

- 让 manifest 路径相对于扩展源/输出模型保持一致,只在你确实表示扩展输出根时才使用 `/` 开头。
- 使用带浏览器前缀的键,而不是为每个浏览器维护单独的 manifest 文件。
- 谨慎处理入口变化;新增/移除 manifest 中的脚本通常会改变开发期的重载语义。
- 在持续集成 (CI) 中校验图标、JSON 资源和内容脚本资产,尽早捕获路径回归。
- 不要把 `manifest.json` 放在 `public/` 下。
