Extension.js 原生支持纯 CSS 以及 Sass 和 Less 预处理器,无需额外配置。它对页面上下文(popup、options、side panel)和内容脚本上下文以不同方式路由样式。

页面样式作为链接资源发布。Extension.js 把内容脚本样式注入到网页的 document 中,使其作用于当前标签页。

![content-sass 模板截图](/images/examples/content-sass/screenshot.png)

## CSS 能力

| 能力                  | 提供给你的功能                                                                  |
| --------------------- | ------------------------------------------------------------------------------- |
| 多上下文样式          | 使用同一套编写模型同时服务于页面和内容脚本                                      |
| Sass/Less 编译        | 当依赖存在时编译 Sass/Less                                                      |
| 上下文感知的输出      | 把页面 CSS 与内容脚本 CSS 输出到正确的目标                                      |
| 开发期更新流程        | 在支持时通过热模块替换(HMR)/重新挂载应用样式更新                              |

## 在哪里引用 CSS

- `manifest.json`(`content_scripts[].css`)
- HTML 文件(`<link rel="stylesheet" href="...">`)
- 脚本导入(`import "./styles.css"`,启用时包括 Sass/Less)

## CSS 支持

Manifest CSS 入口:

| Manifest 字段         | 期望文件类型                      |
| --------------------- | --------------------------------- |
| `content_scripts.css` | `.css`、`.scss`、`.sass`、`.less` |

## 示例:`manifest.json` 中的 CSS

```json
{
  "manifest_version": 3,
  "name": "My Extension",
  "version": "1.0.0",
  "content_scripts": [
    {
      "matches": ["<all_urls>"],
      "css": ["./styles/content.css"],
      "js": ["./scripts/content.ts"]
    }
  ]
}
```

## 示例:扩展页面脚本中的 CSS

```ts
import "./styles/popup.scss";
```

## 按上下文的输出行为

| 上下文            | 输出行为                                                                       |
| ----------------- | ------------------------------------------------------------------------------ |
| HTML/页面上下文   | Extension.js 把 CSS 与页面入口打包在一起(`feature.css`)                      |
| 内容脚本          | Extension.js 把 CSS 作为 `content_scripts/[name].[contenthash:8].css` 资源输出 |

Extension.js 会根据哪个入口点导入了该 CSS,自动划分上下文。

## 开发行为

- 内容脚本的 CSS 导入参与内容脚本的 HMR/重新挂载流程。
- Extension.js 为仅含 CSS 的内容脚本入口添加一个开发辅助脚本,以便传播更新。
- 页面 CSS 遵循常规的页面 HMR 流程行为。
- 结构性的 manifest/内容脚本变更仍可能需要完整的扩展重新加载或重启。

## 模块化与预处理器

- CSS modules 在扩展页面上下文中效果最好。
- 当你安装了相关依赖时,Extension.js 会启用 Sass/Less 支持。
- 当 Extension.js 在项目中检测到 PostCSS 配置,或你显式配置 PostCSS 时,会自动运行 PostCSS。

## 最佳实践

- 有意识地为内容脚本样式划定作用域,减少与宿主页面的冲突。
- 在扩展页面 UI 中优先使用组件局部的模块样式。
- 保持预处理器和 PostCSS 配置显式,避免随时间推移对构建配置造成意外变更。
- 在持续集成(CI)中验证 manifest 字段所引用的 CSS 路径。
