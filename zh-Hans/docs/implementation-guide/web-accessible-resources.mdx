只公开网页或外部扩展上下文必须访问的扩展资源。

Extension.js 会把用户声明的 `web_accessible_resources` 与构建期发现的需求合并,并应用与目标相关的 manifest 规范化。

## Web 可访问资源能力

| 能力                          | 你可以获得什么                                                                                |
| --------------------------- | -------------------------------------------------------------------------------------- |
| 处理 Manifest V2/V3 schema     | 根据 manifest 版本处理数组与对象两种 `web_accessible_resources` 形态                                  |
| 构建感知合并                      | 把声明的资源与构建/运行时所需条目合并                                                                    |
| 路径规范化                       | 把 WAR(web-accessible resource)路径解析到各目标浏览器对应的输出位置                                         |
| 安全控制面                       | 通过 `resources` 与 `matches` 让访问规则保持显式                                                  |

## Manifest schema

### Manifest V3

```json
{
  "web_accessible_resources": [
    {
      "resources": ["images/*.png", "fonts/*.woff2"],
      "matches": ["<all_urls>"]
    }
  ]
}
```

### Manifest V2

```json
{
  "web_accessible_resources": ["images/*.png", "scripts/*.js", "styles/*.css"]
}
```

## `manifest.json` 文件示例

如果你通过受支持的内容脚本或构建路径导入资源,Extension.js 会自动添加所需的 WAR 条目。如果网页需要直接访问特定资源,请手动声明它们。

```json
{
  "manifest_version": 3,
  "name": "My Extension",
  "version": "1.0.0",
  "web_accessible_resources": [
    {
      "resources": ["images/*.png", "scripts/*.js", "styles/*.css"],
      "matches": ["<all_urls>"]
    }
  ],
  "content_scripts": [
    {
      "matches": ["https://example.com/*"],
      "css": ["styles/content.css"],
      "js": ["scripts/content.js"]
    }
  ]
}
```

## Extension.js 自动添加的内容

当运行时需要某些资源对页面可访问时,Extension.js 可以从构建产物中扩展 WAR 条目。

常见示例:

- 内容脚本导入的资源。
- 内容脚本的 CSS 输出。
- 这些条目引用的字体或静态资源。
- 重载行为所需的、仅开发期的运行时支持。

虽然自动加入很方便,但你仍应复审最终对网页公开的资源集合。

## 开发期行为

- 在开发时,Extension.js 会修补 WAR 条目以支持重载和热模块替换 (HMR) 的资源访问。
- 在需要时,Extension.js 会自动在 WAR 中加入与内容脚本相关的资源。
- Extension.js 会通过去除 `public/` 前缀和开头的 `/` 来规范化路径,以产生输出安全的路径。

## 输出与解析说明

- Extension.js 通过 public 资源流程复制 `public/` 中的资源,你可以通过扩展根路径声明它们。
- 相对的 WAR 资源路径从 manifest 文件夹解析。
- 你可以使用 glob,但匹配模式必须符合目标浏览器的约束。
- 在规范化期间,Extension.js 会丢弃没有 `resources` 的 Manifest V3 (MV3) 条目。
- Extension.js 会原样保留 glob,不会把它们扩展为显式的文件列表。

## 运行时用法

使用浏览器运行时 URL 辅助函数,而不是手动构造扩展 URL:

```ts
const logoUrl = chrome.runtime.getURL("/images/logo.png");
```

如果页面或内容脚本无法读取你期望可访问的资源,请检查:

1. 该资源是否真的存在于产物中。
2. WAR 条目是否暴露了正确的 `resources`。
3. `matches` 范围是否包含尝试读取它的页面。
4. 该资源是否应当仅对扩展页面私有保留。

## 最佳实践

- 让 WAR 条目尽量小;只暴露外部上下文必须读取的内容。
- 优先使用显式的资源路径而非宽泛的 glob,以提升安全性。
- 在 Chromium 和 Firefox 目标上校验 WAR 的匹配模式与资源路径。
- 使用 `chrome.runtime.getURL()`/`browser.runtime.getURL()` 创建运行时安全的 URL。
- 在新增内容脚本导入、字体或可被页面访问的静态资源后,重新审视 WAR。

### 安全检查清单

- 把 `matches` 限制在确实需要访问的域名内。
- 在显式小列表已足够时,避免使用宽泛的通配资源 glob。
- 让 WAR 条目专注于非敏感资源。
- 在新增内容脚本导入或运行时资源查找时,重新审视 WAR。

### 范围合理与过宽的示例

```json
{
  "web_accessible_resources": [
    {
      "resources": ["images/logo.png", "fonts/inter.woff2"],
      "matches": ["https://example.com/*"]
    }
  ]
}
```

```json
{
  "web_accessible_resources": [
    {
      "resources": ["*"],
      "matches": ["<all_urls>"]
    }
  ]
}
```
