构建嵌入页面的扩展功能,同时为 JS 和 CSS 更新保持可靠的开发循环。

Extension.js 会从 `manifest.json` 编译内容脚本入口,为它们包装运行时挂载与热模块替换 (HMR) 行为,并输出可预测的 `content_scripts/*` 产物。

## 内容脚本能力

| 能力                            | 你可以获得什么                                                                          |
| ----------------------------- | ------------------------------------------------------------------------------ |
| 由 manifest 驱动的入口              | 直接从 manifest 编译 JS/CSS 内容脚本列表                                                  |
| 开发期 remount 流程                | 通过包装器驱动的行为快速更新脚本/样式                                                            |
| 支持 MAIN 与隔离世界 (isolated world) | 使用 MAIN 世界(页面 JavaScript 上下文)或隔离世界(沙箱式扩展上下文)                                    |
| 可预测的产物布局                      | 输出规范化的 `content_scripts/*` 产物                                                  |

## 支持的 manifest 字段

| Manifest 字段           | 期望的文件类型                            |
| --------------------- | ---------------------------------- |
| `content_scripts.js`  | `.js`、`.jsx`、`.ts`、`.tsx`、`.mjs` |
| `content_scripts.css` | `.css`、`.scss`、`.sass`、`.less`    |

## 内容脚本声明示例

在 `manifest.json` 中声明内容脚本的示例:

```json
{
  "manifest_version": 3,
  "name": "My Extension",
  "version": "1.0.0",
  "content_scripts": [
    {
      "matches": ["<all_urls>"],
      "js": ["./scripts/content-script.ts"],
      "css": ["./styles/content-style.css"]
    }
  ]
}
```

## 编写规范

对于每个类内容脚本入口,Extension.js 期望提供一个 mount 风格的默认导出。这是一个用于设置行为、并可选地返回清理回调的函数:

- 模块应使用 `export default` 导出一个同步函数。
- 该函数应执行初始化工作。
- 它可以返回一个同步的清理回调。
- Extension.js 不支持将类作为默认导出。

这适用于:

- 由 `manifest.json > content_scripts[*].js` 引用的文件。
- 你放在项目 `scripts/` 文件夹中并作为脚本入口使用的文件。

### 有效形式

```ts
export default function main() {
  return () => {};
}
```

```ts
const main = () => {};
export default main;
```

### 无效形式

```ts
export default class App {}
```

```ts
export default {};
```

### 异步内容脚本的指引

即使功能内部要做异步工作,也要保持默认导出是同步的。在函数内部开始异步工作,并返回一个同步的清理回调。

为什么重要:Extension.js 会在开发期间重新挂载内容脚本。没有清理函数时,UI、事件监听器、observer 和定时器可能会重复。

## 违反规范时会发生什么

- 没有默认导出:Extension.js 在开发期间发出警告并跳过挂载。
- 默认导出不是函数:Extension.js 在开发期间发出警告并跳过挂载。
- 默认导出返回 Promise:模块仍会运行,但 Extension.js 不会把该 `Promise` 视为清理回调。

如果你的内容脚本看起来已编译却从未挂载,请先检查默认导出。

## 运行时包装行为

- Extension.js 使用挂载/运行时辅助函数包装内容脚本模块。
- 在开发模式下,Extension.js 添加 HMR 的 accept/dispose 行为以及 remount 流程。
- 在开发模式下,CSS 更新会触发 remount 事件 (`__EXTENSIONJS_CSS_UPDATE__`)。
- Extension.js 遵循 manifest 中的 `run_at` 时机。

## `scripts/` 文件夹行为

`scripts/` 文件夹用于没有任何 HTML 页面入口声明的脚本入口。实际上,这些入口遵循与内容脚本相同的默认导出规范。

也就是说,`scripts/` 并不是放置散乱 JavaScript 文件的通用目录:

- 当脚本入口文件需要挂载行为时,仍应导出一个默认函数
- 在 watch 模式下,Extension.js 会将 `scripts/` 中支持的文件的新增或移除视为结构性变更
- 当该入口集合变化时,Extension.js 可能需要重启开发服务器

## 输出路径

Extension.js 按 manifest 索引规范化内容脚本入口:

```plaintext
content_scripts/
├── content-0.js          # production
├── content-0.abcd1234.js # development (hash-based cache busting)
├── content-0.css
└── ...
```

在**开发模式**下,内容脚本 JS 文件名包含一个简短的 hash 后缀(例如 `content-0.abcd1234.js`)。这样能强制浏览器在每次重新构建后加载一个全新的 `chrome-extension://` URL。

Chrome 会激进地缓存扩展资源,因此 hash 可以防止旧代码被沿用。生产构建则使用干净的 `content-0.js` 名称。

- **`world: "MAIN"` 仅适用于 Chromium。** Firefox 不支持 `world` 字段并会忽略它。你的脚本在 Firefox 上仍会在隔离世界中运行。
- **跨浏览器 MAIN 世界行为:** 使用 `chromium:` manifest 前缀仅为 Chromium 目标声明它,然后为 Firefox 提供隔离世界的回退方案。
- 在开发模式下,MAIN 世界脚本通过内部桥接机制加载代码块并解析 public 路径。
- 在 MAIN 世界中,扩展 API(`chrome.runtime`、`chrome.storage` 等)**不可用**;只能访问页面上下文的全局变量。
- 把 MAIN 世界视为一种进阶路径。请尽早在每个目标浏览器上验证行为。

### 隔离世界与 MAIN 的简短示例

```json
{
  "content_scripts": [
    {
      "matches": ["<all_urls>"],
      "js": ["./scripts/isolated.ts"]
    },
    {
      "matches": ["<all_urls>"],
      "js": ["./scripts/main-world.ts"],
      "world": "MAIN"
    }
  ]
}
```

默认使用隔离世界。仅在需要页面上下文访问时使用 `MAIN`,并考虑扩展 API/运行时的限制。

### 跨浏览器的 MAIN 世界模式

使用[浏览器特定前缀](/docs/features/browser-specific-fields)仅为 Chromium 声明 MAIN 世界,并为 Firefox 提供隔离世界的回退:

```json
{
  "content_scripts": [
    {
      "matches": ["<all_urls>"],
      "js": ["./scripts/isolated.ts"]
    },
    {
      "matches": ["<all_urls>"],
      "chromium:js": ["./scripts/main-world.ts"],
      "chromium:world": "MAIN"
    }
  ]
}
```

Firefox 会完全跳过带 `chromium:` 前缀的字段,因此只有 Chromium 目标会获得 MAIN 世界脚本。

## 匹配与执行指引

浏览器仍然控制内容脚本在何处运行。Extension.js 只负责打包文件,而 manifest 入口仍决定脚本运行的位置与时机。

- 把 `matches` 限制在功能所需的最小范围内。
- 只在功能确实需要时才添加 `exclude_matches`、`all_frames` 或 `match_about_blank`。
- 把 `run_at` 与 `world` 视为功能契约的一部分,而非实现细节。
- 当改变内容脚本的运行位置时,重新检查权限和 host 权限范围。

## 开发期行为

- 编辑内容脚本代码通常会通过包装器驱动的 HMR/remount 流程更新。
- 仅 CSS 的入口会获得开发辅助行为,以便样式更新可以传播。
- 如果 manifest 中的内容脚本入口列表发生变化,Extension.js 可能需要重启开发服务器。

## 最佳实践

- 保持内容脚本入口文件精简,把逻辑下放到共享模块。
- 仔细限定选择器/样式范围,避免与宿主页面冲突。
- 当行为依赖时机/上下文时,优先使用明确的 `run_at` 和 `world` 值。
- 在开发流程中,将 manifest 内容脚本列表的变化视为结构性变更。
- 通过经过校验的消息传递来转发页面派生的数据,而不是直接在内容脚本中做特权工作。
- 默认使用隔离世界,只有在严格需要页面上下文时才切换到 `MAIN`。
