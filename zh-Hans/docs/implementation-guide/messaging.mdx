在扩展上下文之间显式传输数据,并校验每一个特权边界。

Extension.js 会把后台脚本、内容脚本和扩展页面一起打包。你的扩展上下文之间仍通过浏览器扩展消息传递 API 通信。良好的消息结构让各上下文的职责保持清晰,也让调试更可靠。

## 上下文边界

| 从                          | 到                                | 典型 API                                          |
| -------------------------- | -------------------------------- | ------------------------------------------------ |
| popup 或 options 页面          | 后台 service worker                | `runtime.sendMessage()`                          |
| 内容脚本                       | 后台 service worker                | `runtime.sendMessage()` 或 `runtime.connect()`     |
| 后台 service worker           | 内容脚本                            | `tabs.sendMessage()`                             |
| 长期流式或状态同步                  | 后台 service worker 与另一上下文          | `runtime.connect()`                              |

## 推荐契约

把消息视为一种有类型的协议:

- 包含显式的 `type`。
- 在处理前校验 payload 结构。
- 对特权操作,校验发送方上下文。
- 返回结构化的成功与错误响应。

```ts
type Message =
  | { type: "settings:get" }
  | { type: "settings:set"; payload: { theme: "light" | "dark" } };

chrome.runtime.onMessage.addListener((message, sender, sendResponse) => {
  if (
    !message ||
    typeof message !== "object" ||
    typeof message.type !== "string"
  ) {
    sendResponse({ ok: false, error: "invalid_message" });
    return;
  }

  if (message.type === "settings:get") {
    sendResponse({ ok: true, data: { theme: "dark" } });
    return;
  }

  if (message.type === "settings:set") {
    sendResponse({ ok: true });
    return;
  }

  sendResponse({ ok: false, error: "unknown_message" });
});
```

## 何时使用 `sendMessage` vs `connect`

| 用例                            | 优先                       |
| ----------------------------- | ------------------------ |
| 一次请求与一次响应                    | `runtime.sendMessage()`  |
| 长期流式或多次更新                    | `runtime.connect()`      |
| 与特定标签页的内容脚本通信                | `tabs.sendMessage()`     |

## 良好的架构

- 按功能领域为每个领域保留一个消息处理模块,而不是用一个大文件处理所有消息类型。
- 把特权工作集中在后台 service worker 中。
- 让内容脚本收集页面数据,然后只转发最小必要的 payload。
- 让 popup 和 options 页面保持精简,把浏览器 API 调用委托给后台代码。

## 安全规则

- 永远不要因为内容是从内容脚本来的就信任它(它仍可能来自页面)。
- 在执行特权操作之前,校验发送方身份。
- 显式拒绝未知的消息类型。
- 避免通过消息传递暴露通用的 “任意执行” 或 “任意请求” 命令。

## 常见错误

- 让 UI 页面在多处直接调用特权 API,而不是统一路由到后台。
- 在小而结构化的 payload 就够用时,却发送大段任意的页面快照。
- 因为内容脚本是你的代码就信任它,而忽略它接触不受信任的页面数据。
- 当多个上下文一起演进时,忘记为消息结构做版本化或迁移。

## 实用模式

### popup 请求当前状态

使用一次性的请求-响应消息,从 popup 发送到后台。

### 内容脚本请求特权工作

收集尽可能小的页面 payload,发送给后台,让后台决定是否允许操作。

### 实时订阅

仅当你确实需要持续更新时才使用 `runtime.connect()`,而不是用于简单的请求-响应流程。
