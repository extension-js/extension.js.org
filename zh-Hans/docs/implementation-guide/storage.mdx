Extension.js 并不替代或包装浏览器存储 API;它只编译你调用这些 API 的代码。存储区域的选择仍然重要。它会影响 service worker 可靠性、跨浏览器一致性,以及未来数据结构变更。

## 选择正确的存储区域

| 存储区域          | 适用情况                                          | 说明                                       |
| ----------------- | ------------------------------------------------- | ------------------------------------------ |
| `storage.local`   | 单台设备上的持久扩展状态                          | 大多数扩展数据的最佳默认选择               |
| `storage.sync`    | 跨已登录浏览器的小型用户偏好                      | 受配额限制,且比 local 慢                  |
| `storage.session` | 短暂的会话状态                                    | 适合临时的运行时协调                       |
| `storage.managed` | 企业管理的策略值                                  | 由 schema 驱动,对扩展只读                 |

## 推荐默认值

- 把 `storage.local` 用于功能状态、缓存和持久设置。
- 只把 `storage.sync` 用于小型且对用户有意义的偏好。
- 把 `storage.session` 用于不应在完整浏览器重启后保留的状态。
- 把 `storage.managed` 视为独立的企业路径,而不是你的通用设置机制。

## 对 service worker 友好的模式

Manifest V3(MV3)后台 service worker 可能在事件之间停止和重启,所以不要仅依赖内存状态。

- 按需从存储恢复关键状态。
- 仅把内存缓存作为优化手段。
- 保持启动工作较小,以便事件处理保持响应。
- 持久化带版本的设置变更,而不是假设后台脚本会无限期运行。

## 示例设置模块

```ts
const SETTINGS_KEY = "settings";

export async function getSettings() {
  const result = await chrome.storage.local.get(SETTINGS_KEY);
  return result[SETTINGS_KEY] ?? { theme: "system" };
}

export async function setSettings(nextSettings: {
  theme: "system" | "light" | "dark";
}) {
  await chrome.storage.local.set({ [SETTINGS_KEY]: nextSettings });
}
```

## 存储设计清单

1. 确定哪些值必须在浏览器重启后保留。
2. 确定哪些值应当跨设备同步(如果有)。
3. 在首次发布之前为存储数据形态加版本。
4. 为重命名的键或更改的结构准备迁移路径。
5. 避免在客户端可读的扩展存储中保存机密信息,除非你完全接受任何扩展代码都可以读取它们。

## 受管存储

如果你使用 `storage.managed`,需要在 `manifest.json` 中配对一个有效的 `storage.managed_schema` 文件。Extension.js 会校验并把该 schema 文件输出到输出路径。

JSON 资源的详细信息,请参见 [JSON](/docs/implementation-guide/json)。

## 常见错误

- 把 `storage.sync` 用于大型数据集或缓存。
- 假设后台内存变量在 MV3 中是持久的。
- 在没有版本策略的情况下存储多个互不关联的键。
- 把企业受管存储与用户可编辑的设置混在一起,而不分离它们的访问模式。
