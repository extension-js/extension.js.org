## 本节涵盖什么

- 仅看 CLI 文档容易忽略的编写契约。
- 由源文件驱动、影响扩展加载与重载方式的运行时行为。
- 即使在打包之后仍然重要的浏览器扩展架构问题,例如权限、消息传递、存储,以及后台、内容脚本和页面上下文之间的隔离。
- 无论你是手动开发还是与 AI 编程代理协作,都适用的实用指导。

## 重要心智模型

Extension.js 在编译、路径规范化、manifest 重写以及开发体验方面提供帮助。它不会消除底层的浏览器扩展边界:

- 内容脚本仍受页面上下文约束。
- 后台 service worker 仍有生命周期限制。
- Web 可访问资源仍需要显式暴露。
- 权限仍决定扩展能做什么。
- 消息传递与存储仍需要显式设计。

## 新功能的推荐阅读顺序

1. 从 [manifest.json](/docs/implementation-guide/manifest-json) 开始,理解你正在声明的扩展界面。
2. 阅读对应功能页面,例如 [内容脚本](/docs/implementation-guide/content-scripts) 或 [后台](/docs/implementation-guide/background)。
3. 补齐配套内容:[权限](/docs/implementation-guide/permissions-and-host-permissions)、[消息传递](/docs/implementation-guide/messaging)、[存储](/docs/implementation-guide/storage) 和 [web 可访问资源](/docs/implementation-guide/web-accessible-resources)。
4. 在 [可预测的路径解析](/docs/features/path-resolution) 中验证路径假设。
5. 在进入 watch 模式后修改入口前,先查看 [开发期更新行为](/docs/workflows/dev-update-behavior)。
