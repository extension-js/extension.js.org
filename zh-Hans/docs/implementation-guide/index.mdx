## 本节涵盖的内容

- 仅靠 CLI 文档容易忽略的编写契约。
- 由源文件驱动的运行时行为,会影响扩展的加载与重新加载方式。
- 打包之后仍然重要的浏览器扩展架构问题,例如权限、消息传递、存储,以及后台、内容脚本与页面上下文之间的隔离。
- 无论你是手动编写还是使用 AI 编码助手,都适用的实用建议。

## 重要的心智模型

Extension.js 帮你处理编译、路径规范化、manifest 重写以及开发体验,但它并不会消除浏览器扩展底层的边界:

- 内容脚本仍受到页面上下文约束。
- 后台 service worker 仍有生命周期限制。
- Web 可访问资源仍需要显式暴露。
- 权限仍决定扩展能做什么。
- 消息传递和存储仍需要显式设计。

## 新功能的推荐阅读顺序

1. 从 [manifest.json](/docs/implementation-guide/manifest-json) 开始,理解你所声明的扩展表面。
2. 阅读功能相关页面,例如 [内容脚本](/docs/implementation-guide/content-scripts) 或 [后台](/docs/implementation-guide/background)。
3. 补充辅助内容:[权限](/docs/implementation-guide/permissions-and-host-permissions)、[消息传递](/docs/implementation-guide/messaging)、[存储](/docs/implementation-guide/storage) 以及 [Web 可访问资源](/docs/implementation-guide/web-accessible-resources)。
4. 在 [可预测的路径解析](/docs/features/path-resolution) 中验证路径假设。
5. 在主动监听模式下修改入口点之前,先查看 [开发更新行为](/docs/workflows/dev-update-behavior)。
