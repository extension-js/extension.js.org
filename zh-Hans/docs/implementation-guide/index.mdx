## 本章涵盖的内容

- 仅靠 CLI 文档容易遗漏的编写契约。
- 由你的源文件驱动、影响扩展加载与重载方式的运行时行为。
- 打包之后仍然重要的浏览器扩展架构问题，例如权限、消息传递、存储，以及 background、content script 和页面上下文之间的边界。
- 无论是手工编写还是配合 AI 编码代理，都适用的实用建议。

## 重要心智模型

Extension.js 在编译、路径规范化、manifest 重写和开发体验上提供帮助。它并不消除浏览器扩展底层的边界：

- Content scripts 仍然在页面上下文的约束下运行。
- Background service worker 仍然有生命周期限制。
- Web-accessible resources 仍然需要显式暴露。
- 权限仍然决定扩展能做什么。
- 消息传递和存储仍然需要显式设计。

## 新功能的推荐阅读顺序

1. 先看 [manifest.json](/zh-Hans/docs/implementation-guide/manifest-json)，理解你正在声明的扩展表面。
2. 阅读对应功能的页面，例如 [content scripts](/zh-Hans/docs/implementation-guide/content-scripts) 或 [background](/zh-Hans/docs/implementation-guide/background)。
3. 补齐配套部分：[permissions](/zh-Hans/docs/implementation-guide/permissions-and-host-permissions)、[messaging](/zh-Hans/docs/implementation-guide/messaging)、[storage](/zh-Hans/docs/implementation-guide/storage) 和 [web-accessible resources](/zh-Hans/docs/implementation-guide/web-accessible-resources)。
4. 在 [Predictable path resolution](/zh-Hans/docs/features/path-resolution) 中验证路径假设。
5. 在 watch 模式下改动入口点之前，先回顾 [dev update behavior](/zh-Hans/docs/workflows/dev-update-behavior)。
