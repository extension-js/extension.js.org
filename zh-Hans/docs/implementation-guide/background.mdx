在后台上下文中运行扩展级别的逻辑,清晰支持 Manifest V2(MV2)后台脚本与 Manifest V3(MV3)Service Worker。

Extension.js 从 `manifest.json` 读取后台入口,并将其编译为专用的后台输出。开发期间它会应用各浏览器特定的重新加载行为。

## 后台能力

| 能力                       | 提供给你的功能                                                       |
| -------------------------- | -------------------------------------------------------------------- |
| MV2/MV3 兼容               | 支持 `background.scripts` 与 `background.service_worker` 入口        |
| 专用后台输出               | 按当前活跃的 manifest 形态生成后台运行时文件                         |
| 感知重新加载的开发体验     | 对结构性变更应用硬重载/重启行为                                      |
| Worker 模式控制            | 遵循 `background.type` 的 module/classic 运行时行为                  |

## 后台脚本支持

通过以下 `manifest.json` 字段声明后台脚本:

| Manifest 字段               | 期望文件类型                 | 开发行为                                          |
| --------------------------- | ---------------------------- | ------------------------------------------------- |
| `background.service_worker` | `.js`、`.ts`、`.mjs`、`.tsx` | 硬重载流程                                        |
| `background.scripts`        | `.js`、`.ts`、`.mjs`、`.tsx` | 兼容热模块替换(HMR)的路径                       |

`background.type` 还可能影响运行时模式(`module` 与经典 service worker 行为)。

## 后台脚本声明示例

`manifest.json` 中的后台脚本声明示例:

```json
{
  "manifest_version": 3,
  "name": "My Extension",
  "version": "1.0.0",
  "background": {
    "service_worker": "./scripts/background.ts"
  }
}
```

## 开发行为

- Extension.js 追踪 service worker/源文件变更,可触发扩展硬重载。
- 影响后台入口的 manifest 变更可能触发“需要重启”诊断。
- 浏览器启动插件会应用目标特定的硬重载策略(Chromium/Firefox)。
- Extension.js 对结构性入口变更比普通模块编辑更严格。

## Firefox:Service Worker 转换为事件页面

Firefox 不会运行 MV3 的 `background.service_worker`(加载这类扩展会失败,提示 _“background.service_worker is currently disabled. Add background.scripts.”_)。你无需特殊处理:对于 Firefox 构建,Extension.js 会自动把 `background.service_worker` 入口转换为指向同一编译产物的 `background.scripts` 事件页面。只需编写一个 `background.service_worker`,它就能同时在 Chromium 和 Firefox 上加载。

## MV3 Service Worker 生命周期

`background.service_worker` 在浏览器的 service worker 生命周期下运行,并不是一个长时间运行的进程。

这意味着:

- 内存中的状态可能在事件之间消失。
- 长时间运行的工作应围绕事件设计,而不是依赖进程持久性。
- 启动应保持轻量且可预测。
- 你的代码应当从存储中恢复重要状态,或安全地重新计算。

仅在需要 MV2 风格兼容的情况下使用 `background.scripts`。对于以 MV3 为先的扩展,把 service worker 作为浏览器 API 调用和跨上下文通信的中心协调者。

## 输出行为

常见的输出包括:

- `background/service_worker.js`
- `background/scripts.js`

具体输出取决于浏览器过滤后哪个 manifest 字段仍然处于活跃状态。

## Module 与 classic 说明

- `background.type: "module"` 使用 module worker 语义。
- Classic service worker 模式使用基于 `importScripts` 的代码块加载行为。
- 避免在必须于启动时立即加载的后台代码中使用动态导入。

## 推荐架构

- 保持后台入口文件精简,把功能逻辑放入共享模块。
- 把后台上下文作为消息传递、存储访问和浏览器 API 调用的中心协调者。
- 从存储中恢复持久化状态,而不是假设后台脚本会一直运行。
- 使用 alarms、明确的事件监听器和小的功能模块,而不是一个庞大的启动路径。

## 常见错误

- 把 service worker 当作永久运行的服务进程。
- 把关键状态只保存在内存中。
- 在每次事件唤醒时执行昂贵的启动工作。
- 把过多功能逻辑放在 popup 或内容脚本代码中,而实际上它需要后台级别的浏览器 API 访问。

## 最佳实践

- 对于以 MV3 为先的扩展,优先使用 `background.service_worker`。
- 保持后台入口文件较小,把逻辑委托给共享模块。
- 避免在 service worker 中进行昂贵的启动工作;在安全时进行惰性初始化。
- 在开发流程中,把 manifest 后台字段的编辑视为结构性变更。
- 通过已验证的消息处理器路由特权操作。
- 把持久化设置和缓存保存在浏览器存储中,而不仅仅是模块级变量中。
