为你的扩展界面选择合适的语言与 UI 框架方案，然后借助 Extension.js 的默认配置，把搭建成本降到最低。

## 按目标选择

| 目标                                       | 从这里开始                                                                               |
| ------------------------------------------ | ---------------------------------------------------------------------------------------- |
| 更强的类型与更安全的重构                   | [TypeScript](/zh-Hans/docs/languages-and-frameworks/typescript)                          |
| 丰富的 UI 组件与庞大的生态                 | [React](/zh-Hans/docs/languages-and-frameworks/react)                                    |
| 更小的类 React 运行时                      | [Preact](/zh-Hans/docs/languages-and-frameworks/preact)                                  |
| Vue 单文件组件工作流                       | [Vue.js](/zh-Hans/docs/languages-and-frameworks/vue)                                     |
| 现代模块语法                               | [ECMAScript modules](/zh-Hans/docs/languages-and-frameworks/ecmascript-modules)          |
| 默认带作用域的样式                         | [CSS modules](/zh-Hans/docs/languages-and-frameworks/css-modules)                        |
| Sass 或 Less 样式工作流                    | [Sass](/zh-Hans/docs/languages-and-frameworks/sass)，[Less](/zh-Hans/docs/languages-and-frameworks/less) |
| 浏览器安全的 Node 兼容性决策               | [浏览器扩展中的 Node API](/zh-Hans/docs/languages-and-frameworks/node)                   |
| 性能敏感的 Wasm 工作负载                   | [WebAssembly](/zh-Hans/docs/languages-and-frameworks/webassembly)                        |

## 建议的推进顺序

1. 选择语言/运行时（`TypeScript`、`React`、`Vue` 或 `Preact`）。
2. 选择样式方案（`CSS modules`、`Sass` 或 `Less`）。
3. 仅在需要时再添加构建/运行时方面的约束（`Node APIs`、`WebAssembly`）。
