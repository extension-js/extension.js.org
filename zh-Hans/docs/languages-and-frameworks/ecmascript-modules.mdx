在所有扩展上下文中使用现代 `import`/`export` 语法编写扩展代码。

Extension.js 在后台脚本、内容脚本和扩展页面中支持 ECMAScript Modules（ESM），并使用默认的 Rspack 管线进行模块解析。

## 适合使用 ESM 的场景

- 你希望在扩展和 Web 代码之间保持一致的模块语法。
- 你需要与现代 ESM 优先的包共享代码。
- 你需要更清晰的 tree-shaking（自动移除未使用代码）和明确的依赖边界。

## 在现有扩展中使用

你可以直接在扩展源码文件（`.js`、`.mjs`、`.ts`、`.tsx` 等）中使用 ESM 语法，无需自定义打包器配置。

如果你的 Node.js 项目配置文件也需要以 ESM 方式运行，请在 `package.json` 中设置 `"type": "module"`。这适用于自定义脚本和配置约定：

```json
{
  "name": "my-extension",
  "version": "1.0",
  "description": "My Extension Example",
  "type": "module",
  "devDependencies": {
    "extension": "latest"
  },
  "scripts": {
    "dev": "extension dev",
    "start": "extension start",
    "build": "extension build"
  }
}
```

## 关于 manifest 与 service worker 的说明

对于 Manifest V3 的后台 worker：

- 当你希望 service worker 中使用原生 ES 模块加载时，请将 `background.type` 设为 `"module"`。
- 如果未指定 module worker 类型，Extension.js 会使用经典的 worker 加载方式（不带原生模块语法的打包脚本）。

```json
{
  "background": {
    "service_worker": "src/background.ts",
    "type": "module"
  }
}
```

## ESM 与 CommonJS 的注意事项

在编写 ESM 模块时：

### 相对导入需包含文件扩展名

```diff
// my-file.mjs
import React from 'react'

// local imports
- import myImport from './myImport'
+ import myImport from './myImport.js'
```

### 与非 ESM 模块的互操作性可能不同

从 ESM 导入 CommonJS 包时，请按各个包提供的兼容性说明操作。

### 严格 ESM 上下文中无法使用 CommonJS 全局变量

避免在 ESM 模块中依赖 `require`、`module.exports`、`__filename` 和 `__dirname`。

## 在 ECMAScript 模块中处理环境变量

Extension.js 同时支持：

- `process.env.EXTENSION_PUBLIC_*`
- `import.meta.env.EXTENSION_PUBLIC_*`

对于希望在扩展代码中可用的变量，请使用 `EXTENSION_PUBLIC_` 前缀。

```js
// example.mjs

console.log(import.meta.env.EXTENSION_PUBLIC_API_KEY);
console.log(process.env.EXTENSION_PUBLIC_API_KEY);
```

## 导入是否必须包含文件扩展名？

Node.js 在 ESM 的导入说明符中要求显式的文件扩展名（`import "./util.js"`，而不是 `import "./util"`）。该规则适用于 Node 直接运行的代码，常常让从服务器端代码切换到扩展开发的人困惑。

在 Extension.js 中，由打包器而非 Node 来解析导入，因此在扩展代码中两种写法都可用：

```js
import { parse } from "./util"; // resolved by the bundler
import { parse } from "./util.js"; // also fine
```

仍然适用 Node 规则的两种情况：

- 由你的 `package.json` 直接通过 Node 运行的脚本（构建辅助脚本、代码生成）：在那里扩展名是必需的。
- 同时被 Node 工具链和扩展代码共享的 `.mjs` 文件：编写扩展名时确保两种解析器都能接受。
