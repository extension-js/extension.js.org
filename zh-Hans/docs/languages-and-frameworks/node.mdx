Extension.js 默认面向浏览器运行时，不会自动为 Node 核心模块提供 polyfill，从而保持较小的打包体积。如果某个依赖需要 `buffer`、`stream` 或 `crypto`，构建时会出现解析错误。下面的章节说明何时应添加 polyfill，何时应选择浏览器原生的替代方案。

## 适合使用 Node polyfill 的场景

- 某个必需的依赖需要在浏览器运行时中使用 Node 核心模块。
- 你正在逐步迁移以 Node 为中心的包代码。
- 你可以为特定的运行时能力接受更大的打包体积。

## 默认行为

- 构建目标以浏览器为主（`web`）。
- 解析优先匹配浏览器导出（`browser`、`module`、`main`）。
- Extension.js 默认禁用 Node 核心模块的兜底（例如 `crypto`、`path` 和 `fs`）。

这意味着除非你显式添加兜底，否则导入 Node API 可能会失败。

## 设置 Node polyfill

使用 `extension.config.js`（或 `.mjs` / `.cjs`）扩展 Rspack 配置并定义安全的兜底。

```js
import NodePolyfillPlugin from "node-polyfill-webpack-plugin";
import { createRequire } from "node:module";

const require = createRequire(import.meta.url);

export default {
  config: (config) => {
    config.resolve = config.resolve || {};
    config.resolve.fallback = {
      ...(config.resolve.fallback || {}),
      crypto: require.resolve("crypto-browserify"),
      path: require.resolve("path-browserify"),
      fs: false,
    };
    config.plugins = config.plugins || [];
    config.plugins.push(new NodePolyfillPlugin());
    return config;
  },
};
```

## 扩展 API 与 Node API 的区别

在 CLI/配置中设置 `polyfill: true` 会启用 `webextension-polyfill` —— 这是一个兼容层，让 Chromium 浏览器可以访问 `browser.*` API 命名空间。它并不会启用 Node 核心模块的 polyfill。

仅在那些无法使用标准 Web API 运行的库上使用 Node polyfill。

## 注意事项

- 不要假设在扩展运行时中可访问文件系统；除非你有专门的浏览器安全策略，否则请保留 `fs: false`。
- 在可能的情况下，优先使用 Web Crypto（`crypto.subtle`），而不是体积庞大的 Node crypto 兼容层。
- Polyfill（Node API 的浏览器兼容替代品）会增加打包体积，并可能影响扩展页面和内容脚本的启动时间。
- 部分使用 `node:` 说明符的包可能需要显式处理兜底。
