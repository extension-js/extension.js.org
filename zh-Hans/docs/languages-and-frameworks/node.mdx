Extension.js 默认面向浏览器运行时。它不会自动为 Node 核心模块加 polyfill，这能让 bundle 保持精简。如果某个依赖用到了 `buffer`、`stream` 或 `crypto`，你会在构建期看到解析错误。下面的小节会说明什么时候应该加 polyfill，什么时候应该选择浏览器原生替代方案。

## Node polyfill 适合什么场景

- 某个必须的依赖在浏览器运行时中需要 Node 核心模块。
- 你在从以 Node 为中心的包中逐步迁移代码。
- 为了某些特定的运行时能力，你可以接受更大的 bundle。

## 默认行为

- 构建目标以浏览器为先（`web`）。
- 模块解析优先选择浏览器导出（`browser`、`module`、`main`）。
- Extension.js 默认禁用 Node 核心 fallback（例如 `crypto`、`path` 与 `fs`）。

这意味着如果不显式添加 fallback，导入 Node API 可能会失败。

## 配置 Node polyfill

使用 `extension.config.js`（或 `.mjs` / `.cjs`）来扩展 Rspack 配置，并定义安全的 fallback。

```js
import NodePolyfillPlugin from "node-polyfill-webpack-plugin";
import { createRequire } from "node:module";

const require = createRequire(import.meta.url);

export default {
  config: (config) => {
    config.resolve = config.resolve || {};
    config.resolve.fallback = {
      ...(config.resolve.fallback || {}),
      crypto: require.resolve("crypto-browserify"),
      path: require.resolve("path-browserify"),
      fs: false,
    };
    config.plugins = config.plugins || [];
    config.plugins.push(new NodePolyfillPlugin());
    return config;
  },
};
```

## 扩展 API 与 Node API

在 CLI/配置中设置 `polyfill: true` 会启用 `webextension-polyfill`，这是一层兼容性封装，让 Chromium 浏览器也能访问 `browser.*` 命名空间。这并不会启用 Node 核心模块的 polyfill。

只有当某个库无法用标准 Web API 运行时，才使用 Node polyfill。

## 注意事项

- 不要假设扩展运行时可以访问文件系统；除非你有明确的浏览器安全方案，否则保持 `fs: false`。
- 在可能的情况下，优先使用 Web Crypto（`crypto.subtle`），而不是体积庞大的 Node crypto shim。
- Polyfill（Node API 的浏览器兼容替代实现）会增大 bundle 体积，并可能影响扩展页面与 content scripts 的启动时间。
- 一些使用 `node:` 前缀的包可能需要显式的 fallback 处理。
