无需自定义打包器配置即可在你的扩展中使用 WebAssembly。

Extension.js 在 Rspack 管线中内置了 WebAssembly（Wasm）默认配置，因此常见的 Wasm 工作流无需额外配置即可工作。

## 适合使用 WebAssembly 的场景

- 你需要在扩展中运行计算密集型任务（解析、媒体、光学字符识别（OCR）、变换）。
- 你需要在常被执行的代码路径上获得接近原生的性能。
- 你在复用来自 Web 项目的现有 Wasm 模块。

## WebAssembly 能力

| 能力                          | 它为你提供的内容                                   |
| ----------------------------- | -------------------------------------------------- |
| 内置 Wasm 管线默认值          | 无需手动接线即可运行常见的 Wasm 流程               |
| 异步 Wasm 支持                | 在扩展上下文中加载 `.wasm` 模块                    |
| 运行时兼容辅助                | 处理常见的 Wasm 生态资源模式                       |
| 常规命令工作流                | 使用相同的 `dev` 和 `build` 命令配合 Wasm 工作流   |

## Extension.js 启用了什么

Wasm 插件配置：

- `experiments.asyncWebAssembly = true`
- 模块解析扩展名中包含 `.wasm`
- 为常见 Wasm 库提供路径别名（例如用于媒体处理的 ffmpeg、图像处理的 imagemagick 以及 OCR 的 tesseract），使运行时的导入能正确解析。

这能减少在扩展脚本/页面中导入 Wasm 模块时的样板代码。

## 典型使用场景

- 计算密集型的解析/变换。
- 媒体处理管线。
- OCR / 图像处理工作流。
- 在 Web/运行时上下文之间共享的性能敏感算法。

## 最佳实践

- 让 Wasm 模块专注于热点路径；将编排逻辑保留在 JavaScript/TypeScript 中。
- 验证扩展上下文中（后台、内容脚本、页面）的产物大小和加载时机。
- 在你发布的浏览器目标上测试 Wasm 重度流程（`chrome`、`edge`、`firefox`）。
