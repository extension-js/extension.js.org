无需自定义打包器配置即可在扩展中使用 WebAssembly。

Extension.js 在其 Rspack 流水线中内置 WebAssembly (Wasm) 默认设置,因此常见的 Wasm 工作流无需额外配置即可使用。

## 何时适合使用 WebAssembly

- 你在扩展中运行计算密集型任务(解析、媒体、光学字符识别 (OCR)、转换等)。
- 你需要对频繁执行的代码路径接近原生的性能。
- 你正在复用 Web 项目中已有的 Wasm 模块。

## WebAssembly 能力

| 能力                          | 它能为你带来什么                                |
| ----------------------------- | ----------------------------------------------- |
| 内置 Wasm 流水线默认设置      | 无需手动配置打包器即可运行常见 Wasm 流程         |
| 异步 Wasm 支持                | 在扩展上下文中加载 `.wasm` 模块                  |
| 运行时兼容性辅助              | 处理常见的 Wasm 生态资源模式                     |
| 常规命令工作流                | 使用同样的 `dev` 和 `build` 命令配合 Wasm        |

## Extension.js 启用了什么

Wasm 插件会配置:

- `experiments.asyncWebAssembly = true`
- 在模块解析扩展名中加入 `.wasm`
- 为常见 Wasm 库设置路径别名(例如用于媒体处理的 ffmpeg、用于图像处理的 imagemagick、用于 OCR 的 tesseract),以便导入在运行时能正确解析。

这减少了从扩展脚本/页面导入 Wasm 模块时的样板代码。

## 典型使用场景

- 计算密集型的解析/转换。
- 媒体处理流水线。
- OCR / 图像处理工作流。
- 在 Web/运行时上下文之间共享的对性能敏感的算法。

## 最佳实践

- 让 Wasm 模块聚焦于热点路径;将编排逻辑保留在 JavaScript/TypeScript 中。
- 验证 Wasm 模块在扩展上下文(后台、内容脚本、页面)中的输出体积与加载时机。
- 在你要发布的浏览器目标(`chrome`、`edge`、`firefox`)上测试 Wasm 密集型流程。
