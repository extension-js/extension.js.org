在你的扩展中使用 WebAssembly，无需自定义打包器配置。

Extension.js 在其 Rspack 管线中内置了 WebAssembly（Wasm）的默认配置，常见的 Wasm 工作流无需额外配置即可工作。

## WebAssembly 适合什么场景

- 你需要在扩展中运行计算密集型任务（解析、媒体处理、光学字符识别（OCR）、各类转换）。
- 你需要让频繁执行的代码路径接近原生性能。
- 你在复用来自 Web 项目的现有 Wasm 模块。

## WebAssembly 能力

| 能力                          | 你能获得什么                                                |
| ----------------------------- | ----------------------------------------------------------- |
| 内置 Wasm 管线默认值          | 无需手动接线即可运行常见的 Wasm 工作流                      |
| 异步 Wasm 支持                | 在扩展上下文中加载 `.wasm` 模块                             |
| 运行时兼容性辅助              | 处理常见的 Wasm 生态资源模式                                |
| 一致的命令工作流              | 使用相同的 `dev` 和 `build` 命令运行 Wasm                   |

## Extension.js 启用了什么

Wasm 插件会配置：

- `experiments.asyncWebAssembly = true`
- 把 `.wasm` 加入模块解析的扩展名列表
- 为常见 Wasm 库设置路径别名（例如用于媒体处理的 ffmpeg、用于图像处理的 imagemagick、用于 OCR 的 tesseract），以便在运行时正确解析导入。

这减少了在扩展脚本/页面中导入 Wasm 模块时的样板代码。

## 典型使用场景

- 计算密集型的解析/转换。
- 媒体处理流水线。
- OCR / 图像处理工作流。
- 在 Web/运行时上下文之间共享的、对性能敏感的算法。

## 最佳实践

- 让 Wasm 模块专注于热点路径；把编排逻辑留在 JavaScript/TypeScript 中。
- 针对扩展上下文（background、content scripts、页面）验证输出体积和加载时机。
- 在你要发布的所有浏览器目标（`chrome`、`edge`、`firefox`）上测试 Wasm 密集的流程。
