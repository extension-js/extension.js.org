Manifest V3（MV3）以 service worker 取代背景頁、收緊 content security policy，並重新設計擴充功能宣告網路與主機存取的方式。大部分日常痛點都有同一個根源:MV3 假設背景程式碼是短暫且事件驅動的。Chrome 與 Firefox 對部分鍵的解讀也有些差異。

本頁收錄建構 MV3 擴充功能時最常遇到的問題，並列出確切修正方式，以及 Extension.js 已為你處理哪些事。

## 含 `type: "module"` 的 Manifest V3 背景 `service_worker`

Chromium 使用 service worker 作為 MV3 背景。要從中匯入 ES 模組，manifest 需要 `type: "module"`：

```json
{
  "manifest_version": 3,
  "background": {
    "service_worker": "service_worker.js",
    "type": "module"
  }
}
```

沒有 `"type": "module"` 的話，`import` 陳述式會在註冊階段失敗，且不會在擴充功能 console 留下清楚的錯誤。加上之後，你就能在 worker 檔中使用現代 ES 模組語法。

當你的背景進入點使用 ES 模組語法時，Extension.js 會自動設定 `type: "module"`。它也會把 `.ts`/`.tsx` worker 編譯成單一打包輸出，因此正式版的模組解析依然能正常運作。

## 為什麼 `background.js` 在 Manifest V3 中的行為不同

在 Manifest V2 中，`background.js` 跑在帶有 DOM 的持久背景頁。在 Chromium 的 Manifest V3 中，背景以 service worker 形式執行：

- 沒有 DOM。`window`、`document`、`XMLHttpRequest` 與 `localStorage` 都不存在。請使用 `fetch` 與 `chrome.storage`。
- worker 在閒置時**可能隨時被終止**，並在下一個事件被喚醒。不要在模組層級變數儲存狀態;請改存在 `chrome.storage`。
- 允許 top-level `await`，但長時間的初始化本身不會讓 worker 保持存活。
- 將事件監聽器**同步**註冊在檔案最上方，不要放在 async 回呼中。註冊在 async 回呼裡的監聽器，不會在喚醒 worker 的那個事件上觸發。

Firefox 仍使用非持久事件頁，而不是 service worker。Extension.js 會從同一份原始碼,自動把 Chromium 路由到 `service_worker`、把 Firefox 路由到 `scripts`，因此同一個背景進入點能對各目標正確編譯。

## Manifest V3 中的 `web_accessible_resources`

Manifest V3 把 `web_accessible_resources` 從扁平的檔案陣列改成 `{ resources, matches }` 區塊清單：

```json
{
  "web_accessible_resources": [
    {
      "resources": ["images/logo.png", "pages/injected.html"],
      "matches": ["https://example.com/*"]
    }
  ]
}
```

常見錯誤：

- 列出沒有出現在建置輸出中的路徑。Extension.js 只會輸出被 `manifest.json`、進入 HTML 或被匯入的程式碼引用的檔案。請把該檔案加為資源，讓它出現在 `dist/<browser>`。
- 忘記 `matches`。少了它，資源不會暴露給任何來源。
- 使用 V2 風格的扁平字串。瀏覽器在 MV3 中會默默忽略它們。

完整模式（包括把擴充功能 URL 注入到 content script）請參閱 [`web_accessible_resources` 實作](/zh-Hant/docs/implementation-guide/web-accessible-resources)。

## `host_permissions` 與 `permissions`

Manifest V3 把主機存取從 `permissions` 陣列中拆出來：

| Key                                                    | 控制什麼                                                                                       |
| ------------------------------------------------------ | ---------------------------------------------------------------------------------------------- |
| `permissions`                                          | 具名 API 介面（`storage`、`tabs`、`cookies`、`scripting`、`alarms` 等）。                       |
| `host_permissions`                                     | 擴充功能可讀寫的 URL 比對樣式（`https://*/*`、`*://api.example.com/*`）。                       |
| `optional_permissions` 與 `optional_host_permissions`  | 在執行階段透過 `chrome.permissions.request` 請求的權限。                                       |

混用兩者的徵狀：

- `chrome.cookies.get` 對你應該有權存取的站點回傳空值:該 URL 需要 `host_permissions`，不只是 `cookies`。
- `chrome.scripting.executeScript` 出現「Cannot access contents of url」錯誤:把該 URL 加入 `host_permissions`。
- 當你只需要一個來源時，Web Store 卻警告使用者「全站存取」:把主機樣式收得更窄。

逐 API 的參考請見 [Permissions 與 host permissions](/zh-Hant/docs/implementation-guide/permissions-and-host-permissions)。

## Firefox 中的 `declarative_net_request`

`declarative_net_request`（DNR）是 MV3 取代阻擋型 `webRequest` 的方案。Firefox 支援它，但有一些限制：

- 靜態規則資源檔（`rule_resources`）必須是有效的 JSON 陣列。Firefox 對空或格式錯誤的規則檔比 Chrome 嚴格。
- Chrome 支援的一些規則動作與條件在 Firefox 上仍只有部分支援。請查閱 [MDN 的 `declarativeNetRequest` 參考](https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/API/declarativeNetRequest)取得目前的支援矩陣。
- 對跨來源請求要套用 DNR 規則，需要 `host_permissions`（或 `<all_urls>`）。

Extension.js 在建置時驗證 DNR 資源，並針對每個瀏覽器輸出產物，這樣 Firefox 專屬的規則檔就不會出現在 Chrome 建置裡。

## Content script、worker 與擴充功能 URL

MV3 有三個常讓人踩雷的地方：

- **Content script 無法存取頁面的 JavaScript 範疇。** 它們共用 DOM，但不共用 window。請使用 `window.postMessage`（或指向 `web_accessible_resources` 條目的 `<script>` 標籤）與頁面程式碼溝通。
- **Service worker 在安裝期間無法可靠地呼叫 `chrome.runtime.getURL`。** 請改在需要的事件處理器內惰性解析 URL。
- **擴充功能 URL 是受來源範圍限制的。** 擴充功能內部的兩個頁面可以互相溝通，但除非它們列在 `web_accessible_resources` 內,否則無法從一般網頁 fetch。

## Extension.js 在 Manifest V3 上的幫助

- 自動把背景進入點路由到 `service_worker`（Chromium）或 `scripts`（Firefox）。
- 當背景使用 ES 模組語法時,自動設定 `type: "module"`。
- 在建置時依目前 manifest 版本驗證 `web_accessible_resources`、`permissions` 與 `host_permissions`。
- 輸出每個瀏覽器各自的 `dist/<browser>` 產物，讓 Chrome 與 Firefox 在 MV3 上的差異彼此隔離。
- 在 `extension dev` 期間，於儲存時重新載入 content script、service worker 與 HTML 頁面。
