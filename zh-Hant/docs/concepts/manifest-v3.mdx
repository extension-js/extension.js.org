Manifest V3（MV3）以 service worker 取代背景頁面、收緊內容安全政策，並重塑了擴充功能宣告網路與主機存取的方式。大多數日常痛點都源自同一個根本原因：MV3 假設背景程式碼是短暫的、事件驅動的。Chrome 與 Firefox 對少數鍵也有不同的解讀。

本頁面整理在建置 MV3 擴充功能時最常見的問題，附上精確的修正方式以及 Extension.js 替你處理的部分。

## Manifest V3 背景 `service_worker` 與 `type: "module"`

Chromium 在 MV3 中使用 service worker 作為背景。要從其中 import ES 模組，manifest 需要設定 `type: "module"`：

```json
{
  "manifest_version": 3,
  "background": {
    "service_worker": "service_worker.js",
    "type": "module"
  }
}
```

沒有 `"type": "module"`，`import` 陳述句會在註冊時失敗，而擴充功能控制台上不會出現明確的錯誤訊息。加上之後，你就可以在 worker 檔案中撰寫現代 ES 模組語法。

當你的背景進入點使用 ES 模組語法時，Extension.js 會自動設定 `type: "module"`。它也會把 `.ts`/`.tsx` worker 編譯成單一的打包輸出，使模組解析在正式環境中仍可正常運作。

## 為什麼 `background.js` 在 Manifest V3 中的行為不同

在 Manifest V2 中，`background.js` 在具有 DOM 的常駐背景頁面中執行。在 Chromium 的 Manifest V3 中，背景改以 service worker 方式執行：

- 沒有 DOM。`window`、`document`、`XMLHttpRequest` 與 `localStorage` 都消失了。請改用 `fetch` 與 `chrome.storage`。
- 當 worker 閒置時，**可能隨時被終止**，並在下一次事件時喚醒。請不要將狀態存在模組層級的變數中；改用 `chrome.storage` 持久化。
- 允許頂層 `await`，但長時間的初始化本身無法讓 worker 保持存活。
- 將事件監聽器**同步**註冊在檔案頂端，而不是在非同步回呼內。註冊在非同步回呼內的監聽器，無法在載入 worker 的喚醒事件上觸發。

Firefox 使用非常駐的事件頁面，而不是 service worker。Extension.js 會從同一份原始碼為 Chromium 路由到 `service_worker`、為 Firefox 路由到 `scripts`，使同一份背景進入點能依各目標正確編譯。

## Manifest V3 中的 `web_accessible_resources`

Manifest V3 將 `web_accessible_resources` 從扁平的檔案陣列改為一組 `{ resources, matches }` 區塊：

```json
{
  "web_accessible_resources": [
    {
      "resources": ["images/logo.png", "pages/injected.html"],
      "matches": ["https://example.com/*"]
    }
  ]
}
```

常見錯誤：

- 列出未在建置輸出中的路徑。Extension.js 只會輸出從 `manifest.json`、進入點 HTML 或匯入程式碼中參照到的檔案。請將檔案以資產形式加入，才能進入 `dist/<browser>`。
- 忘了 `matches`。沒有它，該資源不會被任何來源存取。
- 使用 V2 風格的扁平字串。瀏覽器會在 MV3 中靜默忽略它們。

完整模式（包括將擴充功能 URL 注入到 content script）請參見 [`web_accessible_resources` 實作](/docs/implementation-guide/web-accessible-resources)。

## `host_permissions` 與 `permissions`

Manifest V3 將主機存取權從 `permissions` 陣列中拆分出來：

| 鍵                                                       | 控制的內容                                                                                       |
| -------------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| `permissions`                                            | 指名的 API 介面（`storage`、`tabs`、`cookies`、`scripting`、`alarms` 等）。                       |
| `host_permissions`                                       | 擴充功能可讀取或修改的 URL 比對模式（`https://*/*`、`*://api.example.com/*`）。                   |
| `optional_permissions` 與 `optional_host_permissions`    | 透過 `chrome.permissions.request` 在執行時請求的權限。                                            |

混用會出現的徵兆：

- `chrome.cookies.get` 對你有存取權的站點回傳空值：URL 需要 `host_permissions`，而不只是 `cookies`。
- `chrome.scripting.executeScript` 失敗並顯示「Cannot access contents of url」：請將該 URL 加入 `host_permissions`。
- Web Store 警告使用者擴充功能有「所有網站」存取權，但其實你只需要一個來源：請縮小主機模式。

各 API 的對應參考請參見[權限與主機權限](/docs/implementation-guide/permissions-and-host-permissions)。

## Firefox 中的 `declarative_net_request`

`declarative_net_request`（DNR）是 MV3 用來取代封鎖式 `webRequest` 的機制。Firefox 支援它，但有一些限制：

- 靜態規則資源檔案（`rule_resources`）必須是有效的 JSON 陣列。對於空白或格式錯誤的規則檔案，Firefox 比 Chrome 更嚴格。
- Chrome 支援的某些規則動作與條件，在 Firefox 中仍是部分支援。當前對照可參考 [MDN 的 `declarativeNetRequest` 文件](https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/API/declarativeNetRequest)。
- 要讓 DNR 規則套用於跨來源請求，需要 `host_permissions`（或 `<all_urls>`）。

Extension.js 會在建置時驗證 DNR 資源，並輸出各瀏覽器專屬的成品，使 Firefox 專屬的規則檔案不會出現在 Chrome 建置中。

## Content script、worker 與擴充功能 URL

MV3 在三個地方常讓人困惑：

- **Content script 無法存取頁面的 JavaScript 範圍。** 它們共享 DOM，但不共享 window。使用 `window.postMessage`（或指向 `web_accessible_resources` 條目的 `<script>` 標籤）與頁面程式碼通訊。
- **Service worker 在安裝期間無法可靠地呼叫 `chrome.runtime.getURL`。** 請在需要的事件處理器內延後解析 URL。
- **擴充功能 URL 以來源（origin）為界。** 你的擴充功能內的兩個頁面可以彼此通訊，但除非列在 `web_accessible_resources` 中，否則無法從網頁抓取它們。

## Extension.js 如何協助處理 Manifest V3

- 自動將背景進入點路由到 `service_worker`（Chromium）或 `scripts`（Firefox）。
- 當背景使用 ES 模組語法時，自動設定 `type: "module"`。
- 在建置時依當前的 manifest 版本驗證 `web_accessible_resources`、`permissions` 與 `host_permissions`。
- 輸出各瀏覽器的 `dist/<browser>` 成品，使 Chrome 與 Firefox 之間的 MV3 差異保持隔離。
- 在 `extension dev` 期間，於儲存時重新載入 content script、service worker 與 HTML 頁面。
