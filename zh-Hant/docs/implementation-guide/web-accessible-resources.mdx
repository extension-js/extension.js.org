僅暴露網頁或外部擴充功能情境必須存取的擴充功能資產。

Extension.js 會將使用者宣告的 `web_accessible_resources` 與建置期探索到的需求合併，並套用特定目標的 manifest 正規化。

## Web-accessible 資源能力

| 能力                           | 為您提供的功能                                                                                  |
| ------------------------------ | ----------------------------------------------------------------------------------------------- |
| Manifest V2/V3 schema 處理     | 根據 manifest 版本處理陣列與物件兩種 `web_accessible_resources` 形狀                            |
| 具建置意識的合併               | 將宣告的資源與建置／執行階段所需項目合併                                                        |
| 路徑正規化                     | 將 WAR（web-accessible resource）路徑解析至每個瀏覽器目標的正確輸出位置                         |
| 安全控制面                     | 透過 `resources` 與 `matches` 明確保留存取規則                                                  |

## Manifest schema

### Manifest V3

```json
{
  "web_accessible_resources": [
    {
      "resources": ["images/*.png", "fonts/*.woff2"],
      "matches": ["<all_urls>"]
    }
  ]
}
```

### Manifest V2

```json
{
  "web_accessible_resources": ["images/*.png", "scripts/*.js", "styles/*.css"]
}
```

## 範例 `manifest.json` 檔案

若您透過支援的 content-script 或建置路徑匯入資產，Extension.js 會自動加入所需的 WAR 進入點。若網頁需要直接存取特定資產，請手動宣告這些資源。

```json
{
  "manifest_version": 3,
  "name": "My Extension",
  "version": "1.0.0",
  "web_accessible_resources": [
    {
      "resources": ["images/*.png", "scripts/*.js", "styles/*.css"],
      "matches": ["<all_urls>"]
    }
  ],
  "content_scripts": [
    {
      "matches": ["https://example.com/*"],
      "css": ["styles/content.css"],
      "js": ["scripts/content.js"]
    }
  ]
}
```

## Extension.js 會自動加入哪些內容

當執行階段需要某些建置輸出資產可被頁面存取時，Extension.js 可以從建置輸出擴充 WAR 進入點。

常見範例包括：

- 由 content scripts 匯入的資產。
- Content-script CSS 輸出。
- 由那些進入點參照、已輸出的字型或靜態資產。
- 重新載入行為所需的開發專用執行階段支援。

自動納入很方便，但您仍應檢視最終暴露給網頁的資產集合。

## 開發行為

- 在開發期間，Extension.js 會修補 WAR 進入點以支援重新載入與 hot module replacement（HMR）資產存取。
- Extension.js 可在需要時自動納入 content-script 相關的資產到 WAR。
- Extension.js 透過移除 `public/` 前綴與開頭的 `/` 來正規化路徑，以產生輸出安全的路徑。

## 輸出與解析註記

- Extension.js 透過公開資產流程複製 `public/` 中的資源，您可以從擴充功能根目錄路徑宣告它們。
- 相對 WAR 資源路徑會從 manifest 資料夾解析。
- 您可以使用 globs，但 match 樣式必須符合目標瀏覽器的限制。
- Extension.js 在正規化期間會丟棄沒有 `resources` 的 Manifest V3（MV3）進入點。
- Extension.js 會原樣保留 globs，不會將其展開為明確的檔案清單。

## 執行階段用法

請使用瀏覽器執行階段 URL 助手，而非手動建構擴充功能 URL：

```ts
const logoUrl = chrome.runtime.getURL("/images/logo.png");
```

若頁面或 content script 無法讀取您期望可存取的資產，請檢查：

1. 該資產是否實際存在於輸出中。
2. WAR 進入點是否暴露了正確的 `resources`。
3. `matches` 範圍是否包含嘗試讀取它的頁面。
4. 該資產是否應該僅保持為擴充功能頁面私有。

## 最佳實務

- 將 WAR 進入點保持最小；僅暴露外部情境必須讀取的內容。
- 為更好的安全性，優先使用明確的資源路徑而非廣泛的 globs。
- 跨 Chromium 與 Firefox 目標驗證 WAR match 樣式與資源路徑。
- 使用 `chrome.runtime.getURL()`／`browser.runtime.getURL()` 建立執行階段安全的 URL。
- 在新增 content-script 匯入、字型或頁面可存取靜態資產後重新審視 WAR。

### 安全檢查清單

- 將 `matches` 範圍限縮至實際需要存取的網域。
- 在小型明確清單已足夠的情況下，避免使用廣泛的萬用字元資源 globs。
- 將 WAR 進入點專注於非敏感資產。
- 在新增 content script 匯入或執行階段資產查詢時重新審視 WAR。

### 限縮 vs 過於廣泛的範例

```json
{
  "web_accessible_resources": [
    {
      "resources": ["images/logo.png", "fonts/inter.woff2"],
      "matches": ["https://example.com/*"]
    }
  ]
}
```

```json
{
  "web_accessible_resources": [
    {
      "resources": ["*"],
      "matches": ["<all_urls>"]
    }
  ]
}
```
