讓您的擴充功能保持功能性，同時不請求超過功能所需的權限。

Extension.js 會編譯並驗證您的擴充功能，但瀏覽器仍會強制執行 `manifest.json` 中的 `permissions`、`host_permissions` 與 `optional_*` 欄位。良好的權限設計能提升信任，並減少商店審查時的問題。

## 權限策略

| 需求                                                   | 建議使用                    |
| ------------------------------------------------------ | --------------------------- |
| 擴充功能運作所需的核心 API 存取                        | `permissions`               |
| 安裝時所需的主機存取                                   | `host_permissions`          |
| 之後可以請求的能力                                     | `optional_permissions`      |
| 僅部分使用者需要的網域存取                             | `optional_host_permissions` |

## 常見模式

### 由 background 驅動的功能

針對所需的瀏覽器 API 使用 `permissions`，並僅加入功能所需的主機樣式：

```json
{
  "manifest_version": 3,
  "permissions": ["storage", "tabs", "scripting"],
  "host_permissions": ["https://example.com/*"]
}
```

### 可選能力

若首次執行體驗不需要某項功能，請將其保留為可選：

```json
{
  "manifest_version": 3,
  "permissions": ["storage"],
  "optional_permissions": ["notifications"],
  "optional_host_permissions": ["https://api.example.com/*"]
}
```

## 實務規則

- 僅在功能確實需要時才請求 API 權限。
- 將 `host_permissions` 限縮至可運作的最小來源集合。
- 對於次要或進階功能優先使用可選權限。
- 當新增 background 動作、content-script 注入或遠端 API 呼叫時，重新審視權限。

## 各類功能的權限設計

| 功能                                       | 通常涉及                                                |
| ------------------------------------------ | ------------------------------------------------------- |
| 在已知網站上的 content-script 強化         | 該網站的 `host_permissions`，有時還有 `scripting`       |
| 僅 Popup 或 options UI                     | 通常完全不需要主機權限                                  |
| 程式化注入                                 | `scripting` 加上對應的主機權限                          |
| 跨分頁工作流程                             | `tabs`，有時還有 `activeTab`                            |
| 持久化設定                                 | `storage`                                               |
| 通知或徽章                                 | `notifications`、與 `action` 相關的 manifest 欄位       |

## 常見錯誤

- 在僅需要一兩個來源時使用 `<all_urls>` 之類的廣泛萬用字元。
- 將 `host_permissions` 混入原本可透過範圍較窄的使用者觸發工作流程實現的功能。
- 忘記 content scripts、web-accessible 資源，以及透過 `chrome.scripting` 進行的 script 注入各有不同的安全意涵。
- 在功能文件中記錄權限，卻沒有說明每個權限存在的原因。

## 審查檢查清單

1. 列出每個面向使用者的功能。
2. 將每個功能對應到所需的精確 API 與主機權限。
3. 在可能的情況下將非核心權限移至可選權限。
4. 移除舊有實驗留下的過時權限。
5. 在變更後重新測試安裝提示與瀏覽器商店預期。
