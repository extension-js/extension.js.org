## 本章涵蓋的內容

- 僅靠 CLI 文件容易遺漏的撰寫合約。
- 由您的原始檔驅動、影響擴充功能載入與重新載入的執行階段行為。
- 即使在打包之後仍然重要的瀏覽器擴充功能架構議題，例如權限、訊息傳遞、儲存，以及 background、content script 與頁面情境之間的分離。
- 無論您是手動開發或使用 AI 程式助理，都適用的實務指引。

## 重要心智模型

Extension.js 協助處理編譯、路徑正規化、manifest 重寫，以及開發人員體驗。它並未消除底層的瀏覽器擴充功能邊界：

- Content scripts 仍以頁面情境的限制執行。
- Background service worker 仍受到生命週期限制。
- Web-accessible 資源仍需要明確的暴露。
- 權限仍決定擴充功能可以做什麼。
- 訊息傳遞與儲存仍需要明確的設計。

## 新功能的建議閱讀順序

1. 從 [manifest.json](/docs/implementation-guide/manifest-json) 開始，了解您所宣告的擴充功能表面。
2. 閱讀與功能相關的頁面，例如 [content scripts](/docs/implementation-guide/content-scripts) 或 [background](/docs/implementation-guide/background)。
3. 補上支援元件：[權限](/docs/implementation-guide/permissions-and-host-permissions)、[訊息傳遞](/docs/implementation-guide/messaging)、[儲存](/docs/implementation-guide/storage)、[web-accessible 資源](/docs/implementation-guide/web-accessible-resources)。
4. 在 [可預測的路徑解析](/docs/features/path-resolution) 中驗證路徑假設。
5. 在開發監看模式期間變更進入點之前，先檢視 [開發更新行為](/docs/workflows/dev-update-behavior)。
