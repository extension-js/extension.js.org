理解開發實際擴充功能時最重要的合約。先弄清楚 `manifest.json` 控制哪些範圍、Extension.js 會改寫哪些進入點,接著再決定哪些行為可以安心交給自動化處理、哪些限制仍然必須遵守。

## 本章節涵蓋的內容

- 僅靠 CLI 文件容易忽略的開發合約。
- 由原始檔驅動、影響擴充功能載入與重新載入方式的執行階段行為。
- 即使打包完成,依然必須關注的瀏覽器擴充功能架構議題,例如權限、訊息傳遞、儲存,以及背景、content script 與頁面情境之間的隔離。
- 不論你是手動開發或搭配 AI 程式碼代理,都適用的實務指引。

## 重要的心智模型

Extension.js 協助你處理編譯、路徑正規化、manifest 改寫與開發體驗最佳化,但它不會抹除瀏覽器擴充功能本身的邊界:

- Content scripts 仍然受到頁面情境的限制。
- 背景 Service Worker 仍然有生命週期限制。
- Web Accessible Resources 仍然需要明確開放。
- 權限仍然決定擴充功能可以做哪些事。
- 訊息傳遞與儲存仍然需要明確的設計。

## 開發新功能時的建議閱讀順序

1. 從 [manifest.json](/zh-Hant/docs/implementation-guide/manifest-json) 開始,理解你正在宣告的擴充功能介面。
2. 閱讀對應功能的章節,例如 [content scripts](/zh-Hant/docs/implementation-guide/content-scripts) 或 [背景](/zh-Hant/docs/implementation-guide/background)。
3. 補上支援用的構件:[權限](/zh-Hant/docs/implementation-guide/permissions-and-host-permissions)、[訊息傳遞](/zh-Hant/docs/implementation-guide/messaging)、[儲存](/zh-Hant/docs/implementation-guide/storage) 與 [Web Accessible Resources](/zh-Hant/docs/implementation-guide/web-accessible-resources)。
4. 在 [可預期的路徑解析](/zh-Hant/docs/features/path-resolution) 驗證路徑相關假設。
5. 在開發監看模式下變更進入點前,先複習 [開發更新行為](/zh-Hant/docs/workflows/dev-update-behavior)。
