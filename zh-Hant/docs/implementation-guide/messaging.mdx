明確地在擴充功能情境之間移動資料，並驗證每個特權邊界。

Extension.js 會將 background scripts、content scripts 與擴充功能頁面打包在一起。您的擴充功能情境之間仍透過瀏覽器擴充功能訊息傳遞 API 通訊。良好的訊息傳遞結構能清楚劃分每個情境的職責，讓除錯更可靠。

## 情境邊界

| 從                                 | 到                                            | 典型 API                                       |
| ---------------------------------- | --------------------------------------------- | ---------------------------------------------- |
| Popup 或 options 頁面              | Background service worker                     | `runtime.sendMessage()`                        |
| Content script                     | Background service worker                     | `runtime.sendMessage()` 或 `runtime.connect()` |
| Background service worker          | Content script                                | `tabs.sendMessage()`                           |
| 長時間串流或狀態同步               | Background service worker 與另一個情境        | `runtime.connect()`                            |

## 建議的合約

將訊息視為具型別的協定：

- 明確包含 `type`。
- 在動作前驗證 payload 形狀。
- 對特權操作驗證寄件者情境。
- 回傳結構化的成功與錯誤回應。

```ts
type Message =
  | { type: "settings:get" }
  | { type: "settings:set"; payload: { theme: "light" | "dark" } };

chrome.runtime.onMessage.addListener((message, sender, sendResponse) => {
  if (
    !message ||
    typeof message !== "object" ||
    typeof message.type !== "string"
  ) {
    sendResponse({ ok: false, error: "invalid_message" });
    return;
  }

  if (message.type === "settings:get") {
    sendResponse({ ok: true, data: { theme: "dark" } });
    return;
  }

  if (message.type === "settings:set") {
    sendResponse({ ok: true });
    return;
  }

  sendResponse({ ok: false, error: "unknown_message" });
});
```

## 何時用 `sendMessage` 與 `connect`

| 使用情境                                          | 建議使用                |
| -------------------------------------------------- | ----------------------- |
| 單一請求與單一回應                                 | `runtime.sendMessage()` |
| 長時間串流或多次更新                               | `runtime.connect()`     |
| 與特定分頁的 content script 通訊                   | `tabs.sendMessage()`    |

## 良好架構

- 每個功能領域維護一個訊息處理器模組，而非用一個大型檔案處理所有訊息類型。
- 將特權工作集中於 background service worker。
- 使用 content scripts 收集頁面資料，僅轉發所需的最小 payload。
- 透過將瀏覽器 API 呼叫委派至 background 程式碼，讓 popup 與 options 頁面保持輕薄。

## 安全規則

- 切勿因訊息來自 content script 就信任頁面派生的內容。
- 在執行特權動作前驗證寄件者身分。
- 明確拒絕未知的訊息類型。
- 避免透過訊息傳遞暴露通用的「執行任何內容」或「擷取任何內容」命令。

## 常見錯誤

- 讓 UI 頁面在多處直接呼叫特權 API，而非透過 background 路由。
- 當小型結構化 payload 已足夠時，傳送大型任意頁面快照。
- 把 content scripts 當作是受信任的，因為它們是您寫的程式碼，但它們仍會看到不受信任的頁面資料。
- 在多個情境一起演進時忘記為訊息形狀進行版本控制或遷移。

## 實務模式

### Popup 請求目前狀態

從 popup 到 background 採用單一請求-回應訊息。

### Content script 請求特權工作

收集盡可能小的頁面 payload，將其傳送到 background，由 background 決定是否允許該動作。

### 即時訂閱

僅在您真正需要連續更新時才使用 `runtime.connect()`，不要用於簡單的請求-回應流程。
