在問題到達執行階段之前,讓擴充功能程式碼品質保持可預期。ESLint 在建置管線之外執行。從你的指令稿或持續整合(CI)中執行它。

## 何時適合使用 ESLint

- 你希望團隊貢獻之間保持一致的程式碼標準。
- 你需要 CI 檢查盡早捕捉常見的 JS/TS 錯誤。
- 你正在發布前強制執行擴充功能特定的品質規則。

## ESLint 能力

| 能力                              | 它能為你提供什麼                                                          |
| --------------------------------- | ------------------------------------------------------------------------- |
| JavaScript 與 TypeScript 檢查     | 捕捉常見錯誤並強制執行專案標準                                            |
| 扁平設定支援                      | 在新專案中使用現代的 `eslint.config.mjs`                                  |
| 適合 CI 的執行                    | 在本機指令稿和 pull request 管線中執行 lint 檢查                          |
| 擴充功能特定的規則集              | 當團隊需要更嚴格的檢查時,加入瀏覽器擴充功能外掛/規則                      |

## 與既有擴充功能一起使用

安裝核心套件:

<PackageManagerTabs command="install -D eslint @eslint/js globals typescript-eslint" />

建立 `eslint.config.mjs`:

```js
import globals from "globals";
import pluginJs from "@eslint/js";
import tseslint from "typescript-eslint";

export default [
  { files: ["**/*.{js,mjs,cjs,ts,tsx,jsx}"] },
  {
    languageOptions: {
      globals: globals.browser,
    },
  },
  pluginJs.configs.recommended,
  ...tseslint.configs.recommended,
];
```

從你的指令稿執行 lint 命令:

<CodeGroup>

```bash npm
npm run lint
```

```bash pnpm
pnpm lint
```

```bash yarn
yarn lint
```

```bash bun
bun run lint
```

```bash bun
bun run lint
```

</CodeGroup>

## 最佳實務

- 將 lint 規則保留在原始碼控制中,並在 CI 中套用。
- 使用 lint-staged 或預先 commit hook 獲得更快的本機回饋。
- 僅在需要時加入框架特定外掛(例如 React)。
