在問題抵達執行期前,讓擴充功能的程式碼品質保持可預期。ESLint 在建置 pipeline 之外執行。請從你的腳本或持續整合(CI)中執行它。

## 何時適合用 ESLint

- 你希望團隊貢獻間有一致的程式碼標準。
- 你需要 CI 檢查及早抓出常見的 JS/TS 錯誤。
- 你想在發布前強制執行擴充功能專屬的品質規則。

## ESLint 能力

| 能力                              | 它能為你提供什麼                                                         |
| --------------------------------- | ------------------------------------------------------------------------ |
| JavaScript 與 TypeScript lint     | 抓出常見錯誤並落實專案標準                                               |
| Flat config 支援                  | 在新專案中使用現代的 `eslint.config.mjs`                                |
| 對 CI 友善的執行                  | 在本機腳本與 pull request pipeline 中執行 lint 檢查                      |
| 擴充功能專屬規則組                | 當團隊需要更嚴格的檢查時,加入瀏覽器擴充功能的 plugin/規則                |

## 與既有擴充功能搭配使用

安裝核心套件:

<PackageManagerTabs command="install -D eslint @eslint/js globals typescript-eslint" />

建立 `eslint.config.mjs`:

```js
import globals from "globals";
import pluginJs from "@eslint/js";
import tseslint from "typescript-eslint";

export default [
  { files: ["**/*.{js,mjs,cjs,ts,tsx,jsx}"] },
  {
    languageOptions: {
      globals: globals.browser,
    },
  },
  pluginJs.configs.recommended,
  ...tseslint.configs.recommended,
];
```

從你的腳本執行 lint 指令:

<CodeGroup>

```bash npm
npm run lint
```

```bash pnpm
pnpm lint
```

```bash yarn
yarn lint
```

```bash bun
bun run lint
```

```bash bun
bun run lint
```

</CodeGroup>

## 最佳實務

- 把 lint 規則納入版本控制,並在 CI 中套用。
- 使用 lint-staged 或 pre-commit hook 來取得更快的本機回饋。
- 只在需要時加入框架專屬 plugin(例如 React)。
