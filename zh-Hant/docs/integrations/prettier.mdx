以最少的人工成本維持擴充功能程式碼庫的格式一致。Prettier 可從腳本、編輯器整合與持續整合(CI)中執行。它不會在 Extension.js 建置 pipeline 中執行。

## 何時適合用 Prettier

- 你希望在本機與 CI 工作流程中取得決定性的格式。
- 你正在招募貢獻者,並需要低摩擦的風格一致性。
- 你希望 ESLint 專注於正確性,而 Prettier 處理格式。

## Prettier 能力

| 能力                       | 它能為你提供什麼                                                |
| -------------------------- | --------------------------------------------------------------- |
| 一致的格式                 | 對 JS、TS、JSON、CSS 與 Markdown 提供一套共用樣式               |
| 腳本與編輯器工作流程       | 在本機儲存時格式化,並在 CI 中強制執行                          |
| 減少 lint 衝突             | 搭配 `eslint-config-prettier` 來避免規則重疊                    |
| 低維護的預設值             | 用最少的設定採用穩定的基準                                      |

## 與既有擴充功能搭配使用

安裝 Prettier:

<PackageManagerTabs command="install -D prettier" />

建立設定檔(例如 `.prettierrc`):

```json
{
  "semi": false,
  "singleQuote": true,
  "trailingComma": "none",
  "printWidth": 80,
  "tabWidth": 2,
  "arrowParens": "always"
}
```

從腳本或手動執行 Prettier:

<CodeGroup>

```bash npm
npm run format
```

```bash pnpm
pnpm format
```

```bash yarn
yarn format
```

```bash bun
bun run format
```

```bash bun
bun run format
```

</CodeGroup>

如果還沒有 format 腳本:

## 與 ESLint 整合

要避免 ESLint 中的格式規則衝突,請安裝:

<PackageManagerTabs command="install -D eslint-config-prettier" />

接著在你的 ESLint 設定鏈中加入 `eslint-config-prettier`,讓 ESLint 把格式議題交給 Prettier 處理。

## 最佳實務

- 在編輯器與 CI 中保持自動格式化,而非手動。
- 在 CI 中使用 `--check`,在本機使用 `--write`。
- 在較大的 repo 中,在 pre-commit 階段格式化暫存的檔案。
