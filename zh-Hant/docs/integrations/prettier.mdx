以最少的手動作業在擴充功能程式碼基底中保持一致的格式化。Prettier 透過指令稿、編輯器整合和持續整合(CI)執行。它不在 Extension.js 建置管線中執行。

## 何時適合使用 Prettier

- 你希望在本機和 CI 工作流程中有確定性的格式化。
- 你正在引入貢獻者,需要低阻力的風格一致性。
- 你希望 ESLint 專注於正確性,而 Prettier 處理格式化。

## Prettier 能力

| 能力                       | 它能為你提供什麼                                              |
| -------------------------- | ------------------------------------------------------------- |
| 一致的格式化               | 為 JS、TS、JSON、CSS 和 Markdown 提供一種共享風格             |
| 指令稿與編輯器工作流程     | 在本機儲存時格式化,在 CI 中強制執行                          |
| 減少 lint 衝突             | 與 `eslint-config-prettier` 搭配以避免規則重疊                |
| 低維護的預設值             | 以最少的設定採用穩定的基準                                    |

## 與既有擴充功能一起使用

安裝 Prettier:

<PackageManagerTabs command="install -D prettier" />

建立一個設定檔(例如 `.prettierrc`):

```json
{
  "semi": false,
  "singleQuote": true,
  "trailingComma": "none",
  "printWidth": 80,
  "tabWidth": 2,
  "arrowParens": "always"
}
```

從指令稿或手動執行 Prettier:

<CodeGroup>

```bash npm
npm run format
```

```bash pnpm
pnpm format
```

```bash yarn
yarn format
```

```bash bun
bun run format
```

```bash bun
bun run format
```

</CodeGroup>

如果還沒有 format 指令稿:

## 與 ESLint 整合

為避免 ESLint 中的格式化規則衝突,安裝:

<PackageManagerTabs command="install -D eslint-config-prettier" />

然後在你的 ESLint 設定鏈中加入 `eslint-config-prettier`,讓 ESLint 將格式化交給 Prettier 處理。

## 最佳實務

- 讓格式化在編輯器和 CI 中自動進行,而不是手動。
- 在 CI 中使用 `--check`,在本機使用 `--write`。
- 在較大的儲存庫中,在 pre-commit 時格式化已 staged 的檔案。
