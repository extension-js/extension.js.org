[chrome-devtools-mcp](https://github.com/ChromeDevTools/chrome-devtools-mcp) 讓 AI agent 能夠原始地控制一個執行中的 Chrome 實例,最近的版本還新增了擴充功能工具(`install`、`list`、`reload`、`trigger`、`uninstall`,以及 service-worker 日誌)。Extension.js MCP 伺服器(`@extension.dev/mcp`)涵蓋了相同的擴充功能能力,但具有建置感知,並支援跨瀏覽器。

它們是互補的,而不是競爭關係。兩個一起執行:讓 chrome-devtools-mcp 驅動 Chrome 本身,讓 Extension.js 擁有所有依賴於你專案的內容 —— 建置、熱重新載入、產生的 manifest、storage、跨上下文的日誌,以及在 Firefox、Edge 和 Safari 上的相同工作流程。

## 何時使用哪個

| 任務                                                            | 使用                                                       |
| --------------------------------------------------------------- | ---------------------------------------------------------- |
| 建置、開發伺服器、熱重新載入、正式環境打包                      | Extension.js(`extension_dev`、`extension_build`)          |
| 讀取產生的跨瀏覽器 manifest                                     | Extension.js(`extension_manifest_validate`)               |
| 可指令稿地重新播放工具列 action 處理函式,無需 popup             | Extension.js(`extension_open`,`surface: "action"`)        |
| 重新播放鍵盤快捷鍵命令處理函式                                  | Extension.js(`extension_open`,`surface: "command"`)       |
| 在開發瀏覽器中列出具有即時上下文的擴充功能                      | Extension.js(`extension_list_extensions`)                 |
| 跨 SW、內容指令稿、popup、options、sidebar 的日誌               | Extension.js(`extension_logs`)                            |
| 讀寫 `chrome.storage`,在任意上下文中執行 eval                   | Extension.js(`extension_storage`、`extension_eval`)       |
| 在 Firefox、Edge 或 Safari 上使用相同流程                       | Extension.js(原生跨瀏覽器)                                |
| 驅動任意網頁,點擊/捲動、填寫表單                               | chrome-devtools-mcp                                        |
| 效能追蹤、Lighthouse 稽核、網路檢查                             | chrome-devtools-mcp                                        |
| 檢查一個不是你建置的擴充功能(沒有專案)                        | chrome-devtools-mcp                                        |
| 使用**真實手勢**觸發 action(授予 `activeTab`)                  | chrome-devtools-mcp(`trigger_extension_action`)           |

一個實用的經驗法則:如果任務涉及你的**專案**(原始碼、建置、manifest、重新載入),選用 Extension.js。如果它涉及一般性的**瀏覽器**操作(頁面、網路、效能),選用 chrome-devtools-mcp。

## 安裝兩個伺服器

Extension.js 的工具以 `extension_*` 命名空間,因此永遠不會與 chrome-devtools-mcp 的工具衝突。熟練使用其中一個的 agent 也能轉移到另一個上。

<CodeGroup>

```bash Claude Code
# Extension.js — 具有建置感知的擴充功能控制
claude mcp add extension-dev npx @extension.dev/mcp

# chrome-devtools-mcp — 原始 Chrome 偵錯(擴充功能工具需選擇啟用)
claude mcp add chrome-devtools npx chrome-devtools-mcp@latest --categoryExtensions
```

```json Claude Desktop (claude_desktop_config.json)
{
  "mcpServers": {
    "extension-dev": {
      "command": "npx",
      "args": ["@extension.dev/mcp"]
    },
    "chrome-devtools": {
      "command": "npx",
      "args": ["chrome-devtools-mcp@latest", "--categoryExtensions"]
    }
  }
}
```

```json Cursor (.cursor/mcp.json)
{
  "mcpServers": {
    "extension-dev": {
      "command": "npx",
      "args": ["@extension.dev/mcp"]
    },
    "chrome-devtools": {
      "command": "npx",
      "args": ["chrome-devtools-mcp@latest", "--categoryExtensions"]
    }
  }
}
```

</CodeGroup>

<Note>
  chrome-devtools-mcp 的擴充功能工具(`--categoryExtensions`)目前需要管道連線。
  透過 WebSocket 端點(`browserUrl` / `wsEndpoint`)連線到一個已經執行的瀏覽器
  在擴充功能類別下還不被支援,直到 Chrome 149。Extension.js 會連線到它為你的開發 session 啟動的瀏覽器,
  因此它的擴充功能工具今天就可以使用。
</Note>

## 能力對應

chrome-devtools-mcp 的每一個擴充功能動作在 Extension.js 中都有對應,並圍繞它建構了一個完整的建置平台。

| chrome-devtools-mcp        | Extension.js                            | 備註                                                                                                                                                                                  |
| -------------------------- | --------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `install_extension`        | `extension_dev`、`extension_preview`    | 載入與真實建置 + 熱重新載入綁定,而非靜態資料夾                                                                                                                                        |
| `list_extensions`          | `extension_list_extensions`             | 列出具有即時上下文的擴充功能;透過 Extensions 域唯讀                                                                                                                                   |
| `reload_extension`         | `extension_reload`                      | 具有重新建置感知;重新載入 background 或分頁                                                                                                                                           |
| `trigger_extension_action` | `extension_open`(`surface: "action"`)  | 可移植地重新播放 `onClicked` 處理函式 —— 跨瀏覽器,**無手勢**(見警告)。需要真實手勢點擊(activeTab)時,請使用 chrome-devtools-mcp 的 `trigger_extension_action` |
| —                          | `extension_open`(`surface: "command"`) | 重新播放 `chrome.commands.onCommand` 鍵盤快捷鍵處理函式 —— chrome-devtools-mcp 沒有對應功能                                                                                            |
| `uninstall_extension`      | 由開發 session 生命週期管理             | Extension.js 透過 `dev` / `preview` 擁有載入/卸載                                                                                                                                      |
| service-worker 日誌        | `extension_logs`                        | 跨所有上下文的一個有序時間軸,支援 `follow`                                                                                                                                            |

### action / command 觸發器如何運作 —— 以及一個警告

chrome-devtools-mcp 透過點擊真實的工具列按鈕來觸發 action,這需要一個可見視窗和真實的使用者手勢。Extension.js 採取了不同的方法:它在建置期擷取擴充功能的 `chrome.action.onClicked`(以及 `chrome.commands.onCommand`)監聽器,並按需**重新播放**它們。這使得觸發變得可指令稿化且可重現 —— 這是 agentic 測試的自然模式 —— 而且,因為它只接觸 `addListener`,它在 **Chromium 和 Firefox** 上都可以運作(均已驗證)。單一的 `background.service_worker` 來源在 Firefox 上也能運作 —— Extension.js 會將其轉換為 Firefox 目標的 `background.scripts` 事件頁面,因為 Firefox 不執行 service-worker 背景。

<Warning>
  重新播放會在**沒有使用者手勢**的情況下呼叫你的處理函式。真實的工具列點擊會授予暫時的
  `activeTab` 權限,並滿足受手勢限制的 API
  (`chrome.permissions.request`、互動式的 `identity.getAuthToken`);重新播放則不會。
  因此,依賴 `activeTab` 的處理函式(例如,在作用中的分頁上呼叫
  `chrome.scripting.executeScript` 或 `captureVisibleTab`)會與真實點擊表現不同。
  結果包含 `gesture: false`,當你的 manifest 宣告了 `activeTab` 時還會包含
  `warning`。當你需要真實的點擊保真度時,請使用 chrome-devtools-mcp 的
  `trigger_extension_action`(見下文)。
</Warning>

### 需要真實手勢點擊?使用 chrome-devtools-mcp

兩者是互補的,不是競爭的:

- **`extension_open`(`surface: "action"`)** —— 日常路徑。重新播放處理函式,跨瀏覽器,無頭,無手勢(`activeTab` 不被授予)。用於絕大多數處理函式測試。
- **chrome-devtools-mcp 的 `trigger_extension_action`** —— 透過 Puppeteer(在受支援的管道傳輸上)進行_真實_的 action 呼叫,因此 `activeTab` 會像使用者點擊一樣被授予。僅限 Chromium。僅在你的處理函式確實依賴手勢時才使用它。

由於本頁面已經推薦執行兩個伺服器,讓 chrome-devtools-mcp 擁有真實手勢的瀏覽器驅動,讓 Extension.js 擁有具有建置感知的跨瀏覽器重新播放 —— 無需重複。

請參閱[重新載入與 HMR](/docs/features/reload-and-hmr)。

### 結構上的安全性

重新播放包裝**僅注入到你的開發建置中** —— agent 橋接受一個僅在 `extension dev`/`preview` 期間存在的控制連接埠保護,正式環境建置中不會加入任何內容。`addListener` 包裝會透明地委派給原始函式,因此無論是否有橋接,你的擴充功能都會有相同的行為。
