無需自訂打包器設定即可在擴充功能中使用 WebAssembly。

Extension.js 在其 Rspack 流程中內建 WebAssembly (Wasm) 預設值,讓常見的 Wasm 工作流程無需額外設定即可運作。

## 何時適合使用 WebAssembly

- 你在擴充功能中執行計算密集型任務(解析、媒體、光學字元辨識 (OCR)、轉換等)。
- 你需要對經常執行的程式碼路徑取得接近原生的效能。
- 你正在重複使用既有 Web 專案中的 Wasm 模組。

## WebAssembly 能力

| 能力                          | 它為你帶來什麼                                  |
| ----------------------------- | ----------------------------------------------- |
| 內建 Wasm 流程預設值          | 無需手動串接打包器即可執行常見 Wasm 流程         |
| 非同步 Wasm 支援              | 於擴充功能情境中載入 `.wasm` 模組                |
| 執行時相容性輔助              | 處理常見的 Wasm 生態資源模式                     |
| 常規指令工作流程              | 使用相同的 `dev` 與 `build` 指令搭配 Wasm        |

## Extension.js 啟用了哪些設定

Wasm 外掛會設定:

- `experiments.asyncWebAssembly = true`
- 在模組解析副檔名中加入 `.wasm`
- 為常見 Wasm 函式庫設定路徑別名(例如用於媒體處理的 ffmpeg、用於影像處理的 imagemagick,以及用於 OCR 的 tesseract),讓匯入在執行時能正確解析。

這可在從擴充功能腳本/頁面匯入 Wasm 模組時減少樣板程式碼。

## 典型使用情境

- 計算密集型的解析/轉換。
- 媒體處理流程。
- OCR / 影像處理工作流程。
- 在 Web/執行時情境之間共用的效能敏感演算法。

## 最佳實踐

- 讓 Wasm 模組聚焦於熱點路徑;將協調邏輯保留在 JavaScript/TypeScript 中。
- 驗證 Wasm 在擴充功能情境(背景、內容腳本、頁面)中的輸出大小與載入時機。
- 在你要發布的瀏覽器目標(`chrome`、`edge`、`firefox`)上測試 Wasm 密集型流程。
