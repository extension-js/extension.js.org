無需自訂 bundler 設定即可在你的擴充功能中使用 WebAssembly。

Extension.js 在其 Rspack 流水線中內建 WebAssembly（Wasm）預設值，因此常見的 Wasm 工作流程無需額外設定即可運作。

## 何時適合使用 WebAssembly

- 你在擴充功能中執行計算密集的任務（解析、媒體、光學字元辨識（OCR）、轉換）。
- 你需要在頻繁執行的程式碼路徑中取得接近原生的效能。
- 你正在重複使用網頁專案中既有的 Wasm 模組。

## WebAssembly 能力

| 能力                              | 提供的內容                                            |
| --------------------------------- | ----------------------------------------------------- |
| 內建 Wasm 流水線預設值            | 無需手動設定 bundler 即可執行常見 Wasm 流程            |
| 非同步 Wasm 支援                  | 在擴充功能情境中載入 `.wasm` 模組                      |
| 執行階段相容性輔助               | 處理常見的 Wasm 生態系資產模式                         |
| 一般指令工作流程                 | 以相同的 `dev` 與 `build` 指令使用 Wasm                |

## Extension.js 提供的功能

Wasm 外掛設定：

- `experiments.asyncWebAssembly = true`
- 在模組解析的副檔名中加入 `.wasm`
- 為常見 Wasm 函式庫提供路徑別名（例如媒體處理用的 ffmpeg、影像處理用的 imagemagick，以及 OCR 用的 tesseract），讓匯入在執行階段能正確解析。

這能減少從擴充功能腳本／頁面匯入 Wasm 模組時的樣板程式碼。

## 典型使用情境

- 計算密集的解析／轉換。
- 媒體處理流程。
- OCR ／影像處理工作流程。
- 跨 web／執行階段情境共用的效能敏感演算法。

## 最佳實踐

- 讓 Wasm 模組聚焦於熱路徑；編排邏輯保留在 JavaScript／TypeScript。
- 驗證擴充功能情境（背景、內容腳本、頁面）中的輸出大小與載入時機。
- 在你交付的瀏覽器目標（`chrome`、`edge`、`firefox`）上測試 Wasm 重度流程。
