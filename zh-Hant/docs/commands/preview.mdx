啟動已建置的擴充功能輸出,以進行類生產的手動測試。

`preview` 不會編譯你的專案。它會載入既有的未封裝擴充功能根目錄,並執行瀏覽器啟動流程。

## 何時使用 `preview`

- 在不重新建置的情況下執行既有的建置輸出。
- 快速比較打包後的行為在不同瀏覽器目標之間的差異。
- 除錯與生產產物相關(而非開發/監看模式)的執行階段問題。

## Preview 指令能力

| 能力                 | 它能提供什麼                                            |
| -------------------- | ------------------------------------------------------- |
| 建置輸出驗證         | 測試真實的生產產物,無需重新建置                        |
| 瀏覽器目標檢查       | 在所選瀏覽器目標上執行編譯後的輸出                      |
| 執行器控制           | 依工作流程需要啟動或略過瀏覽器執行器                    |
| 快速人工 QA          | 在發布前快速驗證打包就緒的行為                          |

> `preview` 只負責執行。當 `dist/<browser>` 輸出存在時,它會優先使用該目錄。你也可以讓它指向另一個已包含 `manifest.json` 的未封裝擴充功能資料夾。

## `preview` 如何選擇要執行的內容

`preview` 會依下列順序檢查這些位置:

1. 所選瀏覽器目標對應的 `dist/<browser>`。
2. 提供的專案路徑或目前的工作目錄。

目標資料夾必須包含一個含有 `manifest.json` 的未封裝擴充功能。同一條指令中是否執行過建置並不重要。

## 參數與選項

| 選項                        | 別名               | 作用                                                                       | 預設值                 |
| --------------------------- | ------------------ | -------------------------------------------------------------------------- | ---------------------- |
| `[path]`                    | -                  | 從專案路徑預覽已建置的擴充功能。                                           | `process.cwd()`        |
| `--browser <browser>`       | -                  | 瀏覽器/引擎目標(`chromium`、`chrome`、`edge`、`firefox` 等)。            | `chromium`             |
| `--profile <path\|boolean>` | -                  | 瀏覽器設定檔路徑或布林型設定檔模式。                                       | 全新設定檔             |
| `--chromium-binary <path>`  | -                  | [自訂 Chromium 家族的二進位檔路徑](/docs/browsers/running-other-browsers)。 | 系統預設               |
| `--gecko-binary <path>`     | `--firefox-binary` | [自訂 Gecko 家族的二進位檔路徑](/docs/browsers/running-other-browsers)。    | 系統預設               |
| `--starting-url <url>`      | -                  | 在啟動的瀏覽器中開啟的初始 URL。                                           | 未設定                 |
| `--no-browser`              | -                  | 略過瀏覽器啟動。                                                           | 預設啟用瀏覽器啟動     |
| `--port <port>`             | -                  | 啟用執行器時的執行器/DevTools 埠號。使用 `0` 表示由作業系統分配。          | `8080`                 |
| `--host <host>`             | -                  | 開發伺服器繫結的主機。在 Docker/開發容器中使用 `0.0.0.0`。                 | `127.0.0.1`            |
| `--extensions <list>`       | -                  | 以逗號分隔的伴隨擴充功能清單或商店 URL。                                   | 未設定                 |
| `--author`                  | `--author-mode`    | 啟用維護者診斷模式。                                                       | 停用                   |

## 自動化中繼資料

`preview` 會將就緒中繼資料寫入:

- `dist/extension-js/<browser>/ready.json`

對於 `--no-browser` 流程,它提供確定性的指令狀態:

- 指令初始化期間為 `starting`
- 僅執行的驗證完成後為 `ready`
- 缺少所需輸出或啟動失敗時為 `error`
- `runId` 與 `startedAt` 用於在指令稿/代理中關聯工作階段

`preview` 不提供 `--wait` 閘道選項。對於 `preview` 自動化,請直接消費 `ready.json`。

## 行為說明

- `preview` 只負責執行,絕不會編譯專案。
- `preview` 優先使用既有的建置輸出(`dist/<browser>`),但可回退到另一個未封裝擴充功能根目錄。
- `preview` 不會執行監看模式,也不支援熱模組替換 (HMR)。
- `preview` 不支援原始碼檢視相關選項;請改用 `dev`。
- 對於指令稿/代理,請依賴 `ready.json`,不要剖析終端輸出。

## 最佳實務

- 在測試一個全新的生產產物時,請先執行 `build` 再執行 `preview`。
- 當未封裝擴充功能位於預設的專案輸出之外時,請傳入專案路徑引數。
- 在打包前使用 `--browser` 在多個目標上驗證行為。
