`dev` 用於日常的瀏覽器擴充功能開發,提供監看模式、瀏覽器啟動與 context 感知的更新行為。

`dev` 會執行開發流水線並監看你的專案檔案。它會依變更內容套用更新策略:熱模組替換(HMR)、硬重新載入,或在需要時做完整重啟。

## 何時使用 `dev`

- 開發功能並即時驗證變更。
- 在一個或多個瀏覽器目標上除錯擴充功能行為。
- 透過 `--source` 執行原始碼檢查流程。

需要生產產物時用 `build`、需要生產建置加啟動時用 `start`、只想啟動既有的建置輸出時用 `preview`。

如果你的擴充功能位於 monorepo/submodule 中,請參閱 `extension.config.*` 如何載入 env 檔(含 workspace 根目錄回退):[環境變數](/zh-Hant/docs/features/environment-variables#how-it-works)。

## Dev 指令能做什麼

| 功能                  | 你會得到什麼                                                |
| --------------------- | ----------------------------------------------------------- |
| 監看模式迭代          | 編碼時緊湊的編輯 → 重新建置 → 驗證循環                      |
| 瀏覽器目標控制        | 每個指令明確指定跨瀏覽器驗證                                |
| profile 感知執行      | 依工作流需求選擇可靠的全新或持久 profile                    |
| 原始碼檢查模式        | 透過 source flag 提供結構化輸出與診斷資訊                   |

## 自動化 metadata(建議用於腳本/agent)

`dev` 執行時,Extension.js 會輸出機器可讀的 metadata 到:

- `dist/extension-js/<browser>/ready.json`
- `dist/extension-js/<browser>/events.ndjson`(以換行分隔的 JSON)

進行自動化(Playwright、持續整合 (CI)、AI agent)時,建議優先使用這些檔案,而非解析終端機 log。

把 `ready.json` 視為就緒契約:

- 啟動中時 `status: "starting"`
- runtime 就緒時 `status: "ready"`
- 啟動/編譯失敗時 `status: "error"`
- `runId` 唯一識別一次 runtime session
- `startedAt` 標記 runtime session 的開始時間戳

### `--no-browser` 與就緒同步

`--no-browser` 只會停用瀏覽器啟動,並不會在編譯完成前阻擋外部 runner。

對於 Playwright/CI/AI 工作流:

1. 把 `extension dev --no-browser` 當作長時間執行的程序。
2. 把 `extension dev --wait --browser=<browser>` 當作就緒門檻。
3. 只在 `status: "ready"` 之後啟動外部瀏覽器自動化。

`--wait` 鎖定第二個程序(或 CI 步驟),在 `error`/逾時時以非零代碼退出。
當 `--wait` 看到已死程序留下的過期 `ready.json`(`pid` 不再存活),它會繼續等待一個有效的 producer。
如果你在同一個指令呼叫中同時傳入 `--wait` 與 `--no-browser`,`--wait` 優先。指令會以僅等待模式執行。

### `--no-reload` 提供乾淨的 dev bundle

`--no-reload` 會跳過 content-script 重新注入包裝器以及重新建置後的 reload 分派。dev `dist` 會更接近生產 bundle,且當檔案變更時不會打擾已開啟的分頁。請自行重新載入擴充功能或頁面以套用變更。

`--no-reload` 僅在 `extension dev` 中支援。若傳給 `start`、`preview` 或 `build` 會直接報錯退出。內部會設定 `EXTENSION_NO_RELOAD=true`,讓 develop 程序可以從 CLI argv 之外讀取。

## 原始碼檢查工作流

source 選項只能用於 `dev`。原始碼檢查會在瀏覽器中開啟目標 URL,並在你的 content script 執行 **之後** 印出完整、即時的 HTML。用它來驗證擴充功能實際改了哪些內容。

- `--source [url]` 檢查頁面(預設為 `--starting-url` 或 `https://example.com`)
- `--watch-source` / `--no-watch-source` 在每次重新建置時重新印出 HTML

`start` 與 `preview` 不支援原始碼檢查 flag。

### Source flag

| Flag                                            | 用途                                       | 預設行為                              |
| ----------------------------------------------- | ------------------------------------------ | ------------------------------------- |
| `--source [url]`                                | 啟用原始碼檢查流程。                       | 預設停用                              |
| `--watch-source [boolean]`                      | 在監看到的更新時重新印出原始碼輸出。       | 啟用 `--source` 時預設為 `true`       |
| `--source-format <pretty\|json\|ndjson>`        | 原始碼檢查的輸出格式。                     | 退回到 log 格式或 `json`              |
| `--source-summary [boolean]`                    | 輸出精簡摘要。                             | 預設停用                              |
| `--source-meta [boolean]`                       | 包含頁面 metadata。                        | 啟用 source 時預設啟用                |
| `--source-probe <selectors>`                    | 逗號分隔的 CSS 選擇器以做探測。            | 未設定                                |
| `--source-tree <off\|root-only>`                | 包含精簡的擴充功能 tree 詳情。             | 未設定                                |
| `--source-console [boolean]`                    | 包含 console 摘要。                        | 預設停用                              |
| `--source-dom [boolean]`                        | 包含 DOM 快照/差異。                       | 啟用 watch-source 時預設啟用          |
| `--source-max-bytes <bytes>`                    | 限制輸出 payload 大小。                    | 未設定                                |
| `--source-redact <off\|safe\|strict>`           | 原始碼輸出的遮罩策略。                     | 依格式而定                            |
| `--source-include-shadow <off\|open-only\|all>` | 控制 Shadow DOM 的包含程度。               | 啟用 source 時為 `open-only`          |
| `--source-diff [boolean]`                       | 在監看更新時包含 diff metadata。           | 啟用 watch-source 時預設啟用          |

## Monorepo 與 workspace 根目錄

你可以把 `dev`(以及 `build`)指向 **monorepo 根目錄**,而不是擴充功能套件本身。Extension.js 會偵測 workspace 根目錄,並自動解析其中的擴充功能套件:

```bash
extension dev .            # 從 monorepo 根目錄執行
```

當剛好找到一個擴充功能套件時,Extension.js 會解析它並印出:

```text
Workspace root detected — resolved extension package: packages/my-extension
```

當有多個候選時,它會列出讓你指向你要的那一個:

```bash
extension dev packages/my-extension
```

## 最佳實務

- **瀏覽器相容性:** 在不同瀏覽器中測試你的擴充功能,以確認在每個目標上都能運作。
- **Polyfill:** 如果 Firefox 或 Gecko 系瀏覽器也是目標,請使用 `--polyfill`。此 flag 會在 Chromium 系瀏覽器啟用 `browser.*` API 相容性。
- **原始碼檢查工作流:** 需要原始碼檢查選項時,使用 `dev`(而非 `start`/`preview`)。
- **自動化可靠性:** 把 `dev` 當作監看模式搭檔(`--no-browser` + `dev --wait`)。把 `start` 當作生產搭檔(`--no-browser` + `start --wait`)。腳本與 CI 自動化請使用 `--wait-format=json`。
