使用 `dev` 進行日常的瀏覽器擴充功能開發，搭配監看模式、瀏覽器啟動與情境感知更新行為。

`dev` 會執行開發管線並監看你的專案檔案。它會依據變更的內容套用不同的更新策略：模組熱替換（HMR）、強制重新載入，或在變更需要時進行完整重啟。

## 何時使用 `dev`

- 建構功能並即時驗證變更。
- 在一個或多個瀏覽器目標上偵錯擴充功能行為。
- 使用 `--source` 執行原始碼檢視工作流程。

使用 `build` 產生生產產出物、`start` 進行生產建置 + 啟動，使用 `preview` 僅執行既有的建置輸出。

如果你的擴充功能位於 monorepo／submodule 內，請查看 `extension.config.*` 如何載入 env 檔案（包含 workspace 根目錄回退）：[環境變數](/docs/features/environment-variables#how-it-works)。

## Dev 指令功能

| 功能                  | 提供內容                                                |
| --------------------- | ------------------------------------------------------- |
| 監看模式迭代          | 撰寫程式時形成緊密的編輯 → 重建 → 驗證循環              |
| 瀏覽器目標控制        | 每個指令明確進行跨瀏覽器驗證                            |
| 具設定檔意識的執行    | 依工作流程需要使用可靠的全新或持久設定檔                |
| 原始碼檢視模式        | 透過 source 旗標取得結構化輸出與診斷資訊                |

## 自動化中繼資料（建議用於指令稿／代理）

`dev` 執行時，Extension.js 會在下列位置輸出機器可讀的中繼資料：

- `dist/extension-js/<browser>/ready.json`
- `dist/extension-js/<browser>/events.ndjson`（newline-delimited JSON）

對自動化（Playwright、持續整合（CI）、AI 代理）而言，請優先使用這些檔案而非解析終端機記錄。

將 `ready.json` 視為就緒契約：

- 啟動中時為 `status: "starting"`
- 執行階段就緒時為 `status: "ready"`
- 啟動／編譯失敗時為 `status: "error"`
- `runId` 唯一識別一個執行階段工作階段
- `startedAt` 標示執行階段工作階段的開始時間戳記

### `--no-browser` 與就緒同步

`--no-browser` 僅停用瀏覽器啟動。它並不會等到編譯完成後再阻塞外部執行器。

對於 Playwright／CI／AI 工作流程：

1. 將 `extension dev --no-browser` 作為長時間執行的程序執行。
2. 將 `extension dev --wait --browser=<browser>` 作為就緒閘道執行。
3. 僅在 `status: "ready"` 之後啟動外部瀏覽器自動化。

`--wait` 是針對第二個程序（或 CI 步驟），並會在 `error`／逾時時以非零結束。
當 `--wait` 偵測到死掉程序留下的過期 `ready.json`（`pid` 已不存在）時，它會持續等待活躍的產生者。
若同一個指令同時傳入 `--wait` 與 `--no-browser`，`--wait` 優先。指令會以僅等待模式執行。

### `--no-reload` 取得乾淨的 dev bundle

`--no-reload` 會略過內容指令稿重新注入的封裝以及重建時的重新載入派送。dev 的 `dist` 會更接近生產 bundle，且檔案變更時不會干擾已開啟的分頁。請自行重新載入擴充功能或頁面以套用變更。

`--no-reload` 僅 `extension dev` 支援。傳遞給 `start`、`preview` 或 `build` 會以錯誤結束。內部會設定 `EXTENSION_NO_RELOAD=true`，讓 develop 程序能從 CLI argv 之外讀取它。

## 原始碼檢視工作流程

source 選項僅與 `dev` 搭配使用。原始碼檢視會在瀏覽器中開啟目標 URL，並印出在你的內容指令稿執行**之後**的完整即時 HTML。可藉此驗證你的擴充功能實際做了什麼變更。

- `--source [url]` 檢視頁面（預設為 `--starting-url` 或 `https://example.com`）
- `--watch-source` / `--no-watch-source` 在每次重建時重印 HTML

`start` 與 `preview` 不支援原始碼檢視旗標。

### Source 旗標

| 旗標                                            | 作用                                       | 預設行為                            |
| ----------------------------------------------- | ------------------------------------------ | ----------------------------------- |
| `--source [url]`                                | 啟用原始碼檢視流程。                       | 停用                                |
| `--watch-source [boolean]`                      | 在監看更新時重印原始碼輸出。               | 啟用 `--source` 時為 `true`         |
| `--source-format <pretty\|json\|ndjson>`        | 原始碼檢視的輸出格式。                     | 回退到記錄格式或 `json`             |
| `--source-summary [boolean]`                    | 輸出精簡摘要。                             | 停用                                |
| `--source-meta [boolean]`                       | 包含頁面中繼資料。                         | 啟用 source 時啟用                  |
| `--source-probe <selectors>`                    | 以逗號分隔的 CSS 選擇器進行探測。          | 未設定                              |
| `--source-tree <off\|root-only>`                | 包含精簡的擴充功能樹狀詳情。               | 未設定                              |
| `--source-console [boolean]`                    | 包含主控台摘要。                           | 停用                                |
| `--source-dom [boolean]`                        | 包含 DOM 快照／差異。                      | 啟用 watch-source 時啟用            |
| `--source-max-bytes <bytes>`                    | 限制輸出 payload 大小。                    | 未設定                              |
| `--source-redact <off\|safe\|strict>`           | 原始碼輸出的去識別策略。                   | 依格式而定                          |
| `--source-include-shadow <off\|open-only\|all>` | 控制 Shadow DOM 的包含方式。               | 啟用 source 時為 `open-only`        |
| `--source-diff [boolean]`                       | 在監看更新時包含 diff 中繼資料。           | 啟用 watch-source 時啟用            |

## Monorepo 與工作區根目錄

你可以將 `dev`（與 `build`）指向 **monorepo 根目錄**，而非擴充功能套件本身。Extension.js 會偵測工作區根目錄並自動解析其中的擴充功能套件：

```bash
extension dev .            # 從 monorepo 根目錄執行
```

當僅找到一個擴充功能套件時，Extension.js 會解析它並輸出：

```text
Workspace root detected — resolved extension package: packages/my-extension
```

當存在多個候選時，會列出它們，讓你指向想要的那個：

```bash
extension dev packages/my-extension
```

## 最佳實務

- **瀏覽器相容性：** 在不同瀏覽器中測試你的擴充功能，確保在每個目標都能運作。
- **使用 polyfill：** 若 Firefox 或 Gecko 系列瀏覽器也是目標，請使用 `--polyfill`。此旗標可在 Chromium 系列瀏覽器中啟用 `browser.*` API 相容性。
- **原始碼檢視工作流程：** 需要原始碼檢視選項時使用 `dev`（而非 `start`／`preview`）。
- **自動化可靠性：** 將 `dev` 視為監看模式的同伴（`--no-browser` + `dev --wait`）。將 `start` 視為生產的同伴（`--no-browser` + `start --wait`）。在指令稿與 CI 自動化中使用 `--wait-format=json`。
