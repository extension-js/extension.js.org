`dev` 用於日常的瀏覽器擴充功能開發,提供監看模式、瀏覽器啟動以及具情境感知的更新行為。

`dev` 會執行開發管線並監看你的專案檔案。它會依據變更內容套用相應的更新策略:熱模組替換 (HMR)、硬重新載入,或在必要時進行完整重啟。

## 何時使用 `dev`

- 即時建構功能並驗證變更。
- 在一個或多個瀏覽器目標中除錯擴充功能行為。
- 使用 `--source` 執行原始碼檢視工作流程。

使用 `build` 產生生產產物;使用 `start` 進行生產建置並啟動;使用 `preview` 只執行既有的建置輸出。

如果你的擴充功能位於 monorepo/子模組中,請查看 `extension.config.*` 如何載入環境變數檔(含工作區根目錄回退):[環境變數](/docs/features/environment-variables#how-it-works)。

## Dev 指令能力

| 能力               | 它能提供什麼                                          |
| ------------------ | ----------------------------------------------------- |
| 監看模式迭代       | 編碼時獲得緊湊的編輯 → 重建 → 驗證循環                |
| 瀏覽器目標控制     | 每條指令都能明確進行跨瀏覽器驗證                      |
| 設定檔感知的執行   | 依工作流程需要使用可靠的全新或持久化設定檔            |
| 原始碼檢視模式     | 透過原始碼相關選項取得結構化輸出與診斷                |

## 自動化中繼資料(建議用於指令稿/代理)

`dev` 執行時,Extension.js 會在以下位置輸出機器可讀的中繼資料:

- `dist/extension-js/<browser>/ready.json`
- `dist/extension-js/<browser>/events.ndjson`(以換行分隔的 JSON)

對於自動化情境(Playwright、持續整合 (CI)、AI 代理),建議優先使用這些檔案,而不是剖析終端記錄。

把 `ready.json` 視為就緒契約:

- 啟動過程中為 `status: "starting"`
- 執行階段就緒時為 `status: "ready"`
- 啟動/編譯失敗時為 `status: "error"`
- `runId` 用以唯一識別一次執行階段工作階段
- `startedAt` 紀錄執行階段工作階段的開始時間戳記

### `--no-browser` 與就緒同步

`--no-browser` 只會停用瀏覽器啟動,不會在編譯完成前阻擋外部執行器。

對於 Playwright/CI/AI 工作流程:

1. 將 `extension dev --no-browser` 作為長時間執行的處理程序。
2. 將 `extension dev --wait --browser=<browser>` 作為就緒閘道。
3. 僅在出現 `status: "ready"` 之後再啟動外部瀏覽器自動化。

`--wait` 是針對第二個處理程序(或 CI 步驟),在 `error`/逾時情況下以非零狀態結束。
當 `--wait` 偵測到來自已結束處理程序的過期 `ready.json`(`pid` 已不存在)時,會繼續等待存活的產生者。
若在同一條指令中同時傳入 `--wait` 與 `--no-browser`,`--wait` 優先生效,指令會以僅等待模式執行。

### 使用 `--no-reload` 取得乾淨的開發打包

`--no-reload` 會略過內容腳本的重新注入包裝,以及重建後的重新載入分發。開發 `dist` 會更接近生產打包,且檔案變更也不會打擾已開啟的分頁。請自行重新載入擴充功能或頁面以套用變更。

`--no-reload` 僅在 `extension dev` 中支援。將其傳給 `start`、`preview` 或 `build` 都會以錯誤結束。內部會設定 `EXTENSION_NO_RELOAD=true`,以便開發處理程序可以從 CLI argv 之外讀取它。

## 原始碼檢視工作流程

原始碼相關選項只能與 `dev` 一起使用。原始碼檢視會在瀏覽器中開啟一個目標 URL,並在你的內容腳本執行 **之後** 列印完整、即時的 HTML。用它來驗證你的擴充功能實際變更了哪些內容。

- `--source [url]` 用以檢視頁面(預設為 `--starting-url` 或 `https://example.com`)
- `--watch-source` / `--no-watch-source` 用以在每次重建後重新列印 HTML

`start` 與 `preview` 不支援原始碼檢視相關選項。

### 原始碼選項

| 選項                                            | 作用                                       | 預設行為                              |
| ----------------------------------------------- | ------------------------------------------ | ------------------------------------- |
| `--source [url]`                                | 啟用原始碼檢視流程。                       | 停用                                  |
| `--watch-source [boolean]`                      | 在被監看的更新發生時重新列印原始碼輸出。   | 啟用 `--source` 時為 `true`           |
| `--source-format <pretty\|json\|ndjson>`        | 原始碼檢視的輸出格式。                     | 回退到記錄格式或 `json`               |
| `--source-summary [boolean]`                    | 輸出緊湊的摘要。                           | 停用                                  |
| `--source-meta [boolean]`                       | 包含頁面中繼資料。                         | 啟用原始碼時啟用                      |
| `--source-probe <selectors>`                    | 以逗號分隔的 CSS 選擇器進行探測。          | 未設定                                |
| `--source-tree <off\|root-only>`                | 包含緊湊的擴充功能樹狀詳細資訊。           | 未設定                                |
| `--source-console [boolean]`                    | 包含主控台摘要。                           | 停用                                  |
| `--source-dom [boolean]`                        | 包含 DOM 快照/差異。                       | 啟用 watch-source 時啟用              |
| `--source-max-bytes <bytes>`                    | 限制輸出酬載大小。                         | 未設定                                |
| `--source-redact <off\|safe\|strict>`           | 原始碼輸出的去識別化策略。                 | 取決於格式                            |
| `--source-include-shadow <off\|open-only\|all>` | 控制是否包含 Shadow DOM。                  | 啟用原始碼時為 `open-only`            |
| `--source-diff [boolean]`                       | 在監看更新時包含差異中繼資料。             | 啟用 watch-source 時啟用              |

## Monorepo 與工作區根目錄

你可以將 `dev`(以及 `build`)指向 **monorepo 的根目錄**,而不是擴充功能套件本身。Extension.js 會偵測工作區根目錄並自動解析其中的擴充功能套件:

```bash
extension dev .            # run from the monorepo root
```

當剛好找到一個擴充功能套件時,Extension.js 會解析它並列印:

```text
Workspace root detected — resolved extension package: packages/my-extension
```

當有多個候選時,會列出讓你指明要使用哪一個:

```bash
extension dev packages/my-extension
```

## 最佳實務

- **瀏覽器相容性:** 在不同瀏覽器中測試擴充功能,以確認它在每個目標上都能運作。
- **Polyfill:** 如果目標包含 Firefox 或基於 Gecko 的瀏覽器,請使用 `--polyfill`。這個選項會在基於 Chromium 的瀏覽器中啟用 `browser.*` API 相容。
- **原始碼檢視工作流程:** 需要原始碼檢視相關選項時,請使用 `dev`(而非 `start`/`preview`)。
- **自動化可靠性:** 將 `dev` 視為監看模式夥伴(`--no-browser` + `dev --wait`),將 `start` 視為生產夥伴(`--no-browser` + `start --wait`)。在指令稿與 CI 自動化中使用 `--wait-format=json`。
