當你想用一行指令完成生產建置並立刻啟動瀏覽器時,使用 `start`。

`start` 指令會先執行生產建置,然後使用與 `preview` 指令相同的流程啟動已建置好的擴充功能。

## 何時使用 `start`

- 編譯完成後立刻手動驗證生產行為。
- 重現監看模式與生產輸出之間的 runtime 差異。
- 在本機進行類似生產的檢查,而不需另外執行 `build` 再 `preview`。

## Start 指令能做什麼

| 功能                          | 你會得到什麼                                              |
| ----------------------------- | --------------------------------------------------------- |
| Build + 啟動工作流            | 一個步驟內執行生產編譯與瀏覽器啟動                        |
| 目標選擇                      | 直接在所選瀏覽器或引擎目標中啟動                          |
| Runner 控制                   | 只需要建置驗證時可跳過瀏覽器啟動                          |
| 類似生產的驗證                | 檢查真實編譯輸出,而非監看模式狀態                        |

## 與其他指令的差別

- `dev`:dev server + 熱模組替換(HMR)/監看模式
- `build`:僅生產建置
- `preview`:啟動既有的已建置擴充功能,不進行建置
- `start`:依序執行 `build` + `preview`

## 自動化 metadata

`start` 會把就緒 metadata 寫入:

- `dist/extension-js/<browser>/ready.json`

在使用 `--no-browser` 時對自動化很有幫助:

- 在啟動外部 runner 前等待 `status: "ready"`。
- 把 `status: "error"` 當作確定性的失敗訊號來處理。
- 用 `runId` 與 `startedAt` 關聯特定 runtime session。

### `--no-browser` 與就緒同步

`--no-browser` 只會停用瀏覽器啟動,並不會在生產建置完成前阻擋外部 runner。

對於生產導向的 Playwright、持續整合 (CI) 與 AI 工作流:

1. 把 `extension start --no-browser` 當作 producer 程序。
2. 把 `extension start --wait --browser=<browser>` 當作就緒門檻。
3. 只在 `status: "ready"` 之後啟動外部瀏覽器自動化。

`--wait` 在 `error`/逾時時以非零代碼退出,並忽略已死程序留下的過期契約(`pid` 不再存活)。
如果你在同一次呼叫中同時傳入 `--wait` 與 `--no-browser`,`--wait` 優先。指令會以僅等待模式執行。

## 行為備註

- `start` 不會執行 dev server,也不提供熱模組替換(HMR)或監看模式。
- `start` 是生產模式導向;迭代式本機開發請使用 `dev`。
- `start` 不支援原始碼檢查選項。要使用該工作流請改用 `dev --source ...`。
- 機器消費者請解析 `dist/extension-js/<browser>/ready.json`,而非終端機文字。
