你在一個 CRXJS 擴充功能專案上執行 `vite build` 並得到:

```text
[crx:manifest-post] Content script fileName is undefined: "src/content.ts"
```

建構失敗、manifest 從未被寫入,而錯誤中的路徑確實是你專案中存在的檔案。本頁說明為什麼會發生、有哪些可讓 CRXJS 繼續運作的解決方法,以及結構性的修復。

## 這個錯誤代表什麼

`@crxjs/vite-plugin` 會分兩階段建構你的擴充功能。在 `generateBundle` 期間,`crx:manifest-post` 步驟會走訪你 manifest 中每個 `content_scripts` 條目,並向打包工具詢問為每個原始檔輸出的 chunk 的檔名。當打包工具沒有為該原始路徑回傳任何 chunk 時,查表會回傳 `undefined`,外掛便會丟擲錯誤。

因此這個錯誤的意義不是「你的檔案不存在」。它代表的是外掛與打包工具對於內容腳本所輸出的 chunk 的命名或鍵不一致。

## 為何 Vite 8 會觸發它

Vite 8 將預設打包工具從 Rollup 換成了 [Rolldown](https://vite.dev/blog/announcing-vite8)。CRXJS 是針對 Rollup 的 chunk 輸出模型撰寫的,其內容腳本解析仰賴在 Rolldown 下行為不同的內部細節。在特定設定下,這個錯誤其實先前就存在(在 [crxjs/chrome-extension-tools#883](https://github.com/crxjs/chrome-extension-tools/issues/883) 中追蹤),但 Vite 8 的預設值讓它從邊緣案例變成常見案例。

## 可讓 CRXJS 繼續運作的解決方法

請依序嘗試:

1. **將 Vite 鎖定在 7。** Vite 7 仍使用 Rollup,CRXJS 能理解:

   ```bash
   npm install vite@^7 --save-dev --save-exact
   ```

   這今天就能解除阻塞,代價是落後在 Vite 上。

2. **將 `@crxjs/vite-plugin` 更新到最新版。** 該專案一直在逐步加入 Rolldown 相容性。在鎖定任何版本前,請先查閱 [release 頁面](https://github.com/crxjs/chrome-extension-tools/releases) 與上述 issue 了解目前的支援狀態。

3. **檢查 manifest 如何參照腳本。** 查表是以路徑為鍵的。讓 `content_scripts.js` 條目與你專案根的路徑風格一致(相對路徑、不要有前導斜線),並確認同一個檔案沒有在其他地方以 `?script` 字尾被動態匯入。

如果以上對你的設定都不奏效,剩下的選項就是完全不再依賴 Vite 外掛層。

## 結構性的修復

根本問題在於架構。CRXJS 在通用打包工具的外掛 API 之上重建擴充功能語意(manifest 連接、內容腳本輸出、HMR),所以每次大版的打包工具變更都可能破壞它。Extension.js 是一個 [瀏覽器擴充功能框架](/docs/compare/extension-js-vs-wxt),其中 manifest 是唯一事實來源,打包工具是你永遠不必接線的內部細節:

```bash
npx extension@latest dev
```

- 你的 `manifest.json` 直接宣告 `content/index.ts`。沒有外掛去解析 chunk 名稱;框架會編譯 manifest 所指的任何東西。
- 沒有 `vite.config.ts`、沒有 `manifest.config.ts`,也沒有需要對齊的外掛版本矩陣。
- 同一份原始碼產出 Chrome、Firefox 與 Edge,每個介面都有對應的重新載入行為。

你的 React、Vue 或 Svelte 元件以及 `chrome.*` 呼叫可以原封不動延續。完整步驟約十分鐘:[從 CRXJS 遷移到 Extension.js](/docs/migrate/from-crxjs)。
