[CRXJS](https://crxjs.dev) 是一個給 Chrome 擴充功能使用的 Vite 外掛。如果你已經超出僅支援 Chrome 的範圍、想要一流的 Firefox 輸出，或遇到了重新載入迴圈的奇怪問題，本指南會在不重寫 UI 程式碼的前提下，將一個典型的 CRXJS 專案遷移到 Extension.js。

<Note>
  你是不是因為 `vite build` 在 Vite 8 上失敗並顯示 `[crx:manifest-post] Content
  script fileName is undefined` 才來到這裡？在遷移之前，請參見[專屬修正頁面](/docs/migrate/crxjs-content-script-filename-undefined)
  了解解決方法。
</Note>

## 哪些會變、哪些不變

**不變：** 你的 React/Vue/Svelte 元件、Tailwind 設定、測試以及 `chrome.*` API 呼叫。

**會變：**

- `vite.config.ts` + `@crxjs/vite-plugin` 會變成 `extension.config.js`（或完全不需要設定檔）。
- `manifest.config.ts`（TypeScript 模組）會變成單純的 `manifest.json`，並使用[瀏覽器前綴鍵](/docs/features/browser-specific-fields)。
- `vite` / `vite build` 腳本會變成 `extension dev` / `extension build`。
- Extension.js dev 取代 Vite 開發伺服器，內建瀏覽器啟動、設定檔管理與各目標重新載入。

## 步驟 1：安裝 Extension.js

```bash
npm install extension@latest --save-dev
npm uninstall @crxjs/vite-plugin vite
```

如果你的專案也將 Vite 用於非擴充功能介面（例如行銷網站），請將 Vite 保留在該工作區的範圍內。

## 步驟 2：轉換 manifest

CRXJS 使用 TypeScript 模組：

```ts manifest.config.ts
import { defineManifest } from "@crxjs/vite-plugin";
import pkg from "./package.json";

export default defineManifest({
  manifest_version: 3,
  name: pkg.name,
  version: pkg.version,
  action: { default_popup: "src/popup/index.html" },
  background: { service_worker: "src/background/index.ts" },
  content_scripts: [{ matches: ["<all_urls>"], js: ["src/content/index.ts"] }],
});
```

在 Extension.js 中，請撰寫真正的 `manifest.json` 並參照真實檔案：

```json manifest.json
{
  "manifest_version": 3,
  "name": "my-extension",
  "version": "1.0.0",
  "action": { "default_popup": "popup/index.html" },
  "background": { "service_worker": "background/index.ts" },
  "content_scripts": [{ "matches": ["<all_urls>"], "js": ["content/index.ts"] }]
}
```

注意事項：

- 在 manifest 中副檔名維持 `.ts`/`.tsx`。Extension.js 會在建置時編譯它們。
- 如果你打平了目錄結構，請去掉 `src/` 前綴。Extension.js 會依 manifest 宣告的路徑運作。
- 對於瀏覽器特定值，請使用[前綴鍵](/docs/features/browser-specific-fields)（`firefox:browser_specific_settings`），而不是每個建置一份 manifest。

## 步驟 3：更新 package.json 腳本

把：

```json
{
  "scripts": {
    "dev": "vite",
    "build": "vite build"
  }
}
```

換成：

```json
{
  "scripts": {
    "dev": "extension dev",
    "build": "extension build",
    "start": "extension start"
  }
}
```

完整指令集請參見[指令參考](/docs/commands/index)。

## 步驟 4：移除 vite.config.ts

如果你的 `vite.config.ts` 只是為了串接 CRXJS，請刪除它。如果它含有其他外掛，請將對應的功能搬到 [`extension.config.js`](/docs/features/extension-configuration)，或對於進階打包工具客製，搬到 [Rspack 設定](/docs/features/rspack-configuration)。

## 步驟 5：處理 HMR 差異

CRXJS 的 HMR 透過 Vite 的 WebSocket 推送更新。Extension.js 使用[重新載入與 HMR](/docs/features/reload-and-hmr)中所述的不同模型：

- Popup、options、devtools 頁面：HMR。
- Content script：目標式重新載入。
- 背景 service worker：完整重啟。

如果你的程式碼曾依賴 Vite 的 `import.meta.hot` 來處理 content script 邏輯，請把這些分支替換為一般的模組程式碼。Extension.js 會在你的原始碼之外處理重新載入的協調。

## 步驟 6：跨瀏覽器輸出

CRXJS 以 Chromium 為目標。要同時開始輸出 Firefox：

```bash
extension build --browser=chrome,firefox --zip
```

你會得到 `dist/chrome` 與 `dist/firefox`，內含為對應瀏覽器調整的 manifest，以及可上架 Chrome Web Store 與 addons.mozilla.org 的 `.zip` 檔。參見[跨瀏覽器相容性](/docs/features/cross-browser-compatibility)。

## 步驟 7：驗證

```bash
extension dev --browser=chrome
```

驗證 popup、options、content script 與背景行為。接著驗證 Firefox：

```bash
extension dev --browser=firefox
```

如果你的程式碼呼叫 `chrome.*`，又希望同一份原始碼能在 Firefox 上執行而不需修改，請使用 `--polyfill`。

## 常見陷阱

- **Service worker 的 `import` 陳述句：** 當你的背景進入點使用 ESM 語法時，Extension.js 會自動設定 `type: "module"`。CRXJS 也是如此。不需任何操作。
- **`web_accessible_resources` 的型別：** Manifest V3 採用 `[{resources, matches}]` 區塊。兩個框架都會輸出正確結構；直接複製既有條目即可。
- **帶 hash 的資產路徑：** Extension.js 在 [`dist/<browser>`](/docs/features/path-resolution) 下使用穩定路徑。如果你的程式碼寫死了 Vite 風格的帶 hash 檔名，請改用相對於 manifest 的路徑。
