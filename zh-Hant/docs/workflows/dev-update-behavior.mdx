知道哪些變更會立即套用、哪些需要重新啟動,可以讓迭代保持快速。Extension.js 會針對每個檔案變更挑選最輕量的安全更新模式。

## 更新模式

| 模式                        | 發生什麼                                                       | 典型影響                                                  |
| --------------------------- | -------------------------------------------------------------- | --------------------------------------------------------- |
| 熱模組替換(HMR)             | 模組/樣式更新就地套用                                          | 最快的迭代速度,不需重新啟動整個擴充功能                  |
| `Hard reload`               | 目標瀏覽器重新載入擴充功能的執行階段                            | 擴充功能 context 重新啟動,但 dev 程序持續執行            |
| `Restart required`          | 編譯期診斷會要求你重新執行 `extension dev`                      | 適用於結構性入口/清單變更                                  |

## 常見範例

| 變更                                                | 預期行為                                  |
| --------------------------------------------------- | ----------------------------------------- |
| 修改 content script/元件程式碼                       | HMR/remount 路徑(在支援時)                |
| 修改 content script/頁面使用的 style 匯入            | 走 HMR 風格的更新路徑(視 context 而定)    |
| 修改 `manifest.json` 中的值(非結構性)                | Hard reload 流程                          |
| 新增/移除 manifest 中的入口參照                      | 需要重新啟動                              |
| 新增/移除 `pages/` 或 `scripts/` 入口                | 需要重新啟動                              |
| 修改 `_locales/*/messages.json`                     | Hard reload 流程                          |

## 需要重新啟動時該怎麼做

當 Extension.js 回報 `Restart required` 時,請先停止並重新執行 `extension dev`,再繼續其他工作。

典型的觸發條件:

- 新增或移除 manifest 中的入口參照
- 新增或移除位於 `pages/` 或 `scripts/` 中、且被當作擴充功能入口的檔案
- HTML 入口中結構性的 script/style 入口變更

## 實務指引

- 在開發過程中,優先修改既有的模組/資源,以取得最快回饋。
- 結構性的 manifest 變更(入口清單修改)集中處理,只需重新啟動一次。
- 把「需要重新啟動」的診斷視為權威指示;不重新啟動就繼續可能讓執行階段處於過時狀態。
