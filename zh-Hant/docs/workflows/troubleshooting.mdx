透過先檢查最常見的失敗原因來快速解除本機開發的阻塞。在深入除錯之前，請將本頁面作為實用的分流路徑。

## 疑難排解能力

| 領域                  | 它能協助你驗證什麼                                       |
| --------------------- | -------------------------------------------------------- |
| 重新啟動診斷          | 結構性變更是否需要重新啟動                               |
| 相依套件設定          | 目前技術堆疊是否缺少選用工具                             |
| 瀏覽器啟動            | 目標/瀏覽器二進位設定是否有效                            |
| 建置/預覽流程         | 預覽所需的輸出是否存在並與目標相符                       |
| Manifest 路徑         | 參考的檔案是否能正確解析                                 |

## 快速分流流程

1. 用單一目標重現（`--browser=chromium`）。
2. 移除自訂 profile 與二進位旗標。
3. 確認問題是出現在 `dev`、`build` 或兩者皆有。
4. 一次只修一類錯誤（重新啟動、相依套件、路徑或啟動）。

## 快速診斷決策樹

| 如果這個失敗了...        | 先檢查                                                | 下一步動作                                                                                  |
| ------------------------ | ----------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| 擴充功能無法載入         | 建置與 manifest 錯誤                                  | 驗證參考的檔案並重新執行 `dev`                                                              |
| 瀏覽器無法啟動           | `--browser` + 二進位路徑                              | 移除自訂二進位/profile 旗標並重試                                                           |
| 變更未更新               | 更新類型（熱模組替換 vs 需要重新啟動）                | 查看 [Dev 更新行為](/docs/workflows/dev-update-behavior)，必要時重新啟動                    |
| `preview` 失敗           | 既有的 `dist/<browser>` 輸出                          | 先執行 `extension build --browser=<target>`                                                 |
| 執行階段仍有問題         | 最小可重現設定                                        | 縮小範圍並擷取精確的命令 + 錯誤                                                             |

## 常見問題

### 需要重新啟動的診斷

如果你看到需要重新啟動的錯誤，請停止並重新執行 `extension dev`。

典型觸發條件：

- Manifest 進入點清單變更。
- 在 `pages/` 或 `scripts/` 中新增/移除檔案。
- HTML 中結構性的 script/style 進入點變更。

完整對照表請參見 [Dev 更新行為](/docs/workflows/dev-update-behavior)。

### 缺少選用相依套件

CLI 會按需安裝某些整合（例如 Vue/Preact 重新整理工具、樣式前置處理器）。

如果 CLI 提示安裝選用相依套件：

1. 允許安裝。
2. 重新啟動 `extension dev`。
3. 相依套件安裝完成後再執行一次。

### 瀏覽器二進位問題

如果目標瀏覽器啟動失敗：

- 驗證自訂二進位路徑（`--chromium-binary` / `--gecko-binary`）。
- 確認瀏覽器目標與提供的二進位相容。
- 退回預設的 `chromium` 目標以隔離設定問題。

### 建置可執行但預覽失敗

`preview` 預期已存在的建置輸出。

- 執行 `extension build --browser=<target>`。
- 然後執行 `extension preview --browser=<target>`。

### Manifest 參考錯誤

如果建置回報缺少 manifest 檔案：

- 驗證路徑是否相對於 manifest 位置。
- 驗證開頭 `/` 的使用（public 根語義）。
- 確認檔案確實存在於原始碼/public 位置。

### Docker、開發容器與 Codespaces

當在容器內執行時，dev 伺服器預設繫結到 `127.0.0.1`，因此主機無法存取。

- 傳入 `--host 0.0.0.0` 以繫結到所有介面。
- 如果 `8080` 被占用，請使用 `--port 0` 讓作業系統指派可用連接埠。
- 如果在持續整合（CI）容器中瀏覽器連線緩慢或不穩定，請使用[瀏覽器傳輸調校變數](/docs/features/environment-variables#browser-transport-tuning-variables)來提高連線逾時。

```bash
extension dev --host 0.0.0.0 --port 0
```

### `scripts/` 資料夾中的 Node.js 指令稿

如果建置失敗並回報 `scripts/ is a reserved folder in Extension.js`，代表你在 `scripts/` 特殊資料夾中放了一個 Node.js 檔案。Extension.js 會用瀏覽器內容指令稿執行階段包裹 `scripts/` 中的每個檔案。僅用於 Node.js 的檔案在此上下文中會失敗。

請將該檔案移動到專案根目錄下的其他資料夾（例如 `bin/`、`tools/` 或 `ci-scripts/`）。詳見[特殊資料夾](/docs/features/special-folders#nodejs-scripts-are-not-allowed-in-scripts)。

### 修復後仍受阻

如果問題持續存在：

- 清除暫時的假設並從乾淨的終端機工作階段重新重現。
- 縮小到仍能重現失敗的最小 manifest/進入點案例。
- 擷取精確的命令 + 錯誤輸出以便回報 issue。

## 除錯檢查清單

- 先用單一瀏覽器目標重現（`--browser=chromium`）。
- 在不帶自訂 profile/二進位旗標的情況下重現。
- 檢查問題是否同時出現在 `dev` 與 `build` 中。
- 在修改 manifest + 進入點檔案時一次只做一個變更。
