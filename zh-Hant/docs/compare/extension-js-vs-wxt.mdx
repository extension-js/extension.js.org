Extension.js 與 [WXT](https://wxt.dev) 都是用於建構跨瀏覽器擴充功能的現代框架。它們在使命上有重疊,但在理念上有所不同。本頁是一個事實性比較,讓你可以依據專案需求選擇,而不是依賴行銷文案。

## 一句話總結:你該選哪個?

<CardGroup cols={2}>
  <Card title="如果符合以下情況,選擇 Extension.js……">
    - 你希望 **以 `manifest.json` 作為唯一事實來源**,而不是由設定產生。
    - 你喜歡貼近原生 WebExtension 模型(真實檔案、透明輸出)。
    - 你希望用一行指令就能開發遠端範例:`extension dev <github-url>`。
    - 你想要 Rspack 以 Rust 等級速度的建構,以及一流的 [AI/MCP 工具](/docs/ai-access)。
  </Card>
  <Card title="如果符合以下情況,選擇 WXT……">
    - 你偏好 **檔案系統慣例**(`entrypoints/`)而不是顯式的 manifest。
    - 你需要 **Manifest V2** 作為主要目標。
    - 你希望預設在所有地方自動 polyfill `browser.*` 命名空間。
  </Card>
</CardGroup>

本頁其餘部分會逐個面向支撐上述摘要。

## 概覽

| 面向                 | Extension.js                                                                | WXT                                                                  |
| -------------------- | --------------------------------------------------------------------------- | -------------------------------------------------------------------- |
| 打包工具             | [Rspack](/docs/features/rspack-configuration)(以 Rust 為基礎)              | Vite(正在遷移至 Rolldown)                                          |
| Manifest             | 一個 `manifest.json`,瀏覽器前綴的鍵會在編譯時被過濾                         | `wxt.config.ts` 於每次建構時產生 `manifest.json`                     |
| 進入點               | 由 `manifest.json` 參照的檔案                                               | `entrypoints/` 下的檔案系統慣例                                      |
| 瀏覽器目標           | Chrome、Edge、Firefox、Chromium、Gecko、自訂執行檔                          | Chrome、Edge、Firefox、Safari(社群)、自訂執行檔                   |
| Manifest V3          | 預設                                                                        | 預設                                                                 |
| Manifest V2          | 不支援作為主要目標                                                          | 透過 `manifestVersion: 2` 支援                                       |
| 重新載入模型         | popup/options/devtools 的 HMR,內容腳本與 SW 的針對性重新載入               | popup/options/devtools 的 HMR,內容腳本的針對性重新載入              |
| `browser.*` polyfill | `--polyfill` 旗標                                                           | 自動 polyfill 的 `browser` 命名空間                                  |
| 範本                 | React、Preact、Vue、Svelte、TypeScript、JavaScript、init                    | React、Vue、Svelte、Solid、Preact、vanilla                           |
| TypeScript           | 一流支援                                                                    | 一流支援                                                             |
| AI 接觸面            | 託管 MCP 伺服器 + `llms.txt`([詳情](/docs/ai-access))                      | 提供 `llms.txt`                                                      |

## 思維模型

**Extension.js 貼近平台。** 你撰寫 `manifest.json` 並參照真實檔案。CLI 會依瀏覽器編譯、過濾並打包。如果你已經理解瀏覽器擴充功能的結構,框架不會擋路。

**WXT 抽象化了平台。** 進入點由目錄佈局推斷(`entrypoints/popup/`、`entrypoints/content.ts`),manifest 則由你的設定與原始碼產生。如果你偏好「慣例優於設定」,WXT 在你每寫一個檔案時會為你做更多。

兩種方式都不是普遍更好。選擇取決於你想要 **看見** 你的 manifest,還是 **宣告** 你的 manifest。

## CLI 介面

### Extension.js

```bash
extension dev --browser=chrome,firefox
extension build --browser=chrome,firefox --zip
extension dev https://github.com/user/repo/tree/main/path
```

引數也可以是遠端 GitHub URL 或 ZIP 壓縮檔,對於不需克隆就能啟動範例非常實用。參見 [立即開始](/docs/getting-started/immediately)。

### WXT

```bash
wxt
wxt build
wxt build -b firefox
wxt zip
```

WXT 將 dev/build/zip 拆成獨立指令。Extension.js 則把打包合併進 `build --zip`。

## 跨瀏覽器策略

兩個框架都以同一份程式碼發佈到多個瀏覽器,但機制不同:

- **Extension.js** 在同一份 `manifest.json` 內使用 [瀏覽器前綴的 manifest 欄位](/docs/features/browser-specific-fields)(`chrome:`、`firefox:`、`gecko:` 等)。沒有前綴的鍵在所有地方生效;有前綴的鍵只會進入對應的建構。
- **WXT** 由 `wxt.config.ts` 以及各目標覆寫計算出 manifest。瀏覽器差異存在於 TypeScript 設定中,而不是 manifest 本身。

如果你的團隊裡有閱讀 `manifest.json` 的設計師或 PM,前綴鍵能讓該檔案保持為唯一事實來源。如果你的團隊希望擴充功能設定能與其他 TypeScript 設定放在一起,WXT 的做法讀起來更自然。

## 遷移路徑

如果你已經在用 WXT,並希望評估 Extension.js,典型的遷移會涉及三件事:

1. 將 `entrypoints/` 內容移回手寫 `manifest.json` 所參照的扁平檔案。
2. 用 [`extension.config.js`](/docs/features/extension-configuration) 取代 `wxt.config.ts` 作為建構預設值。
3. 將 `wxt` / `wxt build` 腳本改為 `extension dev` / `extension build`。

大多數 React、Vue 與 Svelte 原始檔可以無改動直接複製。

## 何時選擇 Extension.js

- 你希望 manifest 作為唯一事實來源,而不是被產生的。
- 你希望用一個 CLI 引數就能開發遠端範例(`extension dev <github-url>`)。
- 你希望文件有一流的託管 MCP 用於 AI 工具。
- 你希望 Rspack 為大型擴充功能提供 Rust 級的編譯速度。

## 何時選擇 WXT

- 你偏好檔案系統慣例,而不是顯式的 manifest。
- 你需要 Manifest V2 作為主要目標。
- 你希望預設在所有地方自動 polyfill `browser.*`。
