瀏覽器擴充功能框架處理瀏覽器本身不做的工作。它為每一個擴充功能介面編譯 TypeScript 與現代 JavaScript、將 manifest 連接到實際輸出檔、在開發期間重新載入 service worker 與內容腳本,並依瀏覽器輸出建構成品。本頁比較目前活躍的選項,讓你能在掌握完整資訊後做選擇。

## 這些框架

| 面向            | Extension.js                          | WXT                          | CRXJS                          | Plasmo                        |
| --------------- | ------------------------------------- | ---------------------------- | ------------------------------ | ----------------------------- |
| 模型            | `manifest.json` 作為唯一事實來源      | 檔案慣例 + 設定              | 讀取 manifest 的 Vite 外掛    | 檔案慣例                      |
| 打包工具        | Rspack(內部,零設定)                | Vite                         | Vite(外掛層)                | Parcel                        |
| 跨瀏覽器        | 從同一份原始碼輸出 Chrome、Edge、Firefox | Chrome、Firefox,支援 MV2  | 專注於 Chromium                | 目標為 Chrome、Firefox        |
| 設定            | 無須設定                              | `wxt.config.ts`              | `vite.config.ts` + 外掛       | `package.json` 中的 manifest 欄位 |
| 與打包工具的耦合 | 內部細節                              | 跟隨 Vite 大版本             | 打包工具變更時容易壞掉         | 與 Parcel 綁定                |

有兩項結構性差異比任何功能列都更重要:

- **manifest 放在哪裡。** Extension.js 讀取你的 `manifest.json`,並編譯它所宣告的一切。WXT 與 Plasmo 從檔案慣例與設定產生 manifest。CRXJS 也會讀取 manifest,但透過 Vite 的外掛 API 進行解析,因此打包工具升級可能會破壞它(參見 [Vite 8 fileName 錯誤](/docs/migrate/crxjs-content-script-filename-undefined))。
- **誰擁有打包工具。** 包裹通用打包工具的框架會繼承該打包工具的破壞性變更。Extension.js 把打包工具視為內部細節:你永遠不需要設定它,升級是框架的問題,而不是你的。

## 不必承諾即可試用

只需一行指令,就能執行任何擴充功能範本,或 GitHub 上任何擴充功能儲存庫:

```bash
npx extension@latest dev
```

你的元件、`chrome.*` 呼叫與樣式可以從上述任何框架延續過來;[遷移指南](/docs/migrate/from-crxjs) 會涵蓋需要調整的連接部分。
