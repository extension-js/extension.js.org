    <AccordionGroup>
      <Accordion title="什麼是瀏覽器擴充功能？">
        瀏覽器擴充功能是為 Chrome、Firefox、Edge 或其他瀏覽器新增功能的軟體。擴充功能可以注入 UI、回應瀏覽器事件，並呼叫頁面本身無法使用的特權 API。完整入門請參見 [什麼是瀏覽器擴充功能？](/docs/concepts/what-is-a-browser-extension)。
      </Accordion>
      <Accordion title="Extension.js 是 Chrome 擴充功能框架嗎？">
        是。Extension.js 既是 Chrome 擴充功能框架，也是 Firefox 與 Edge 擴充功能框架，且皆來自同一個專案。透過 `--browser=chrome`、`--browser=firefox` 或 `--browser=edge` 選擇目標。參見 [可用瀏覽器](/docs/browsers/browsers-available)。
      </Accordion>
      <Accordion title="Extension.js 在開發期間如何處理模組熱替換？">
        執行 `npx extension dev`，CLI 會監看專案變更。對 popup、內容指令稿、service worker 與選項頁的編輯，會觸發最快且安全的更新路徑。CLI 在支援時採用模組熱替換（HMR），否則進行定向重新載入。它會自動解決常見痛點，例如過時的 service worker 與漏掉的內容指令稿更新。
      </Accordion>
      <Accordion title="我要如何為 Chrome、Edge 與 Firefox 建置同一個擴充功能？">
        Extension.js 將單一程式碼庫編譯為特定瀏覽器的輸出。一組指令即可為 Chrome、Edge、Firefox 以及其他 Chromium 系瀏覽器產出正確打包的建置。每個建置都會得到正確的 manifest 格式、路徑與平台調整。你的持續整合（CI）流水線只需在同一個建置步驟使用不同的瀏覽器旗標。這能減少各商店送審之間的落差。
      </Accordion>
      <Accordion title="我可以使用 React、Preact、Vue、Svelte 或 TypeScript 嗎？">
        可以。Extension.js 提供面向 React、Preact、Vue、Svelte、TypeScript 與原生 JavaScript 的生產級範本。你可以用與其他前端專案一樣的方式撰寫元件。框架選擇不會影響核心工作流。
      </Accordion>
      <Accordion title="Extension.js 支援 Manifest V3 嗎？">
        支援。工具鏈圍繞 Manifest V3 以及瀏覽器商店目前強制執行的限制設計：service worker、宣告式 API、更新後的權限與更嚴格的打包。Manifest V3（MV3）是預設選項，而非事後補丁。常見陷阱請參見 [Manifest V3 疑難排解](/docs/concepts/manifest-v3)。
      </Accordion>
      <Accordion title="如何排除 Manifest V3 的 service worker 與權限問題？">
        大多數 Manifest V3 疑難排解可歸納為幾類：service worker 註冊與 `type: "module"`、新的 `web_accessible_resources` 結構、`permissions` 與 `host_permissions` 的拆分，以及 Firefox 中 `declarative_net_request` 的差異。修復方式請參見 [Manifest V3 疑難排解](/docs/concepts/manifest-v3)。
      </Accordion>
      <Accordion title="如何從自訂的 webpack 或 Vite 設定遷移？">
        Extension.js 取代了手寫打包器設定中與擴充功能相關的「黏合層」。它負責 manifest 版本管理、多瀏覽器輸出、內容指令稿注入與重新載入編排。將原始檔與 manifest 移入 Extension.js 專案，CLI 會處理其餘工作。你的 React/Vue/Svelte/TS 程式碼無需重寫。
      </Accordion>
      <Accordion title="Extension.js 是否支援 monorepo 與環境變數？">
        Extension.js 支援 pnpm、npm 與 Yarn 的 workspaces。路徑解析與共享依賴可跨工作區邊界運作。Extension.js 依瀏覽器與建置模式注入環境變數，在 JS、JSON 與 HTML 中完成替換。開發環境的密鑰不會外洩到生產組合包中。
      </Accordion>
      <Accordion title="Extension.js 是免費的嗎？">
        是。Extension.js 在
        [MIT 授權](https://github.com/extension-js/extension.js/blob/main/LICENSE) 下開源。
        沒有付費方案、使用量上限或需強制開啟遙測的模式。本框架由贊助者與貢獻者資助。具體資料結構請查看
        [遙測契約](/docs/features/telemetry-and-privacy)。
      </Accordion>
      <Accordion title="Extension.js 支援 Safari 嗎？">
        目前還不支援。`--browser=safari` 目前不是 CLI 目標。你仍然可以用共享的 Web 程式碼撰寫 Safari 擴充功能，並在此工作流外加入 Safari 專屬的建置步驟。目前範圍請參見 [可用瀏覽器](/docs/browsers/browsers-available#safari-and-other-webkit-targets)。
      </Accordion>
      <Accordion title="Extension.js 會為我的組合包增加多少內容？">
        生產建置（`extension build`）只輸出擴充功能引用到的程式碼，外加一個小型的重新載入執行階段（在生產中會被移除）。開發建置則包含額外的 HMR 與重新載入黏合程式碼。輸出位於 `dist/<browser>` 下，方便你比較各瀏覽器之間的體積。內容指令稿組合包的體積接近於「你的原始碼 + 你選擇的 UI 框架」自身產出的體積。
      </Accordion>
      <Accordion title="如何從 CRXJS、WXT 或 Plasmo 遷移？">

        [帶瀏覽器前綴的鍵](/docs/features/browser-specific-fields)），以及更新
        `package.json` 中的指令稿。詳細步驟請參見 [從 CRXJS 遷移](/docs/migrate/from-crxjs)。
